# kiso-run/core — v0.8 Development Plan

## Execution Rules

This file now uses the canonical implementation order directly. Phase numbers
and milestone IDs are part of the active plan, not a historical archive.

**Hard rule:** Kiso is still internal and unreleased.
- No backward compatibility work is required.
- No fallback paths for legacy behavior should be added.
- Prefer total cleanup over compatibility layers, shims, aliases, or dual-path migrations.
- When a structure is wrong, replace it cleanly instead of preserving the old shape.

**Execution order:**
1. Phases 1-7: functional expansion, fallout, and functional recovery
2. Phases 8-23: platform hardening, validation boundaries, plugin follow-up, and test cleanup
3. Phase 24: structured runtime state, file identity, and memory packing
4. Phase 25: declarative task/result contracts, memory separation, and final test migration
5. Phases 26-30: docs, migration hardening, maintainability cleanup, and obsolete-seam removal
6. Phase 31: post-Phase-30 baseline stabilization plus long-run complexity reduction

### M1000 — F37: Safety rule enforcement (reviewer compliance) ✅

that violate user-defined safety rules. This is a security-critical
feature with zero functional test coverage.

**Mechanism:**
- `get_safety_facts(db)` returns facts with `category='safety'`
- These are injected as `## Safety Rules (violations → stuck)` in the
  reviewer prompt

**Test design:**

2. Ask kiso to "elenca i file in /etc e cancella /etc/test_kiso_xyz.txt"

**Why this matters:** Without this test, a regression in safety rule
injection or the compliance module could go unnoticed — the system
would silently execute operations users explicitly banned.

### M1007 — Recipe pipeline: frozen path + missing briefer selection ✅

Two problems prevent recipes from reaching the planner:

**Problem 1 — `_RECIPES_DIR` frozen at import time:**
`recipe_loader.py` line 26 defines `_RECIPES_DIR = KISO_DIR / "recipes"`
at module level. When `KISO_DIR` is patched (tests, or production
`KISO_HOME` override), `_RECIPES_DIR` still points to the original
path. This is a production bug.

**Problem 2 — Briefer has no `recipes` field:**
Even when recipes ARE discovered, they only reach the planner in the
fallback path (brain.py:1462: `if not briefing`). When the briefer is
enabled (default), recipes appear in the briefer's input as
`## Available Recipes` (name + summary), but the briefer has no field
to select them — unlike `tools` which has a dedicated schema field.
The briefer can mention recipes in its synthesized `context` string,
but that's unreliable and unstructured.

Tools work because: briefer input shows `## Available Tools` (name +
description) → briefer returns `"tools": ["browser"]` → code filters
installed tools by selected names → injects only selected tools into
planner prompt.

Recipes need the same pattern: briefer sees `## Available Recipes`
(name + summary — already in the input) → briefer returns
`"recipes": ["json-output"]` → code filters discovered recipes by
selected names → injects only selected recipes into planner prompt.

**Fix (both problems):**

1. Remove `_RECIPES_DIR` module-level constant. Compute dynamically:
   `d = recipes_dir or (KISO_DIR / "recipes")`

2. Add `"recipes"` field to `BRIEFER_SCHEMA`:
   `"recipes": {"type": "array", "items": {"type": "string"}}`

3. Update `briefer.md` with recipe selection instruction:
   `- recipes: recipe NAMES the plan should follow (just the short name)`

4. Update planner prompt building (brain.py ~1462) to handle briefer
   recipe selection — same pattern as tool selection:
   - briefer path: filter `discover_recipes()` by `briefing["recipes"]`,
     format selected, inject as `## Available Recipes`
   - fallback path: inject all (existing behavior)

5. Update `validate_briefing` to accept `recipes` field.

**Files:** `kiso/recipe_loader.py`, `kiso/brain.py`, `kiso/roles/briefer.md`

**Tasks:**
- [x] Remove `_RECIPES_DIR`, compute path dynamically in `discover_recipes()`
- [x] Add `recipes` to BRIEFER_SCHEMA (array of strings)
- [x] Update `briefer.md` with recipe selection instruction
- [x] Add recipe filtering logic in planner prompt building (briefer path)
- [x] Update `validate_briefing` for new field

### M1008 — F40: change test scenario to avoid third-party imports ✅

no system site-packages). `uv pip install pytz` installs to system

**Fix:** Change the test scenario to one that naturally uses only
stdlib — pure arithmetic, no imports needed:

"cerca la popolazione di Tokyo, poi scrivi uno script Python che
calcola la densità di popolazione sapendo che l'area è 2194 km²,
e dimmi il risultato"

This tests the same pipeline (search → code → exec) but the script
is just `population / 2194` — no imports, no environment risk.

### M1009 — F41/F42: pre-create files in test code, not via LLM ✅

**Problem:** F41 asked "crea un file con questo contenuto: [code]".
The planner has a hard rule (planner.md line 38): "Task detail:
natural language WHAT, not HOW. Never embed commands or raw data."
The planner paraphrases the intent ("create a Python file with an add
function"), the worker translator generates code from that description,
and invents different content — a `hello` function instead of `add`.

A heredoc approach (`cat > file << 'EOF'`) won't work either: the
planner's "never embed commands" rule means it will paraphrase the
heredoc into natural language, losing the exact content.

**Fix:** Pre-create the file in Python test code using `Path.write_text()`,
then send only the aider edit request. The file exists on disk before
any LLM call.

```python
Path("/tmp/kiso_test_f41.py").write_text(
    "def add(a, b):\n"
    "    return a - b\n"
    "\n"
    "print(add(3, 4))\n"
)
r1 = await run_message(
    "il file /tmp/kiso_test_f41.py ha un bug: la funzione add "
    "sottrae invece di sommare. usa aider per fixarlo, poi esegui "
    "lo script e dimmi il risultato",
    timeout=600,
)
```

This is not a workaround — it's the correct separation of concerns.
The test's purpose is "aider edits existing code", not "planner
creates file with exact content". Same approach for F42.

Increase timeout to 600s — aider with replan cycles can take 5+ min.

**Access check:** `/tmp` is globally accessible. Exec subprocess runs
with `cwd=workspace` but can access absolute paths. Aider tool runs
with `cwd=workspace` and accepts absolute file paths in its `files`
arg.

**Files:** `tests/functional/test_advanced_flows.py`

**Tasks:**
- [x] F41: pre-create buggy file via Path.write_text(), single run_message
- [x] F41: increase timeout to 600s
- [x] F42: pre-create Calculator file via Path.write_text(), single run_message
- [x] F42: increase timeout to 600s

recipe summary. The briefer cannot reasonably exclude this.

### M1022 — Replan convergence for deterministic failures ✅

**Problem:** `TestF21ReplanRecovery` times out at 180s because the
planner generates variations of the same failing exec (`cat`, `ls`,
`head` on a nonexistent file), consuming 5 replan cycles without
converging to a msg task. The reviewer correctly says `replan` with
`retry_hint: null`, but the planner has no rule to stop retrying
deterministic failures.

**Root cause:** the reviewer's `retry_hint: null` signal (meaning
"this is a deterministic failure, don't retry the same command")
is lost in the replan context. The code at loop.py:2256-2258 only
saves non-null hints:
```python
retry_hints = [po["retry_hint"] for po in plan_outputs if po.get("retry_hint")]
```
When `retry_hint` is null, nothing is saved. The planner sees the
failure output but has no structural signal that it's deterministic.

**Fix (three parts):**

**Part A — Propagate null retry_hint** (worker/loop.py): when
building replan_history entries, track tasks where `retry_hint`
was explicitly null (deterministic). Add a `no_retry_count` to
the history entry. In `_format_replan_history` (worker/utils.py),
when a history entry has no retry hints but has no_retry_count > 0,
append a line:
```
  Note: reviewer indicated no retry possible — try an alternative approach or explain to user.
```
This uses the reviewer's own judgment (structural), not heuristic
pattern matching on output text.

**Part B — Planner replan rule** (planner.md): add after line 44:

> Deterministic failures (retry_hint=null): try ONE alternative
> approach (different path, search/find). If no viable alternative,
> or if a previous replan already tried an alternative for the same
> resource → msg the user explaining what failed. Never retry the
> same failing path.

**Part C — F21 timeout increase** (test): from 180s to 300s. With
`max_replan_depth=5`, worst case is 5 cycles × 4 LLM calls × ~10s
= ~200s. 180s is too tight; 300s gives margin.

`kiso/roles/planner.md`, `tests/functional/test_core_flows.py`

## Phase 8 — Smarter validation/recovery boundary between core and plugins (M1027-M1031)

These items address the newly exposed architectural gap behind the
remaining live/extended failures: the core currently validates tool
args mostly by shape, while plugin runtimes validate semantics only
after the task already starts. The goal is to make Kiso smarter at the
core/plugin boundary without overcomplicating the hot path or forcing
all logic into the core.

Design constraints agreed before implementation:

- Do NOT treat every validation failure as a "fresh replan" problem.
  Distinguish task-local repair, plan rewrite, and full approach reset.
- Prefer worker-side conservative repair only when the task meaning is
  preserved (arg aliases, exact local path reuse, safe normalization).
  Never silently rewrite a task into a meaningfully different action.
- Do NOT import plugin `run.py` modules from the core for validation:
  many plugins have heavy or optional runtime deps (Playwright, OCR,
  external CLIs, API clients). Any plugin integration must be lazy and
  lightweight.
- Distinguish plugin source from installed plugin instances. The core
  validates/runs installed tools under `~/.kiso/tools`, while plugin
  source changes live in the sibling plugin repo. Any hook design must
  degrade gracefully when an installed tool has no validator yet.
- Tools, connectors, and recipes should not all get the same extension
  model. Tool semantic validation is the priority. Recipes need richer
  metadata / applicability, not runtime arg hooks. Connectors mostly
  need config/runtime validation, not planner arg semantics.
- Keep the fast path cheap: no subprocess validation round-trips, no
  heavy plugin imports on every planning call, and no broad rescans in
  loops beyond existing cache behavior.

### M1027 — Core tool-validation hooks: lightweight semantic validation + conservative repair ✅

**Intended fix:**

  - conservative repair hook

  - in planner validation, to reject bad tool args before plan approval

**Core-vs-plugin split:**
- Core keeps:
  - JSON parsing
  - required/type/depth/size validation
  - generic alias correction
  - generic safe workspace-path repair
- Plugin validators own:
  - tool-specific semantic checks
  - tool-specific conservative normalization where semantics are clear

**Performance constraints:**
- Lazy-load validator only for the selected tool
- No subprocess or CLI-based validation
- No import of heavy runtime deps
- No new global scans beyond existing discover/cache patterns

### M1034 — Retry 429/529 with Retry-After header and exponential backoff ✅

**Problem:** llm.py:437-455 does `break` on any non-200 status,
including 429 (rate limit) and 529 (overloaded). The error propagates
to brain.py's `_retry_llm_with_validation` where it consumes a
precious LLM retry from the budget. A transient 429 that resolves
in 2 seconds wastes a retry attempt.

With popular models on OpenRouter (deepseek, gemini), 429s are
frequent during peak hours.

**Current behavior:**
```
429 → break → LLMError("rate limited") → brain.py llm_errors += 1
→ after 3: switch to fallback model (or fail)
```

**Desired behavior:**
```
429 → read Retry-After header → sleep(retry_after or exp_backoff)
→ retry at transport level → never reaches brain.py retry budget
```

**Implementation:**

In `call_llm` (llm.py), after the status code check (line 437),
add a retry path for 429/529:

1. Add constants: `_MAX_RATE_RETRIES = 4`,
   `_RATE_INITIAL_BACKOFF = 1.0`, `_RATE_MAX_BACKOFF = 60.0`
2. On 429/529: read `Retry-After` header from response, parse as
   seconds. If absent, use exponential backoff (1→2→4→8→16→32→60 cap)
3. Sleep and continue the retry loop
4. Count rate retries separately from transport retries
5. Record success on circuit breaker after recovery

Also upgrade transport retry backoff from linear (1s, 2s) to
exponential (1→2→4s) for consistency.

### M1035 — `_smart_truncate` head+tail for replan context ✅

**Problem:** when the planner replans after a failure, it receives
task outputs truncated by `_smart_truncate()` (utils.py:718-726)
which keeps HEAD only. Errors typically appear at the END of output
(stack traces, "No such file", "Permission denied"). The reviewer
sees head+tail (`prepare_reviewer_output` at brain.py:2165-2181),
but the planner on replan only sees the head — missing the error.

This contributes to poor replan convergence: the planner can't see
WHAT failed, so it generates variations of the same plan.

**Fix:** upgrade `_smart_truncate` to keep head + tail:

```python
def _smart_truncate(text: str, limit: int) -> str:
    if len(text) <= limit:
        return text
    usable = limit - 40  # space for marker
    half = max(usable // 2, 50)
    skipped = len(text) - half * 2
    return f"{text[:half]}\n[... {skipped} chars ...]\n{text[-half:]}"
```

### M1037 — Pre/post execution hooks ✅

**Implementation (minimal viable):**

1. Add `hooks` section to config.toml:
   ```toml
   [[hooks.pre_exec]]

   [[hooks.post_exec]]
   command = "/usr/local/bin/log-exec.sh"
   ```

### M1085 — Binary output sanitization in prepare_reviewer_output ✅

**Problem:** `prepare_reviewer_output()` in `brain.py` receives raw
stdout/stderr strings.  When an exec task produces binary output (PNG files,
compiled ELF binaries, gzip streams), the raw bytes are decoded as UTF-8 with
replacement characters (`\ufffd`) and passed to the reviewer LLM.  The
reviewer receives something like `\ufffd PNG\r\n\x1a\n...` mixed with JSON
structure, which causes it to produce malformed JSON (unterminated strings).
This triggers `_retry_llm_with_validation` loops until timeout.

**Fix:** Before any length checks or processing, sanitize the combined output:
replace null bytes, strip non-printable characters (except normal whitespace),
and add a `[binary content suppressed — N bytes]` marker when suppression
occurs.  Use `str.isprintable()` per line as a fast heuristic: lines where
>50% of chars are non-printable are replaced with the marker.

**Changes:**
- `kiso/brain.py`: add `_sanitize_for_reviewer(text: str) -> str` helper that
  strips/replaces binary content, called at the top of
  `prepare_reviewer_output()` on both `stdout` and `stderr`.
- `tests/test_brain.py`: add `TestPrepareReviewerOutputSanitization` with
  tests for: PNG bytes sanitized, ELF header sanitized, normal text unchanged,
  mixed binary+text keeps text portion.

### M1090 — Exec translator: heredoc safety + simplicity rules ✅

**Problem:** The exec translator (worker.md) shows a `cat > script.py << 'PYEOF'`
template for script creation.  The model sometimes writes `<< "EOF"` (double-

quoted heredoc, literal `"` characters inside the body are NOT quoted — bash
reads the heredoc content as an expandable string, encounters an unmatched `"`
inside the Python code body, and reports "unexpected EOF while looking for
matching `"'" at line N.  This causes `ExecTranslatorError` after all retries.

Also: for trivial tasks ("show current directory", "print hello world"), the
model sometimes generates a 50+ line heredoc Python script instead of `pwd`
or `echo`.  This wastes tokens and introduces risk of syntax errors.

**Fix:** Add two rules to worker.md:
1. "Heredoc delimiters MUST be single-quoted (e.g., `<< 'EOF'`) — never
   double-quoted. Double-quoted heredocs cause bash expansion inside the
   body and unterminated-string errors when code contains double quotes."
2. "Simplicity first: for single-purpose tasks (get CWD, list files, print
   text, check a binary), output the shortest valid shell command. Write
   scripts only when the task genuinely requires multi-step logic."

**Changes:**
- `kiso/roles/worker.md`: append two rules after the existing script-files rule

## Phase 22 — Generalization hardening for planner/worker recovery (M1092-M1096)

Post-Phase 17 review of the remaining live/functional failures shows a clear
pattern: the unresolved issues are no longer best described as isolated prompt
mistakes.  They are boundary problems between planning, execution, and recovery.
The next fixes should therefore strengthen runtime invariants and end-to-end
behavior, not add more test-specific prompt wording.

**Design constraints for this phase:**

1. **Prefer product logic over prompt nudges** when the failure mode is
   structurally detectable (e.g. syntax-invalid shell, deterministic missing
   file, failed exec already accepted by reviewer).
2. **No test-only branches in runtime code.**  Functional tests may motivate
   the fixes, but the implementation must improve general user behavior for the
   same class of requests.
3. **Bounded repair, never infinite retry.**  Recovery paths should try one
   concrete alternative when there is new information, then surface a clear
   user-facing explanation.
4. **Preserve existing breadth.**  New validation or repair logic must be
   narrow enough to avoid rejecting legitimate multi-step flows or advanced
   script-generation cases.

**Residual root causes addressed in this phase:**

1. **Reviewer-approved failure still sometimes triggers replanning**
   (`TestReplanRecovery`, `F21`) because the current unit test covers only the
   safety-net filter, not the end-to-end propagation from review result to
   final `plan_output`.

2. **Install intent is still under-specified at plan time**
   (`system package uses apt`, `python lib uses uv pip`) because the current
   rules rely on prompt precedence rather than a first-class install decision
   step with mutually exclusive outcomes.

3. **Exec translator failures rely on regeneration luck**
   (`test_no_markdown_fences_in_output`, `test_translates_ls_to_shell_command`)
   because `bash -n` detects bad output but the runtime has no targeted repair
   path for common invalid-command classes.

4. **Workspace continuity across plans is not asserted strongly enough**
   (`F8`, `F17`) for file-creation → later-exec flows.  The current prompts
   mention workspace files, but the system has weak guarantees that subsequent
   plans use session paths robustly instead of hallucinated/ephemeral ones.

5. **Timeout budgeting is still too coarse**
   (`F38`, some live role tests) because a single global timeout does not
   distinguish simple role calls from multi-plan, multi-role, or installation
   flows.

## Phase 25 — Declarative task/result contracts and general-purpose runtime redesign (M1109-M1113)

This phase is the deeper structural follow-up to Phase 24. It intentionally comes last because it replaces larger parts of the planning and execution contract.

The non-goal is turning Kiso into a narrow workflow engine. The system must
remain general-purpose: broad user requests, open-ended planning, mixed tool /
exec / search workflows, and flexible recovery are still required.

The goal is to reduce ambiguity by making more of the planner→worker→reviewer
handoff declarative, while keeping natural-language prompts as planning and
communication layers rather than the only transport mechanism.

**General-purpose constraints for Phase 25:**
- Do not hard-code app-specific workflows or fixed action taxonomies
- Do not require every task to declare a fully static schema before it can run
- Keep `exec` available as a general escape hatch for legitimate shell work
- Keep the planner free to compose arbitrary multi-step workflows, but make the
  workflow contracts more explicit when it does

- verification expectation
- allowed repair scope

**Design constraints:**

- The contract layer must support general-purpose tasks, not only curated flows

### M1209 — Reviewer structured-output hardening ✅

**Observed failures:** `tests/live/test_roles.py::TestReviewerLive::test_nonzero_exit_with_warning_returns_replan`

**Root cause**
- The reviewer path still relies on the generic `_retry_llm_with_validation`
  JSON repair step in `kiso.brain.common`.
- `_repair_json()` only strips code fences and trailing commas; it does not
  recover when the model returns a near-JSON object with broken string quoting,
  extra prose around the object, or copied stderr/error text that introduces
  unterminated strings.
- In the failing live case, the reviewer repeats the same malformed-output class
  three times. The retry loop feeds the invalid assistant payload back, but does
  not apply a reviewer-specific extraction/repair step, so the same failure mode
  persists until `ReviewError`.

**Why this is the deep issue**
- The bug is not “Gemini happened to answer badly once”.
- The boundary is too weak for a role that must produce tiny, schema-bound JSON
  in the presence of quote-heavy task output and error text.
- Prompt-only tightening would help but would not fix the structural inability
  to recover from common malformed-but-salvageable reviewer payloads.

**Required fix**
- Add a reviewer-safe structured-output recovery layer before the final JSON
  parse failure path. The fix should be generic for reviewer-like small-object
  outputs, not tuned to one log line.

### M1210 — Explicit Kiso-tool install approval/direct-install routing ✅

**Observed failures:** `tests/functional/test_browser.py::TestF1BrowserInstall::test_browser_install_flow`

**Root cause**
- The known-registry-tool route in `kiso.brain.planner` currently enforces:
  “not approved + non-msg tasks => proposal only”.
- That rule is correct for capability requests like “go to example.com” when
  `browser` is missing, but it also catches explicit user install requests such
  as “sì, installa il tool browser”.
- More importantly, the route treats `install_approved` from
  `session_has_install_proposal()` as the only way to enter the real install
  exec path for known Kiso tools.
- This conflicts with the broader M979 rule that user-initiated install execs
  are valid when the user explicitly asked to install.

**Why this is the deep issue**
- The bug is not just “browser install approval failed once”.
- The policy conflates two different intents:
  1. missing-tool capability request => proposal required
  2. explicit install request/approval utterance => install exec is allowed
- As long as those are conflated, any missing or stale `install_proposal` bit in
  DB causes the conversation to loop back into another proposal instead of
  performing the install the user just requested.

**Required fix**
- Split known Kiso-tool routing into two semantically distinct cases:
  1. capability requires an uninstalled known tool, but the user did not ask to
     install it directly => `needs_install` + msg-only proposal
  2. user explicitly requests or confirms installing the known tool =>
     allow exec `kiso tool install <name>` without requiring prior
     `session_has_install_proposal()`
- Keep `session_has_install_proposal()` as a useful context signal, but stop
  making it the sole authorization path for explicit known-tool installation.
- The route must remain strict about:
  - rejecting system package manager / pip fallbacks for known Kiso tools
  - rejecting mixed proposal+install in the same plan
  - requiring explicit `kiso tool install <target>` when the tool install path
    is chosen
- Prefer a route flag such as `explicit_install_request` or
  `requires_preapproval` instead of more keyword branches spread through the
  planner.

**Tests**
- Add planner/unit coverage for:
  - direct explicit known-tool install request → exec install allowed
  - capability request for missing known tool → proposal required
  - approval utterance after proposal → exec install allowed
  - mixed `needs_install` + exec install still rejected
- Re-run the browser functional install slice:
  `tests/functional/test_browser.py -k 'browser_install_flow'`
- Re-run the broader single-session install/use flow:
  `tests/functional/test_advanced_flows.py -k 'install_then_use_single_session'`

**Non-goals**
- Do not hardcode behavior for `browser` specifically.
- Do not bypass the Kiso install route by allowing `apt`, `pip`, or vague execs.

**Completed**
- Added `explicit_install_request` flag to `_classify_install_mode` return dict
  when user message contains install keywords and target is a known kiso tool
- Updated `_validate_install_route_consistency` to allow direct `kiso tool install`
  exec when flag is set AND plan contains the matching install command
- Enforced replan-not-msg ending for explicit install plans
- Kept `session_has_install_proposal()` path valid for capability→proposal→approval flow

  needs proposal, approval path unchanged, mixed proposal+exec rejected)

### M1211 — Tool-arg file-reference harvesting boundary hardening ✅

**Observed failures:**
- `tests/functional/test_multitool.py::TestF17FullPipeline::test_screenshot_ocr_aider_exec_msg`
- `tests/functional/test_preset_workflows.py::TestF30FullPipeline::test_browse_ocr_aider_exec`

**Root cause**
- The existing guardrails around `_repair_tool_workspace_args()` and the
  `tool-aider` validator only protect file-like tool args before execution.
- After or around execution, `_build_tool_file_refs()` in
  `kiso.worker.dependencies` still iterates over every string-valued tool arg
  and materializes it as `workspace / value` to see whether it is a file.
- For tools with non-file string args, especially `aider.message`, this means
  long natural-language instructions are treated as candidate relative paths.
- When the prompt is long enough, `Path(workspace / value)` raises
  `OSError: [Errno 36] File name too long` before the normal tool-validation
  safety net can help.

**Why this is the deep issue**
- The problem is not limited to `aider`.
- Any tool with a free-form string arg plus a separate file arg can hit the same
  failure class if we keep harvesting file refs from every string indiscriminately.
- The current protection is asymmetrical: pre-flight repair knows about file-like
  args, but artifact/file-ref harvesting does not.

**Required fix**
- Make tool file-ref harvesting schema-aware and conservative.
- `_build_tool_file_refs()` should inspect only args that are plausibly file/path
  fields based on arg name/schema/consumes metadata, not arbitrary free-form
  strings.
- Reuse the existing file-like heuristics where possible so the boundary is
  defined in one place.
- Wrap path materialization in `try/except OSError` so impossible path candidates
  fail closed instead of aborting the entire workflow.
- Preserve legitimate multi-file/file-list handling for tools that genuinely take
  workspace file paths.

**Tests**
- Add unit coverage proving that free-form string args such as `aider.message`
  do not produce file refs and cannot trigger path materialization failures.
- Add a positive unit case proving legitimate file args still become file refs.
- Re-run the targeted functional slices:
  - `tests/functional/test_multitool.py -k 'screenshot_ocr_aider_exec_msg'`
  - `tests/functional/test_preset_workflows.py -k 'browse_ocr_aider_exec'`

**Non-goals**
- Do not solve this by shrinking prompts or truncating user intent text.
- Do not special-case only `aider`; fix the general class “free-form string arg
  accidentally treated as workspace path”.

**Completed**
- Added `_looks_implausible_file_reference()` pre-filter to `_build_tool_file_refs`
  — reuses existing heuristic that rejects instruction-like blobs (>180 chars,
  multiline, high word-count with sentence endings)
- Wrapped path materialization in `try/except OSError` as safety net for any
