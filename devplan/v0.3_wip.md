# Devplan v0.3 (WIP) — Skill Failures & Resilient Replanning

Goal: fix the failure modes that prevent kiso from recovering when skill tasks
fail (wrong args, missing skills, LLM errors) and improve planner awareness of
skill args format.

Derived from a real trace: user asks "vai su guidance.studio, dimmi cosa fa e
mandami screenshot". Browser skill is installed but planner sends `args: null`,
skill validation fails with "missing required arg: action", system terminates
without replanning.

---

## Problem Inventory

### P1 — Skill setup failure is terminal (no replan)
`_handle_skill_task` (loop.py:752) returns `_TaskHandlerResult(stop=True,
stop_success=False)` with NO `stop_replan` reason when:
- Skill is not installed (`Skill '{name}' not installed`)
- Args JSON is invalid
- Args validation fails (missing required args, wrong types)

In `_run_planning_loop` (loop.py:1439), `replan_reason is None` triggers
`_handle_loop_failure` and breaks. The planner never gets a chance to fix
the args or install the skill.

### P2 — No skill args example in planner prompt
`planner.md` line 7 says `args = JSON string` but shows NO example of how
to format it. The skill list (from `build_planner_skill_list`) shows:
```
- browser — Browser automation
  args: action (string, required): the action to perform
  args: url (string, optional): target URL
```
But the planner (deepseek-v3.2) doesn't connect this to the JSON format
needed in the plan. It sets `args: null` for all skill tasks.

### P3 — Plan validation doesn't check skill args
`validate_plan` (brain.py:292-381) validates that the skill NAME is installed
but does NOT validate args against the schema. Invalid args only surface at
execution time (loop.py:740), wasting a full execution cycle.

### P4 — No retry logic for skill tasks
Exec tasks have a retry loop with `max_worker_retries` and `retry_hint`.
Search tasks have the same. Skill tasks have ZERO retry logic — any failure
(setup or execution) is immediately terminal.

### P5 — Exec translator error is terminal
`_handle_exec_task` (loop.py:836) catches `ExecTranslatorError` and returns
`stop=True, stop_success=False` without `stop_replan`. The translator LLM
timeout kills the entire plan.

### P6 — Search task error is terminal
`_handle_search_task` (loop.py:959) catches `SearcherError` and returns
without replan. A transient API error or timeout is unrecoverable.

### P7 — Review LLM error is terminal for all task types
`_run_review_step` errors (loop.py:786, 880, 971) return without replan.
If the reviewer LLM times out on an otherwise-successful task, the plan dies.

### P8 — `extend_replan` accepted but ignored on first plan
The planner can set `extend_replan: 3` on the initial plan (as seen in trace).
It's silently ignored since extend_replan is only processed during replans
(loop.py:1635-1647). This wastes the planner's "budget thinking".

### P9 — No auto-replan on unhandled failures
When `replan_reason is None` (loop.py:1439), the system could auto-generate
a replan reason from the failed task's output instead of giving up. This is
a safety net for any failure mode that forgets to set `stop_replan`.

### M164 — Skill setup failure triggers replan instead of terminal failure
**Problem:** P1
**Files:** `kiso/worker/loop.py` — `_handle_skill_task`
**Change:** When `setup_error` is set (skill not installed, invalid JSON, args
validation failed), instead of:
```python
return _TaskHandlerResult(stop=True, stop_success=False)
```
Return with `stop_replan`:
```python
return _TaskHandlerResult(
    stop=True, stop_success=False,
    stop_replan=f"Skill task failed: {setup_error}",
    plan_output=_make_plan_output(i + 1, "skill", detail, setup_error, "failed", session=ctx.session),
)
```
This way the planning loop gets a replan reason and can retry with corrected
args or install the missing skill.
**Test:** Mock a skill task with invalid args. Verify `_execute_plan` returns
`replan_reason` containing the error message (not None). Verify `plan_outputs`
contains the error so the planner sees it in replan context.

### M166 — Validate skill args at plan validation time
**Problem:** P3
**Files:** `kiso/brain.py` — `validate_plan`
**Change:** When validating a skill task, if the skill is installed (already
checked), also validate args against the schema:
1. Parse the `args` field as JSON (if it's a string) or use it directly (if dict/null)
2. Call `validate_skill_args(args, schema)` from `kiso/skills.py`
3. If validation fails, add to plan errors (same as other validation errors)

This catches bad args BEFORE execution, giving the planner immediate feedback
via the plan validation error path.

Import `validate_skill_args` in brain.py and pass `installed_skills` to the
validation function (already available in `build_planner_messages` context).
**Test:** Build a plan with a skill task that has `args: null` when the skill
requires a mandatory arg. Verify `validate_plan` returns an error mentioning
the missing arg.

### M167 — Add retry logic for skill tasks
**Problem:** P4
**Files:** `kiso/worker/loop.py` — `_handle_skill_task`
**Change:** Add a retry loop similar to exec tasks. When the reviewer returns
`status: "replan"` with a `retry_hint`, retry the skill task with modified
args (up to `max_worker_retries`). The retry context should include the
original args, the error output, and the retry_hint.

Structure:
```python
retries = 0
while True:
    # ... execute skill task ...
    review, review_error = await _run_review_step(ctx, task_row)
    if review_error is not None:
        # ... stop ...
    if review["status"] == REVIEW_STATUS_REPLAN:
        retries += 1
        if retries > ctx.max_worker_retries:
            # ... stop with replan ...
        retry_hint = review.get("retry_hint", "")
        # Rebuild args from retry context (ask translator-like step or just replan)
        # For skills, retry_hint can't easily modify args without LLM, so replan
        break
    # ... success ...
```

Actually, for skill tasks the retry is less useful than for exec (where the
translator can adjust the command). For skills, the args come from the planner.
So: on first failure, replan immediately with the error + retry_hint. Don't
loop internally — let the planner fix the args.

Simpler change: after reviewer says replan, return `stop_replan` with the
reviewer's reason + retry_hint. This is already handled for exec tasks but
currently works for skill tasks too (line 788-794). The gap is only the
**setup_error** path (M164 already fixes this). So M167 reduces to: verify
that skill execution failures (not just setup) also replan properly.

Check: if the skill subprocess fails (exit code != 0) and reviewer says replan,
does the current code handle it? Yes — lines 788-794 already return
`stop_replan`. The only gap was setup errors (M164).

**Revised scope:** This milestone verifies that the existing reviewer-driven
replan path works for skill execution failures, and adds a test for it.
**Test:** Mock a skill task that executes but fails (exit_code=1). Mock reviewer
returning `status: "replan"`. Verify `_execute_plan` returns `replan_reason`
with the reviewer's reason.

### M172 — Auto-replan safety net for unhandled failures
**Problem:** P9
**Files:** `kiso/worker/loop.py` — `_run_planning_loop`
**Change:** When `replan_reason is None` and `success is False`, instead of
immediately calling `_handle_loop_failure`, attempt ONE auto-replan if there
are remaining replan budget slots. Generate a replan reason from the failed
task outputs:
```python
if replan_reason is None:
    # Auto-generate replan reason from failed tasks
    failed_outputs = [
        po for po in plan_outputs
        if po.get("status") == "failed"
    ]
    if failed_outputs and replan_depth < max_replan_depth:
        last_fail = failed_outputs[-1]
        replan_reason = f"Task failed: {last_fail.get('output', 'unknown error')[:200]}"
        log.info("Auto-generating replan reason: %s", replan_reason)
        # Fall through to replan handling below
    else:
        log.info("Plan %d failed (no replan)", current_plan_id)
        await _handle_loop_failure(...)
        break
```
This ensures that even if a handler forgets to set `stop_replan`, the system
still tries to recover.
**Test:** Mock `_execute_plan` returning `(False, None, [], remaining, plan_outputs)`
where `plan_outputs` has a failed entry. Verify the loop auto-replans instead
of calling `_handle_loop_failure`.
