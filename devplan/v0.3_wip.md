# Devplan v0.3 (WIP) — Skill Lifecycle, Volume Management & Resilient Replanning

Goal: fix the full chain of failure modes discovered from a real trace where the
browser skill appeared installed (volume persisted) but its system deps were gone
(image rebuilt), and the system couldn't recover. Covers install.sh bugs, skill
health checks, startup repair, retry_hint gaps, and planner guidance.

Derived from: real trace on fresh install — user asks "vai su guidance.studio,
dimmi cosa fa e mandami screenshot". Volume was supposedly wiped but skill dir
persisted. Playwright binaries missing. System looped without recovering.

Previous milestones (M164-M173) in this file are all completed and kept for
reference below the new milestones.

---

## Problem Inventory

### P1 — `DO_RESET` set but never used (install.sh)
When the user picks "Update instance" and confirms "Reset data?", install.sh
sets `DO_RESET=true` (line 505) but the actual factory reset on line 816 checks
`RESET_REQUESTED` (only set by `--reset` CLI flag, line 108). The interactive
confirmation has **zero effect** — factory reset never runs.

### P2 — Root-owned file cleanup is conditional and incomplete (install.sh)
Lines 591-601 clean root-owned files only if `find` detects them. If the dir was
already chowned (e.g. previous install ran chown), the entire block is skipped.
Even when it runs, it deletes `skills/*/` via glob but doesn't guarantee a clean
slate.

### P3 — Skill deps live in the image, skill dir lives on the volume
`deps.sh` installs system packages (apt-get, playwright install) into the
container image filesystem. The skill directory (kiso.toml, venv) lives on the
volume. On image rebuild, system deps are gone but the skill directory persists.
`discover_skills()` finds the skill (valid kiso.toml, no .installing marker) and
reports it as installed. The planner sends tasks to a broken skill.

### P4 — No skill health check at discovery time
`discover_skills()` (skills.py:101) only checks: dir exists, no `.installing`
marker, valid `kiso.toml`. It does NOT call `check_deps()` to verify binary deps
are present. A skill can appear installed while its runtime is completely broken.

### P5 — Skill handler missing retry_hint carry-forward
The exec handler (loop.py:927-929) and search handler (loop.py:1025-1027) both
propagate `retry_hint` from the reviewer into `plan_output`, which flows to the
replan context's "Suggested Fixes" section. The skill handler (loop.py:797-803)
does NOT — it returns `stop_replan=replan_reason` but never attaches `retry_hint`
to `plan_output_entry`. The planner can't see the reviewer's fix suggestion.

### P6 — Planner has no guidance for broken skill runtimes
When a skill is installed but broken (missing deps), the planner doesn't know
how to fix it. It keeps retrying the same skill or gives up. It needs guidance:
"if a skill fails with a runtime error, try reinstalling its deps."

### P7 — Planner example hardcodes "browser"
The M165 example in planner.md (line 8) uses `"skill": "browser"` which can
mislead the LLM into thinking `browser` is always available. Should use a
generic/placeholder skill name.

## CLI Timeout & Planner Efficiency (M197-M199)

Goal: fix the CLI-side timeout that kills slow tasks (skill installs, large
builds) and reduce unnecessary planner steps by sharing capability-gap
detection results and install idempotency knowledge.

Derived from: real trace where `kiso skill install browser` takes >5 min
(Playwright + 70 system deps). CLI abandons polling at 300s even though the
server is still working. Additionally, the planner creates unnecessary
"check if installed" tasks because it doesn't know install is idempotent,
and fetches the registry even when the system already knows which skill is
needed.

---

### M197 — Remove CLI polling timeout
**Problem:** `_MAX_POLL_SECONDS = 300` in `cli/__init__.py` kills the CLI
polling loop after 5 minutes. We removed `exec_timeout` and `job_timeout`
from the server (M189-M190) so subprocesses can run indefinitely, but the
CLI still abandons. Any slow task (skill install, model download, large
build) gets truncated client-side while the server continues working. The
user sees `"error: timed out waiting for response"` and thinks it failed.
**Files:** `cli/__init__.py`
**Change:** Remove the `_MAX_POLL_SECONDS` timeout block entirely. The
polling loop already has two other exit conditions:
1. Normal completion: plan reaches done/failed status
2. Worker stopped: 3 consecutive polls with no worker running and no plan

These are sufficient. The absolute timeout is redundant and harmful.
Remove lines ~950 and ~1014-1022 (the `_MAX_POLL_SECONDS` constant and the
check block). The "worker stopped without producing a result" check (3
consecutive empty polls) already handles the stale-worker case.
**Test:** Existing CLI tests must pass. Verify `_MAX_POLL_SECONDS` is gone
from the code.

### M198 — Expose capability-gap detection result to planner
**Problem:** `_detect_capability_gap()` in `brain.py` already maps user
keywords to skill names (e.g. "navigate" → "browser", "refactor" → "aider").
When a gap is detected, it injects the generic `planner-plugin-install.md`
appendix. But the planner never sees WHICH skill was detected — it only
gets the 6-step install procedure. So it follows the full flow: fetch
registry → evaluate → install. If the planner knew the system already
identified "browser" as the needed skill, it could skip registry discovery
and go straight to install + replan.

This is general-purpose: any keyword in `_CAPABILITY_MAP` triggers it, not
just browser. The capability map covers screenshot, navigate, click, form,
browse, refactor, debug — all benefit equally.
**Files:** `kiso/brain.py` — `build_planner_messages()`
**Change:** When `_detect_capability_gap()` returns a skill name, add a
line to the planner context:

```python
if _gap:

        f"## Capability Analysis\n"
        f"Skill '{_gap}' is needed for this request but not installed. "
        f"Install it with: exec `kiso skill install {_gap}`, then replan."
    )
```
This goes into the context (not the system prompt), so it's visible as
structured information alongside Skills, System Environment, etc. The
planner can still choose to fetch the registry if it wants more info, but

### M199 — Tell planner that kiso skill install is idempotent

### M201 — Auto-inject intent msg before plan execution
**Problem:** When a plan has long-running tasks (skill install, builds,
large downloads), the user stares at a spinner for minutes with no idea
what's happening. The planner sometimes adds msg tasks, but always AFTER
exec tasks (per the rule "msg tasks MUST come after the tasks whose results
they communicate"). There's no mechanism to tell the user what the plan is
about to do BEFORE execution starts.
**Files:** `kiso/worker/loop.py` — `_execute_plan` or `_run_planning_loop`
**Change:** After the plan is validated and tasks are created in the DB,
but before the first task executes, the system checks if the first task is
already a msg task. If not, it **injects a msg task at position 0** with
a system-generated detail:
```python
if tasks and tasks[0]["type"] != "msg":
    lang = _detect_language(user_message)  # simple heuristic or default "en"
    task_summary = ", ".join(
        f"{t['type']}: {t['detail'][:60]}" for t in tasks[:3]
    )
    intent_detail = (
        f"[Lang: {lang}] Briefly tell the user what you're about to do. "
        f"Plan goal: {goal}. Steps: {task_summary}"
    )
    intent_task_id = await create_task(
        db, plan_id, session, "msg", intent_detail,
        skill=None, args=None, expect=None,
    )
    # Re-fetch tasks so the injected msg is included
    tasks = await get_tasks_for_plan(db, plan_id)
```

The injected msg task goes through the normal messenger LLM, so it:
- Speaks the user's language naturally
- Has the right tone and context
- Is visible in the CLI as `[1/N] msg: ...`

The intent msg is short (the messenger already has a concise style) and
gives immediate feedback: "Sto per installare il browser skill e poi
navigherò su guidance.studio per fare lo screenshot."

**Language detection:** Use a simple heuristic — check if the user message
contains common words in known languages, or default to "en". Alternatively,
reuse the planner's `[Lang: xx]` pattern if available from a previous msg
task in the session.

**Skip conditions:** Don't inject if:
- First task is already a msg
- Plan has only 1 task (a single msg or replan — no long work)

Injection happens for BOTH initial plans AND replans — the user should
always know what the bot decided to do, especially after a failure.

**Implementation:** `_maybe_inject_intent_msg(tasks, goal)` is a pure

### M204 — Skill task internal retry loop
**Problem:** Exec and search tasks have internal retry loops (up to
`max_worker_retries`) when the reviewer returns `replan` with a
`retry_hint`. Skill tasks have NO such loop — any reviewer `replan`
immediately escalates to a full plan-level replan, burning an entire
replan cycle for what might be a transient failure.
**Files:** `kiso/worker/loop.py` — `_handle_skill_task`
**Change:** Add a retry loop to `_handle_skill_task`, mirroring the pattern
in `_handle_exec_task` (lines 861-974). When the reviewer returns
`REVIEW_STATUS_REPLAN` with a non-null `retry_hint` and
`skill_retries < ctx.max_worker_retries`:

1. Increment `skill_retries`
2. Re-execute the skill with the same args (transient failure retry)
3. Re-review

If retries exhausted, escalate to plan-level replan (existing behavior).
For arg-level changes, escalate to replan (planner must fix args).

```python
skill_retries = 0
while True:
    # ... execute skill ...
    # ... review ...
    if review["status"] == REVIEW_STATUS_REPLAN:
        retry_hint = review.get("retry_hint")
        if retry_hint and skill_retries < ctx.max_worker_retries:
            skill_retries += 1
            # Re-execute with same args (transient failure retry)
            continue
        # Exhausted or no hint → escalate
        return _TaskHandlerResult(stop=True, ...)
    break
```

  Verify it retries internally without triggering replan.
- Mock a skill that fails twice with max_worker_retries=1. Verify it
  escalates to replan after exhausting retries.
- Verify retry count is logged and persisted via `update_task_retry_count`.
