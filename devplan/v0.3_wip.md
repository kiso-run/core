# Devplan v0.3 (WIP) — Skill Lifecycle, Volume Management & Resilient Replanning

Goal: fix the full chain of failure modes discovered from a real trace where the
browser skill appeared installed (volume persisted) but its system deps were gone
(image rebuilt), and the system couldn't recover. Covers install.sh bugs, skill
health checks, startup repair, retry_hint gaps, and planner guidance.

Derived from: real trace on fresh install — user asks "vai su guidance.studio,
dimmi cosa fa e mandami screenshot". Volume was supposedly wiped but skill dir
persisted. Playwright binaries missing. System looped without recovering.

Previous milestones (M164-M173) in this file are all completed and kept for
reference below the new milestones.

---

## Problem Inventory

### P1 — `DO_RESET` set but never used (install.sh)
When the user picks "Update instance" and confirms "Reset data?", install.sh
sets `DO_RESET=true` (line 505) but the actual factory reset on line 816 checks
`RESET_REQUESTED` (only set by `--reset` CLI flag, line 108). The interactive
confirmation has **zero effect** — factory reset never runs.

### P2 — Root-owned file cleanup is conditional and incomplete (install.sh)
Lines 591-601 clean root-owned files only if `find` detects them. If the dir was
already chowned (e.g. previous install ran chown), the entire block is skipped.
Even when it runs, it deletes `skills/*/` via glob but doesn't guarantee a clean
slate.

### P3 — Skill deps live in the image, skill dir lives on the volume
`deps.sh` installs system packages (apt-get, playwright install) into the
container image filesystem. The skill directory (kiso.toml, venv) lives on the
volume. On image rebuild, system deps are gone but the skill directory persists.
`discover_skills()` finds the skill (valid kiso.toml, no .installing marker) and
reports it as installed. The planner sends tasks to a broken skill.

### P4 — No skill health check at discovery time
`discover_skills()` (skills.py:101) only checks: dir exists, no `.installing`
marker, valid `kiso.toml`. It does NOT call `check_deps()` to verify binary deps
are present. A skill can appear installed while its runtime is completely broken.

### P5 — Skill handler missing retry_hint carry-forward
The exec handler (loop.py:927-929) and search handler (loop.py:1025-1027) both
propagate `retry_hint` from the reviewer into `plan_output`, which flows to the
replan context's "Suggested Fixes" section. The skill handler (loop.py:797-803)
does NOT — it returns `stop_replan=replan_reason` but never attaches `retry_hint`
to `plan_output_entry`. The planner can't see the reviewer's fix suggestion.

### P6 — Planner has no guidance for broken skill runtimes
When a skill is installed but broken (missing deps), the planner doesn't know
how to fix it. It keeps retrying the same skill or gives up. It needs guidance:
"if a skill fails with a runtime error, try reinstalling its deps."

### P7 — Planner example hardcodes "browser"
The M165 example in planner.md (line 8) uses `"skill": "browser"` which can
mislead the LLM into thinking `browser` is always available. Should use a
generic/placeholder skill name.

### M175 — Unconditional skill cleanup on rebuild in install.sh
**Problem:** P2
**Files:** `install.sh`
**Change:** When `NEED_BUILD=true` (image is being rebuilt) AND
`RESET_REQUESTED=true`, unconditionally remove the `skills/` directory from the
instance dir BEFORE starting the new container — don't rely on the root-owned
file detection:
```bash
if [[ "$NEED_BUILD" == true && "$RESET_REQUESTED" == true && -d "$INST_DIR/skills" ]]; then
    bold "Wiping skills directory (reset requested)..."
    # Use alpine container to handle root-owned files
    docker run --rm -v "${INST_DIR}:/mnt/kiso" alpine rm -rf /mnt/kiso/skills
    green "  skills directory wiped"
fi
```
Place this after the existing root-owned cleanup block (line 601) and before
"Configure" (line 604). This ensures skills are truly gone on a fresh start.
**Test:** Create a fake skills dir with a marker file, run the relevant install.sh
section, verify the skills dir is gone.

### M176 — Skill health check via check_deps() in discover_skills()
**Problem:** P4
**Files:** `kiso/skills.py` — `discover_skills()`
**Change:** After validating the manifest, call `check_deps()` on the skill info.
If deps are missing, add a `healthy` flag and `missing_deps` list to the skill
info dict:
```python
info = { "name": ..., "summary": ..., ... }
missing = check_deps(info)
info["healthy"] = len(missing) == 0
info["missing_deps"] = missing
skills.append(info)
```
This does NOT skip unhealthy skills — they're still discovered, but downstream
code can check `healthy` to decide how to present them.
**Test:** Create a skill with `[kiso.deps] bin = ["nonexistent_binary"]`. Call
`discover_skills()`. Verify the skill is returned with `healthy=False` and
`missing_deps=["nonexistent_binary"]`.

### M178 — Skill deps repair on server startup
**Problem:** P3
**Files:** `kiso/main.py` (lifespan) + new `kiso/skill_repair.py`
**Change:** Add a startup task that runs after `init_db` in the FastAPI lifespan.
For each installed skill that has `healthy=False`:
1. Check if `deps.sh` exists in the skill directory
2. Run `bash deps.sh` with a timeout (60s per skill, 180s total)
3. Log the result (success/failure)
4. Invalidate the skills cache so next discovery picks up the fix

```python
# kiso/skill_repair.py
async def repair_unhealthy_skills(skills_dir: Path | None = None) -> list[str]:
    """Re-run deps.sh for skills with missing binary deps. Returns list of repaired skill names."""
    from kiso.skills import discover_skills, invalidate_skills_cache, check_deps
    ...
```

Call from lifespan:
```python
from kiso.skill_repair import repair_unhealthy_skills
repaired = await repair_unhealthy_skills()
if repaired:
    log.info("Repaired skills: %s", repaired)
```

This auto-heals the image-vs-volume split on every container restart.
**Test:** Create a skill dir on the volume with a `deps.sh` that touches a marker
file. Mock `check_deps` to return missing deps initially. Call
`repair_unhealthy_skills()`. Verify `deps.sh` was executed (marker file exists).

### M180 — Planner guidance for broken skill recovery
**Problem:** P6
**Files:** `kiso/roles/planner.md`
**Change:** Add a rule in the Rules section about handling broken skills:
```
- If a skill task fails with a runtime error (missing binary, deps error, crash),
  do NOT retry the same skill immediately. Instead, plan: (1) exec task to
  reinstall the skill (`kiso skill remove NAME && kiso skill install NAME`),
  (2) retry the skill task, (3) msg task. If the skill is marked [BROKEN] in the
  Skills section, always reinstall before using.
```
**Test:** Read planner.md, verify the broken skill guidance is present.
