## Phase 9 — Recipes and connectors follow-up after tool validation rollout (M1032-M1033)

### M1032 — Recipes: strengthen metadata/applicability without adding runtime validator hooks

**Problem:** Recipes influence planning, but they are not executable tools
and should not inherit the same `validator.py` hook model. The real risk
surface is different: weak applicability metadata or vague planner-facing
instructions can cause irrelevant recipes to stay in scope, while useful
recipes may be excluded too aggressively.

**Goal:** Make recipe selection smarter and more predictable through
metadata/applicability improvements and tighter tests, without adding
runtime-style validator hooks or overcomplicating recipe loading.

**Constraints:**
- Do not introduce `validator.py`-style dynamic hooks for recipes.
- Prefer static metadata and clearer applicability rules over runtime
  code paths.
- Break old recipe frontmatter assumptions if they block a cleaner metadata model.
- Avoid widening planner context unless the metadata is genuinely useful
  for selection.

**Likely changes:**
- Audit recipe frontmatter support in the core and document the current
  effective fields.
- Add cleaner metadata fields where they materially help selection:
  - `applies_to`
  - `excludes`
  - `priority` or similar tie-break guidance only if needed
- Tighten recipe selection/exclusion behavior in the briefer path using
  static metadata first, before relying on free-form recipe text.
- Improve planner-facing recipe text only where an audit shows real
  ambiguity or drift risk.

**Files expected:**
- `kiso/recipe_loader.py`
- planner/briefing logic in `kiso/brain.py` only if selection needs
  metadata-aware filtering
- recipe docs and default preset recipes if their current text causes
  planner ambiguity
- tests such as:
  - `tests/test_recipe_loader.py`
  - `tests/test_brain.py`
  - `tests/functional/test_advanced_flows.py`

**Tests to add/update:**
- unit coverage for richer recipe frontmatter parsing and fallback when
  metadata is absent
- unit coverage for metadata-aware inclusion/exclusion behavior
- preserve one end-to-end recipe influence test, but keep most branching
  logic under unit tests

**Non-goals:**
- no plugin-style runtime validation hooks for recipes
- no broad rewrite of every recipe body
- no coupled connector/tool changes in this milestone

**Tasks:**
- [x] Audit current recipe frontmatter/text and identify actual
      applicability gaps
- [x] Add only the minimal metadata/extensions needed to improve
      selection quality
- [x] Add focused loader/brain tests for metadata-aware selection
- [x] Update docs/default recipes only where wording materially affects
      planner behavior

**Notes:**
- Added optional static recipe metadata only where it changes selection
  quality without introducing runtime hooks: `applies_to` and `excludes`
- Applied metadata filtering before recipes are exposed to the briefer or
  planner; future cleanup should prefer normalizing or dropping weak legacy
  recipe shapes rather than carrying them forward
- No default recipe corpus needed wording changes in this repo because no
  shipped `recipes/*.md` set currently exists

**Verification:**
- `uv run pytest tests/test_recipe_loader.py tests/test_recipes.py -q`
- `uv run pytest tests/test_briefer.py -q -k recipes`

### M1033 — Connectors: improve manifest clarity plus config/preflight validation

**Problem:** Connectors are not primarily selected like tools, so they
do not benefit from tool-style semantic arg validation. Their real
failure modes are configuration drift, missing auth/env prerequisites,
and unclear planner/runtime boundaries around what a connector can
actually deliver.

**Goal:** Harden connector behavior with clearer manifests,
stronger config/preflight validation, and narrower tests, without
introducing unnecessary planner-time complexity.

**Constraints:**
- Do not force connectors into the same hook model as tools.
- Focus on config/runtime/preflight checks rather than planner arg
  semantics.
- Keep startup/runtime overhead low; avoid heavy dynamic probing unless a
  connector is actually in use.
- Simplify connector install/discovery behavior if the current path is
  more legacy-shaped than useful.

**Likely changes:**
- Audit connector manifest text for planner/runtime clarity where it can
  affect delivery expectations.
- Add or tighten connector-side config validation helpers where missing.
- Improve preflight errors so missing tokens, credentials, or transport
  prerequisites fail early with actionable messages.
- Add narrow tests around config validation, manifest expectations, and
  preflight failure modes.

**Files expected:**
- `kiso/connectors.py`
- connector source repos under `../plugins/connector-*` only where the
  audit shows real config/preflight gaps
- connector tests in core and plugin repos

**Tests to add/update:**
- core discovery/manifest tests only if connector manifest requirements
  change
- connector plugin tests for config validation and missing-prerequisite
  failures
- keep transport/integration tests narrow; avoid duplicating full
  connector workflows

**Non-goals:**
- no `validator.py` hook model cloned from tools
- no planner-time semantic repair for connector payloads
- no broad connector refactor unless the audit uncovers a concrete
  structural issue

**Tasks:**
- [x] Audit connector manifests and runtime config/preflight behavior
- [x] Add missing config/preflight validation only where it improves
      failure quality materially
- [x] Add focused tests for connector config and prerequisite failures
- [x] Capture any connector manifest wording changes in plugin devplans
      if the audit shows planner-facing ambiguity

**Notes:**
- Audited the only current connector plugin (`connector-discord`) and kept
  the scope narrow: the real gap was fail-fast startup validation, not
  planner-facing manifest ambiguity or a need for tool-style hooks
- Added connector-side config validation for malformed URLs, invalid
  webhook port values, non-table channel mappings, non-numeric Discord
  channel IDs, and invalid session names before the runtime starts
- Normalized `webhook_address` by trimming a trailing slash so registered
  callback URLs stay stable
- No connector manifest wording change was needed after the audit, so no
  plugin-devplan wording follow-up was required beyond recording that
  result here

**Verification:**
- `uv run python -m pytest tests/test_config.py tests/test_connector.py tests/test_functional.py -q` in `../plugins/connector-discord`
- `uv run python -m pytest tests/test_session.py tests/test_webhook.py -q` in `../plugins/connector-discord`

## Phase 42 — Code cleanup and maintainability (M1236-M1241)

### M1236 — Unify fallback switching in `_run_text_role`

**Problem:** After M1232, the `LLMStallError` and `LLMError` (timeout)
handlers have near-identical 6-line fallback-switching blocks. If one
changes, the other gets forgotten.

**Fix:** Extract the common condition into a boolean
(`should_fallback = stall or timeout, with fallback available`) and use
a single fallback-switch block. Keeps the divergent behavior (stall
breaks if no fallback, timeout retries same model) but eliminates the
copy-pasted switching logic.

increases more than curator upgrade. But review quality directly
affects plan success — a bad review cascades into wrong replans.
