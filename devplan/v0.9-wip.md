### M1248 ‚Äî Fix fast-path chat timeout budget ‚úÖ

**Problem:** `TestF16ScoredFactRetrieval` (tests/functional/test_knowledge.py:373)
fails with "Fast path messenger failed: Messenger timed out after 180s".

**Structural bug (confirmed by code analysis):** `_fast_path_chat`
(loop.py:377) wraps the entire `_msg_task` in
`asyncio.wait_for(timeout=messenger_timeout)` where
`messenger_timeout = llm_timeout = 180s`. Inside `_msg_task_impl`:
1. Briefer: `asyncio.wait_for(run_briefer(...), timeout=30s)`
2. Messenger: `_run_text_role(max_retries=2)` ‚Üí up to 3 √ó `call_llm`

Budget math: briefer (30s) + messenger attempt 1 (up to 180s) = 210s.
The outer timeout (180s) < minimum inner budget (210s). The retry
mechanism (designed for resilience) can never execute. Even without
retries, if briefer takes 20s the messenger has only 160s left.

**Fix:** `chat_timeout = messenger_timeout + _BRIEFER_MSG_TIMEOUT + 30`
- Briefer: 30s budget (unchanged)
- Messenger: full llm_timeout budget (180s, unchanged)
- 30s headroom for DB queries, scored_facts search, etc.

**Tasks:**
- [x] Fix outer timeout in _fast_path_chat to include briefer headroom
- [x] 4032 unit tests pass
- [ ] Functional validation needed for F16 (requires `--functional` + `KISO_LLM_API_KEY`)

### M1249 ‚Äî Change classifier timeout fallback from "plan" to "chat"

**Problem:** `TestF20SpanishResponse` (tests/functional/test_core_flows.py:124)
fails because a Spanish QA question gets routed through the full planner
pipeline after classifier timeout, leading to cascading search timeouts.

**Root cause:** `_process_message` (loop.py:2544-2547) falls back to
`msg_class = "plan"` on classifier timeout. "plan" is the aggressive
path ‚Äî triggers planner, search, exec. For any non-planning question,
this cascades into timeouts and replans.

**Decision: change fallback to "chat".**

The fast-path messenger receives briefer-selected facts, scored facts,
session summary, recent messages, behavior rules, user language. It is
NOT a degraded path ‚Äî it's the same pipeline as chat_kb.

| Scenario | Normal class | Fallback "chat" | Acceptable? |
|---|---|---|---|
| Small talk | chat | Same | Yes |
| Knowledge query | chat_kb | Same (messenger+facts) | Yes |
| Simple action | plan | Messenger explains | Yes (user retries) |
| Complex planning | plan | Messenger explains | Yes (user retries) |
| Tool/install | plan | Messenger explains | Yes (user retries) |

**Why "chat" > "plan" as fallback:**
- Classifier timeout is RARE (30s is generous for a classification call)
- When it happens, "chat" gives a fast, bounded response
- "plan" cascades into search/exec timeouts (unbounded)
- For knowledge queries, "chat" is actually BETTER (messenger has facts)
- For planning requests, "chat" is honestly degraded but visible to user

**Impact assessment:**
- **Unit tests:** test_chat_message_skips_planner (7371),
  test_plan_message_goes_to_planner (7393) ‚Äî both mock the classifier,
  never exercise timeout path ‚Üí no impact
- **Functional tests:** classifier normally completes in <5s. Timeout
  path is only hit under OpenRouter load. No test relies on classifier
  timeout behavior ‚Üí no regression
- **Production:** on rare classifier timeout, user gets chat response
  instead of cascading failure. They can retry for planning.

**Tasks:**
- [ ] Change `msg_class = "plan"` to `msg_class = "chat"` in loop.py:2547
- [ ] Update log message to reflect new behavior
- [ ] Verify unit tests pass (3983+)
- [ ] Document: functional validation needed for F20 specifically

### M1250 ‚Äî Fix F39 browser tool install routing in Docker

**Problem:** `TestF39ToolInstallAndUse` (tests/functional/test_advanced_flows.py:175)
fails because the planner generates `apt install wkhtmltopdf` instead of
`kiso tool install browser`. The replan loop tries different unavailable
apt packages (wkhtmltopdf, chromium-browser) until timeout.

**Root cause (needs verification):** The install routing in
`_classify_install_mode` (common.py:272) should route "browser" to
`_INSTALL_MODE_KISO_TOOL` when registry_hint_names contains "browser".
The registry IS fetched successfully (log shows HTTP 200 for
registry.json). But the routing still produces system package commands.

Possible causes to investigate:
1. **Registry cache expires between stages.** TTL is 300s
   (`_REGISTRY_TTL`), test stages run sequentially. If Stage 1 caches
   at T=0 and Stage 2 runs at T>300, the cache expires. A second fetch
   might fail (GitHub rate limit, network flap), returning empty
   registry. Then "browser" is not in hint_names ‚Üí falls through to
   system package.

2. **`_extract_install_target` fails on Italian.** The message is
   "s√¨, installa il tool browser". The regex `_INSTALL_TARGET_RE` or
   `_NAMED_TOOL_TARGET_RE` might not extract "browser" correctly from
   Italian syntax, returning None ‚Üí mode=NONE ‚Üí planner decides freely.

3. **Planner ignores routing context.** Even with correct routing, the
   install mode context is injected as text into the planner prompt.
   The LLM might ignore it and generate apt commands anyway. This would
   be a prompt clarity issue.

4. **discover_tools() empty in replan.** Log shows
   `discover_tools() returned empty ‚Äî no tools available for planner`
   during replan. If Stage 2 installs browser but it's not discoverable
   (wrong KISO_DIR, tool dir not scanned), Stage 3 planner won't see it
   and will try system alternatives.

**Investigation plan:**
- [ ] Add debug logging to _classify_install_mode to trace routing for "s√¨, installa il tool browser"
- [ ] Check _extract_install_target with Italian input
- [ ] Check registry_hint_names population path through sysenv._load_registry_hints
- [ ] Check discover_tools() in isolated KISO_DIR (functional test fixture)
- [ ] Verify browser tool's kiso.toml is discoverable after install in test environment

**Fix (after investigation):**
- [ ] Address the identified root cause
- [ ] Ensure tool install routing works reliably with Italian input
- [ ] Verify F39 passes end-to-end
