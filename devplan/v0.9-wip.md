# kiso-run/core — v0.9 Development Plan

## Problem Statement

Kiso now has broad test coverage across unit, bash, integration, Docker,
live, functional, and plugin suites, but the remaining risk is no longer
"untested modules". The risk is that several runtime guarantees explicitly
promised by the docs are still verified only indirectly, only at unit level,
or only through weak smoke oracles — and that the suite itself has grown
large enough to slow contributors down without delivering proportional
confidence.

v0.9 should therefore focus on **confidence alignment**: the highest-risk
contracts in startup recovery, worker concurrency, connector protocol,
workspace file handoff, plugin lifecycle, and secret containment need
tests at the level where they can actually fail in production. In parallel,
the suite needs a one-time investment in performance and a deliberate
audit/prune so future tests build on a fast, lean baseline.

## Current Test Topology

Kiso already has a strong suite pyramid:

- **Unit**: dominant layer; strong coverage for planner/worker helpers, CLI,
  storage, security, webhook logic, tool manifests, and many regressions.
- **Bash**: good coverage for `install.sh` and `kiso-host.sh`.
- **Integration**: currently thin but valuable; mocked LLM + real ASGI app,
  ideal for API/runtime protocol guarantees.
- **Docker**: focused isolation/sandbox checks.
- **Live**: verifies role compliance and selected runtime semantics with real
  LLMs, but should stay targeted because it is slow and provider-fragile.
- **Functional**: real end-to-end user scenarios; strongest for observable
  file/tool/runtime behavior, but must avoid weak wording-only assertions.
- **Plugin / interactive / destructive**: useful as smoke and acceptance,
  but not substitutes for deterministic runtime tests.

The suite today: **104 test files, ~66k lines, no parallelization**, all
integration fixtures function-scoped. The default suite is fast enough but
adding 17+ new milestones on top of the current baseline would compound an
already significant cost.

## Problems

### P1 — Runtime guarantees are documented more strongly than they are tested

Docs in `flow.md`, `connectors.md`, and `tools.md` promise behavior such as
startup message recovery, atomic worker spawn, polling fallback, `uploads/`
handoff, and scoped ephemeral secrets. Some of these are covered only by
unit tests on helpers or store functions, not by tests that exercise the
actual runtime seams where they could break.

### P2 — Integration tests underuse their best role

The integration layer is the correct place to validate ASGI routes,
worker spawning, queueing, webhook behavior, and connector protocol with
mocked LLMs and deterministic assertions. Today it covers basic request/flow
cases, but not several of the concurrency and recovery contracts that most
need a host-level, non-LLM oracle.

### P3 — Plugin and connector lifecycle confidence is split across weak layers

Tools/connectors have good unit coverage for manifest validation and CLI
behavior, and plugin smoke tests validate clone/build/test. But update,
refresh, daemon lifecycle, restart/backoff behavior, and connector
protocol recovery are not yet covered as end-to-end operational flows.

### P4 — File and secret boundaries need stronger end-to-end evidence

Kiso's value proposition depends on correct movement of files and secrets
across plans and tools: `uploads/`, `pub/`, `plan_outputs`, scoped
`session_secrets`, and published URLs. Many helper-level tests exist, but
the highest-risk boundary failures are "everything looked valid in unit
tests, but the real runtime handed the wrong path, leaked a secret, or
forced a re-download".

### P5 — Some suites still use weaker oracles than the runtime now supports

The structured-runtime redesign introduced stronger state surfaces:
normalized plan/task contracts, canonical task results, plan/task DB rows,
published artifact URLs, worker/process state, and classified replan
behavior. New tests in v0.9 should prefer these oracles over wording-based
assertions whenever possible.

### P6 — Long-tail surfaces are covered unevenly or only as TODOs

Negative install paths for nonexistent plugins are explicitly TODO in
`docs/testing-live.md`, and the registry matrix beyond the default preset
trio (browser/ocr/aider) is partial. These are less urgent than runtime
recovery, but they should still be captured in the plan so they do not
remain perpetual TODOs.

### P7 — Cross-layer adversarial paths rely on layered defenses, not isolated filters

Kiso's docs and architecture explicitly rely on layered defenses
(paraphraser, reviewer, replan classification, sanitization). Those layers
are valuable only if the important cross-layer handoffs are tested as a
system, not only as isolated helper behavior. Today the handoffs themselves
have no end-to-end tests.

### P8 — The suite is large, function-scoped, and unparallelized

104 files, 66k lines, no `pytest-xdist`, all integration fixtures
function-scoped. Adding 17+ milestones on top of the current baseline would
make local TDD painful and CI slower. The single highest-leverage change
v0.9 can make is paying the perf debt once, so the new tests inherit the
speedup instead of compounding the slowness.

### P9 — Accumulated test redundancy in the largest unit modules

`tests/test_worker.py` has 613 tests in 13.5k lines and `tests/test_brain.py`
825 tests in 10.5k lines — sizes that almost certainly contain duplicate
scenarios accumulated over many milestones. v0.9 should end with a
deliberate audit pass — not a blind delete, but a structured review that
proposes specific cuts/merges for human approval.

## Test Design Principles For v0.9

1. **Test the guarantee at the seam where it can fail.**
   Startup recovery belongs in integration/runtime startup tests, not only
   store tests. Connector polling fallback belongs in integration protocol
   tests, not only in connector docs.
2. **Prefer deterministic host-level assertions over live LLM assertions.**
   Use integration for worker/process/protocol guarantees. Use live or
   functional only when the property depends on actual model behavior.
3. **Use observable runtime state as the oracle.**
   Prefer queue state, worker count, task rows, webhook deliveries, file
   existence, published URLs, and tool stdin payloads over output prose.
4. **Audit before adding.**
   Existing suites already cover meaningful pieces (see "Items intentionally
   NOT in v0.9" at the end). Each milestone that touches their territory
   was checked against the actual code before being added to this plan.
5. **Build durable harnesses once, reuse them everywhere.**
   M1268 exists specifically so subsequent milestones do not each
   reinvent fake tools, fake connectors, webhook collectors, or
   spawn-race fixtures.
6. **Pay perf debt before adding cost.**
   M1267 must land before any new test milestone, so the additions
   inherit xdist parallelism and tmpfs storage.
7. **Keep smoke tests smoke-sized.**
   Plugin clone/build tests and external-service checks are useful, but do
   not upgrade them into blocking semantic coverage unless the oracle is
   robust against external drift.

## Phase 0 — Foundations: performance and harness (M1267-M1268)

### M1267 — Test suite performance baseline, xdist, and tmpfs ✅

**Problem:** P8.

**Files:** `pyproject.toml`, `tests/conftest.py`,
`tests/integration/conftest.py`, `tests/functional/conftest.py`,
`utils/run_tests.sh`, `tests/README.md`.

**Verified facts that shape this milestone:**
- `kiso/store/setup.py::init_db(db_path: Path)` opens an aiosqlite
  connection in WAL journal mode and runs ALTER TABLE migrations.
  In-memory SQLite (`:memory:`) is **not** a drop-in: WAL is degenerate
  for in-memory databases, and the test suite uses `tmp_path` heavily.
  The realistic fast-storage path is **tmpfs**, which is a real
  filesystem and supports WAL and ALTER TABLE without changes.
- `kiso/main.py` has module-level mutable state: `_workers` (dict at
  line 302), `_worker_phases` (dict at line 305), `_rate_limiter`
  (singleton at line 113). xdist isolates per-subprocess so cross-worker
  state is fine, but tests within the same xdist worker share these
  dicts. The cleanup pattern from `tests/test_inflight.py::_cleanup_worker`
  (pop `_workers[session]` and `_worker_phases[session]`, cancel the
  task) is the correct shape — promote it into a shared fixture.
- All fixtures in `tests/integration/conftest.py:175-272` are
  function-scoped. Promoting app/runtime init to session/module scope
  is the largest single fixture-level win.

**Change:**

1. **Baseline first.** Run the unit suite with `pytest --durations=20`
   and record the top-20 slowest tests and total wall time. Same for
   integration. Commit the baseline numbers in `tests/README.md` so
   subsequent gains are measurable.
2. **Add `pytest-xdist`** as a dev dependency and wire `-n auto` into
   `utils/run_tests.sh` for the unit and integration tiers. Add a
   shared autouse fixture (in `tests/conftest.py`) that pops any
   leftover entries from `kiso.main._workers` / `_worker_phases` after
   each test, modeled on `tests/test_inflight.py::_cleanup_worker`.
   Where shared state cannot be cleaned, use `--dist=loadfile` so
   tests in the same file run on the same worker.
3. **tmpfs for the temp tree.** Wire
   `pytest --basetemp=/dev/shm/pytest-kiso` into `utils/run_tests.sh`
   for the unit and integration tiers (with a Linux-only guard and a
   filesystem fallback for macOS contributors). This gives WAL-friendly
   storage at RAM speed without touching `init_db` or any test code.
4. **Promote function-scoped fixtures to session/module scope where
   safe.** Inspection target: `tests/integration/conftest.py:175-272`.
   Candidates for promotion: app/runtime initialization, role prompt
   loading (`kiso/roles/*.md`), planner module assembly cache. Reset
   per-test data via small helpers, not by re-initializing the whole
   stack.
5. **Cache role prompt loading.** `kiso/roles/*.md` are read from disk;
   the planner module assembler (`kiso/brain/planner.py`) reassembles
   modules with a budget cap. Add a session-scoped fixture or
   `functools.lru_cache` so each test does not re-pay this cost.
6. **Tighten the default tier.** No test in the default suite (no
   flags) should take >1s. Mark anything slower with the appropriate
   opt-in marker (`functional`, `integration`, `extended`) and verify
   `pytest_collection_modifyitems` in `tests/conftest.py` handles them.
7. Re-run baselines, record before/after numbers in `tests/README.md`,
   and document the chosen parallelization strategy so future tests
   stay compatible.

**Why:** Adding 17 more milestones on top of an unparallelized 66k-line
suite makes local TDD painful. Doing this first means every Phase 1-7
addition inherits the speedup.

**Test:** Self-evident — measure wall time before and after, on the
same machine, with the same flags. Document the delta.

**Outcome (2026-04-08):**
- Unit tier: **137s → 42s (3.2x)** with `pytest-xdist -n auto` +
  `--basetemp=/dev/shm/...`. 4045 tests still passing, zero state
  isolation issues.
- Integration tier: **10s → 6s (1.8x)** with the same flags.
- Wired into `utils/run_tests.sh` for tier 1 (unit) and tier 3
  (integration), with a Linux-only guard on tmpfs and a graceful
  fallback to default basetemp on macOS.
- Baseline + delta documented in `tests/README.md`.

**Deferred (intentional, with reasons):**
- *Promote integration fixtures to session/module scope* — xdist
  already delivered 3.2x. Promoting `kiso_client` and friends is
  risky because they all initialize the global `app` state via
  `_init_app_state(app, config, db)`; the gain would be marginal and
  the regression risk is real. Defer to a future perf cycle if the
  base time drifts back up.
- *Cache role prompt loading* — verified already cached at module
  level via `kiso/brain/prompts.py:11` `_prompt_cache` dict. No work
  needed.
- *Tighten default tier so no test takes >1s* — the slowest tests
  (`TestInitKisoDirs` 5-6s each, 7 tests) do real filesystem ops with
  PRAGMA WAL. Forcing them under 1s requires substantial test
  refactor and is out of scope for a perf-only milestone. The xdist
  parallelism already addresses the wall-clock pain.
- *Autouse cleanup for `_workers`/`_worker_phases`* — not needed
  because xdist subprocess isolation handles cross-worker, and
  same-worker test ordering already works (no xdist failures
  observed). M1268 may add this as a defensive net for the new
  spawn-race tests.

### M1270 — Same-session worker spawn atomicity (covers /msg and cron) ✅

**Problem:** P1, P2.

**Files:** `tests/integration/test_runtime_concurrency.py` — new
module; uses the M1268 spawn-race barrier.

**Verified facts that shape this milestone:**
- `kiso/main.py:472` shows the cron scheduler calls
  `queue = _ensure_worker(session, db, config)` — the **same** function
  that the `/msg` route uses to spawn workers. So cron-fired jobs and
  user `/msg` requests share the spawn check, and racing them on the
  same session is one race surface, not two.
- `tests/test_inflight.py` covers the **opposite** case: a `/msg`
  arriving when a worker is **already busy** (in-flight classification:
  stop / independent / update / conflict). It uses `_make_busy_worker`
  to pre-create the worker. The boundary with this milestone must be
  stated in the new module's top docstring so future contributors do
  not duplicate either side.
- One subtle gap: cron pushes via `queue.put(msg_payload)` directly
  (`kiso/main.py:473`), so a cron job firing while a worker is busy
  **bypasses** inflight classification entirely. Whether that is a bug
  or by-design is a design question, not a test gap — the milestone
  documents the current behavior with one regression test and flags
  the question for product review, it does not change behavior.

**Change:**

1. Fire multiple `POST /msg` requests concurrently against the same
   session.
2. Use the M1268 spawn-race barrier (an `asyncio.Event` monkeypatched
   into `_ensure_worker`) to widen the race window deterministically.
3. Assert:
   - exactly one worker is created for that session
   - every message is processed exactly once
   - no duplicate tasks or duplicate webhook deliveries are emitted
   - session queue order stays coherent
4. Add a cross-session control test showing concurrent posts to
   different sessions create independent workers as expected.
5. **Cron + /msg interleave test**: drive a cron-fired enqueue and a
   concurrent `/msg` against the same session, and assert the spawn
   check still emits exactly one worker.
6. **Cron + busy worker regression**: with a busy worker already
   present, cron-enqueue a message and assert it lands in the queue
   without invoking inflight classification (current behavior). Add a
   `# DESIGN NOTE:` comment explaining this is the bypass surface and
   that any future change to it must update this test.

**Why:** `flow.md` treats atomic check-and-spawn as a hard runtime
guarantee. This is exactly the kind of race that can pass unit tests
and still fail in production if the runtime seam drifts.

**Test:** Integration-only, mocked LLM, strong counts/oracles.

## Phase 9 — Functional test robustness (M1286)

### M1286 — Drive-until-done install loop and command-vs-data separation ✅

**Problem:** Two functional test failures observed on the v0.9
shipping run (`test_browser.py::TestF1BrowserInstall::
test_browser_install_flow` and `test_multitool.py::TestF17FullPipeline
::test_screenshot_ocr_aider_exec_msg`) revealed two **general** test
robustness gaps that affect any test driving Kiso through real LLM
behavior. Both failures were diagnosed as test issues, not core
regressions; both fixes must be **general** so they do not constrain
the planner to specific phrasings or specific page contents.

**Files:** `tests/functional/test_browser.py`,
`tests/functional/test_multitool.py`,
`tests/functional/conftest.py` (shared install-flow helper),
plus a grep sweep for similar patterns elsewhere.

**Verified diagnoses:**

1. *Install-flow helper assumes a fixed number of turns.*
   `_run_with_install_flow` in `test_browser.py` and
   `_run_with_tool_install` in `test_multitool.py` both assume the
   install conversation resolves in at most 2 user turns: original
   request → "sì, installa il tool X". This is correct only if Turn
   1 actually produces an install proposal. The planner is free to
   attempt a workaround on Turn 1 (e.g. answer the request via a
   curl/echo exec without proposing install), in which case Turn 2
   becomes the *first* install proposal and the test asserts before
   any actual install runs. The same test re-run produced PASS on
   the very next attempt — the test is non-deterministic by LLM
   behavior, with no underlying core regression.
2. *Substring match on data-bearing task fields.* The assertion
   `assert "curl" not in task_blob and "wget" not in task_blob` at
   `test_multitool.py:160` is intended to catch a planner that
   re-downloads a file instead of reusing the local one. But
   `task_blob` is built from `(task.detail or "") + (task.command or
   "")` joined across all tasks of a plan, and the `detail` field
   of an exec task often contains a heredoc body with arbitrary
   stdin data. In this run, the OCR text of the Wikipedia Python
   page (which contains "curly brackets" twice — verified
   deterministically with
   `curl https://en.wikipedia.org/wiki/Python_(programming_language)
   | grep -c "curly"` → 2) was passed as a heredoc body. The
   substring `curl` matched `curly`, producing a false positive.

**Change:**

1. **Drive-until-done install helper.** Refactor the shared install
   flow into a single function in `tests/functional/conftest.py`:

   ```python
   async def drive_install_flow(
       run_message,
       tool_name: str,
       prompt: str,
       *,
       max_turns: int = 4,
       timeout: float = LLM_INSTALL_TIMEOUT,
   ):
       """Drive a conversation forward until *tool_name* is
       installed (or max_turns is reached), then re-issue the
       original prompt so the result reflects the installed-tool
       path. Does not constrain what the planner does on any turn —
       just keeps the conversation moving the way a real user would.
       """
       result = await run_message(prompt, timeout=timeout)
       turns_used = 1
       while not tool_installed(tool_name) and turns_used < max_turns:
           result = await run_message(
               f"sì, installa il tool {tool_name}", timeout=timeout,
           )
           turns_used += 1
       if tool_installed(tool_name):
           result = await run_message(prompt, timeout=timeout)
       return result
   ```

   - Replace `_run_with_install_flow` (in `test_browser.py`) and
     `_run_with_tool_install` (in `test_multitool.py`) with calls
     to `drive_install_flow`.
   - `TestF1BrowserInstall::test_browser_install_flow` currently has
     its own ad-hoc 2-turn loop that asserts directly on the
     "tool_installed" state. Replace it with a call to
     `drive_install_flow` followed by the same assertion. The test
     will then pass regardless of which path the planner picks on
     Turn 1.
   - The helper does NOT prescribe what the planner should do on
     any turn. The planner is free to propose an install, install
     directly, attempt a workaround, or change strategy mid-flow.
     The helper simply drives the conversation forward. This is
     also closer to how a real human user would behave: keep asking
     until the tool works.

2. **Command-vs-data assertion separation.** Introduce a small
   helper in `tests/functional/conftest.py`:

   ```python
   import re

   def assert_no_command_word(tasks, words):
       """Assert no exec task has a command matching any of *words*
       as a whole word. Only the `command` field is inspected — task
       `detail` and heredoc bodies often contain arbitrary data
       (OCR output, API responses, planner reasoning) that may
       incidentally contain substrings matching command names."""
       pattern = re.compile(
           r"\b(" + "|".join(re.escape(w) for w in words) + r")\b"
       )
       for t in tasks:
           if t.get("type") != "exec":
               continue
           command = t.get("command") or ""
           m = pattern.search(command)
           assert m is None, (
               f"forbidden command word {m.group(0)!r} in exec task "
               f"command: {command[:200]}"
           )
   ```

   Replace the failing assertion at `test_multitool.py:160`:
   ```python
   assert_no_command_word(last_plan_tasks, ["curl", "wget"])
   ```

   Properties of the new helper:
   - inspects only the `command` field (the actual shell command),
     never `detail` (which can contain reasoning text or heredoc
     bodies)
   - uses `\b...\b` word boundaries so `curly`, `libcurl`,
     `pycurl`, `wgetopt`, etc. do not match
   - reusable for any future "no fetch / no recompile / no rm"
     style assertion

3. **Suite-wide sweep.** Grep `tests/functional/` for similar
   substring assertions on `task_blob`-style joined fields
   (`detail + command`, `output` joined across tasks, etc.) and
   convert any found to use `assert_no_command_word` (or an
   equivalent word-boundary helper) where the intent is to catch
   shell commands. Do not change assertions that legitimately scan
   *output* text for substrings.

4. Re-run the originally failing tests through
   `./utils/run_tests.sh s` to confirm green:
   - `test_browser_install_flow` should be deterministic regardless
     of which path the planner picks on Turn 1.
   - `test_screenshot_ocr_aider_exec_msg` should be deterministic
     regardless of which words the OCR text contains.

**Why these fixes are general, not overfitted:**

- The drive-until-done loop does NOT constrain the planner. It
  does not require an install proposal on Turn 1, does not pin a
  specific phrasing, and does not encode any knowledge about how
  the planner *should* behave. It just drives the conversation
  forward the way a real user would and accepts any path the
  planner chooses. This preserves Kiso's generalist nature: the
  test proves the install flow eventually converges, not that it
  follows a hard-coded script.
- The command-vs-data helper enforces a general principle: shell
  command assertions belong on the `command` field with word
  boundaries, not on free-form `detail` text or heredoc bodies.
  It applies to any future test that wants to forbid a shell
  command pattern, not just curl/wget. It does not pin the test
  to the current Wikipedia page content or to the current OCR
  output, and it does not change which pages or which tools the
  test exercises.
- Neither change touches kiso core code. The planner, worker,
  install routing (M1234), and codegen guardrail (M1233) are
  unchanged. Only test infrastructure is hardened.

**Test:** Run both originally failing tests via
`./utils/run_tests.sh s "tests/functional/test_browser.py::
TestF1BrowserInstall"` and
`./utils/run_tests.sh s "tests/functional/test_multitool.py::
TestF17FullPipeline"` on multiple consecutive runs to confirm the
flakiness is gone. Then run the full functional + extended tier via
`./utils/run_tests.sh --auto --functional --extended`.

### M1289 — Roles owned by the user dir, with reset hatch ✅

**Problem:** Today `kiso/brain/prompts.py::_load_system_prompt(role)`

   customize them.

`~/.kiso/roles/` at init (without overwriting existing user files),
make `_load_system_prompt` read **only** from the user dir (no
package fallback), and provide a `kiso role reset` command for

**Files:** `kiso/main.py` (`_init_kiso_dirs` extension),

## Phase 12 — `investigate` as 4th classifier category (M1290)

### M1290 — Add `investigate` as a runtime classifier mode ✅

**Problem:** Today the classifier (`kiso/brain/common.py`, around
line 1462) has 3 categories: `plan`, `chat_kb`, `chat`. There is a
real semantic class of user messages that doesn't fit cleanly into
any of them and currently gets routed as `plan`:

- *"Why is nginx returning 502?"* → user wants understanding, not a
  fix
- *"Show me the current config"* → user wants live state, not a
  change
- *"Database is down"* (isolated report) → user wants diagnosis
  before deciding what to do
- *"Is service Y running?"* → user wants live evidence

These all share a structural property: **the answer requires
running commands or reading files but NOT changing them**, and the
right response is *evidence + explanation*, NOT *propose-and-apply
a fix in the same plan*. Today, kiso routes them through `plan`,
which gives the planner license to propose and apply mutations
unprompted — losing the diagnose-first behavior that makes a
healthy debugger/operator.

The `chat_kb` category already established the precedent that a
sub-class of messages deserves first-class classifier treatment when
its handling differs structurally. `investigate` is the same shape:
a stable, well-defined intent class that should be visible at the
classifier level, not buried as a hint inside `plan`.

This was decided after rejecting the recipe-based approach (recipe
would be opt-in per preset, depend on briefer selection, and rely
on the planner honoring an additive context hint — fragile on every
axis vs. a structural classifier category).

**Files (lean scope):** `kiso/brain/common.py` (classifier prompt +
return validation), `kiso/worker/loop.py` (dispatch propagation),
`kiso/brain/planner.py` (`run_planner` flag + modular prompt
inclusion), `kiso/roles/planner.md` (new modular section), tests.

**Verified facts that shape this milestone:**

- The classifier prompt currently lists 3 categories with examples
  in a single string in `kiso/brain/common.py`. Adding a 4th is a
  prompt edit, not a code refactor.
- The planner role uses the modular prompt mechanism
  (`<!-- MODULE: xxx -->` markers, parsed in
  `kiso/brain/prompts.py:_load_modular_prompt`). New sections can
  be added without changing the loader code — just the role file
  and the module list passed by `run_planner`.
- The worker dispatch in `kiso/worker/loop.py::_process_message`
  already branches on classifier output (chat → fast-path, plan →
  full plan). Adding a 4th branch is a few lines.
- `validate_plan` in `kiso/brain/planner.py` will NOT be touched.
  Per the project rule "no heuristics", we trust the planner role
  module to enforce the read-only contract. If the LLM violates it,
  the user sees the proposal in the messenger output and corrects
  with a follow-up — same trust model as the rest of the runtime.

**Change:**

1. **Classifier prompt** (`kiso/brain/common.py`): add 4th category
   to the prompt string with definition + 3-4 boundary examples.
   Final return values become
   `{"plan", "chat_kb", "chat", "investigate"}`. Update the
   validation list (the parser that reads `category:Language` and
   accepts only known values).

   Boundary examples to include in the prompt:
   - *"Why is X failing?"* → investigate
   - *"Fix the X failure"* → plan
   - *"X is failing"* (isolated report, no fix verb) → investigate
   - *"Show me X's current state"* → investigate
   - *"What do you know about X?"* → chat_kb (memory, not system)
   - *"Restart X"* → plan
   - *"X is down, restart it"* → plan (imperative wins)

   When in doubt between investigate and plan, the prompt
   instructs the classifier to **prefer investigate** — it
   preserves user autonomy (better to ask "want me to fix it"
   than to act unprompted).

2. **Worker dispatch**
   (`kiso/worker/loop.py::_process_message` area): when
   `category == "investigate"`, call `run_planner(...)` with the

   - When True, include the new `"investigate"` module in the

4. **`kiso/roles/planner.md` new module**: append a section

   > # Investigate mode
   >
   > This is an *investigate* request. The user wants to understand

   > 1. Plan only **read-only** tasks: `exec` for read-only shell

## Phase 13 — chat_kb → investigate transparent fallback (M1291)

### M1291 — Pre-flight chat_kb facts check, fall back to investigate when empty ✅

**Problem.** The classifier added in M1290 distinguishes
`chat_kb` (info from stored memory) from `investigate` (live
system state) using only the message phrasing and the existing
`Known Entities` hint. But there is a real semantic gap: when
the user asks about something that *could* be in memory but
is not yet, the classifier picks `chat_kb` because the entity
name matches the hint — and then the chat_kb path retrieves
zero facts. The messenger is called anyway, with an empty
briefing context, and produces either "I don't know" or LLM
general knowledge that has nothing to do with the actual
system state.

The right behavior: when chat_kb retrieval comes back empty,
tell the user explicitly and re-route the same message
through the investigate planner so kiso actually inspects
the live system. The classifier *cannot* know in advance
whether a fact exists; the worker *can* check before
committing to the chat_kb response.

This is the missing link the user identified after M1290:
*"come fa il classifier a sapere se quelle info le abbiamo
già o vanno indagate?"*

**Investigation results (2026-04-09):** The original draft of
this milestone said the fallback should live "in
`_fast_path_chat` or whichever helper assembles the chat_kb
response". After reading the code that turned out to be the
wrong layer. The actual situation:

- The classifier dispatch lives in `handle_message()` in
  `kiso/worker/loop.py` around lines 2540-2571. The branch
  `if msg_class in ("chat", "chat_kb"):` calls
  `_fast_path_chat(...)` which immediately commits a plan
  row + msg task row to the DB
  (`kiso/worker/loop.py:323-446`, plan creation around
  lines 349-354).
- The actual facts retrieval does NOT happen in
  `_fast_path_chat`. It happens downstream in
  `_msg_task_impl()` in `kiso/worker/message_flow.py:206-214`,
  via `search_facts_scored(...)`, *after* the briefer has
  already run and produced `entity_id` and `relevant_tags`.
- That means by the time the "is the result empty" signal
  exists, the plan and msg task are already persisted, the
  briefer has already burned an LLM call, and rolling back
  to take the investigate path requires either committing
  garbage rows or deleting them after the fact.

**Design decision: pre-flight check at the dispatcher.**
Instead of trying to short-circuit the chat_kb path mid-flight,
do a cheap deterministic facts query at the dispatcher *before*
calling `_fast_path_chat`. If `msg_class == "chat_kb"` and the
pre-flight returns zero facts, persist the transition message,
reassign `msg_class = "investigate"`, and fall through to the
existing investigate branch at `loop.py:2617`. No rollback, no
refactor of `_msg_task_impl`, no new sentinels.

**Trade-off — explicit and accepted.** The pre-flight uses raw
keywords from the user message (`content.lower().split()[:10]`,
the same pattern `_msg_task_impl` already uses on
`message_flow.py:211`). The normal chat_kb path uses
`entity_id` + `relevant_tags` derived from the briefer, which
is strictly more precise. So the pre-flight can produce false
negatives: the briefer would have found something, the
keyword search does not, and the user gets an investigate plan
instead of a fast chat_kb response. The failure mode is
*"slightly more verbose than ideal"*, not *"wrong answer"* —
the investigate plan still answers the user's question, just
with read-only shell commands instead of a memory lookup. This
is an acceptable cost for avoiding all the rollback complexity
of the alternative.

The reverse failure mode (pre-flight finds something the
briefer would have rejected) is harmless: the chat_kb path
runs as today.

**Files touched.**

- `kiso/worker/loop.py` — `handle_message()` dispatch site
  around lines 2540-2571: add the pre-flight check between
  the classifier call and the existing
  `if msg_class in ("chat", "chat_kb"):` branch.
- `kiso/store/sessions.py` — used as-is via `save_message()`
  at line 63 for the transition message. No changes.
- `kiso/brain/planner.py` — used as-is. `run_planner(...)`
  already accepts `investigate=True` (M1290). No changes.
- `kiso/store/facts.py` (or wherever `search_facts_scored`

## Phase 14 — Roles UX, naming, and discoverability (M1292-M1294)

**Context.** After M1289 the role prompt files live under
`~/.kiso/roles/` and are owned by the user, and after M1290 there
are 12 distinct roles in active use (`planner`, `messenger`,
`worker`, `reviewer`, `briefer`, `curator`, `consolidator`,
`classifier`, `inflight-classifier`, `paraphraser`,
`summarizer-session`, `searcher`). An audit of role usage on
2026-04-09 confirmed that **all 12 roles are alive** — every
`.md` file is loaded by exactly one Python entry point and
called from the worker loop, message flow, planner, or
review flow. There are no dead roles to delete.

The audit did surface three real UX/clarity problems that are
worth a dedicated phase:

1. **Discoverability is zero.** There is no `kiso roles`
   command. The user only learns that roles exist by opening
   `~/.kiso/config.toml` `[models]`, by triggering an error
   that mentions `kiso role reset {role}`, or by listing
   `~/.kiso/roles/`. After M1289 made roles user-owned, the
   absence of a discovery surface is a visible gap.

2. **Naming inconsistencies between role filenames and
   Python entry points.** The most jarring case is
   `worker.md` → `run_exec_translator()`, which collides with
   the unrelated `kiso/worker/` *module* (the actual task
   executor). Other smaller gaps: `classifier.md` →
   `classify_message()`, `inflight-classifier.md` →
   `classify_inflight()`, and `summarizer-session.md` whose
   `-session` suffix is not reflected in `run_summarizer()`
   (and no other summarizer variant exists). None of these
   break anything, but together they make jumping between
   the prompt and the call site noisier than it should be.

3. **`classifier` and `inflight-classifier` may overlap.**
   Two roles, two prompts, two functions, same default
   model (`gemini-2.5-flash`), classifying messages at two
   different points in the lifecycle. They might genuinely
   need to be separate, or they might be a candidate for
   consolidation behind a `phase` parameter. The audit did
   not compare the prompts; that comparison is the milestone.

This phase addresses all three, in dependency order: ship the
discovery surface first (it has standalone value), then the
internal renames (which benefit from the registry built in
M1292), then the investigation-only milestone for the
classifier overlap.

**What is explicitly out of scope for v0.9:**

- No new roles. No removal of any existing role.
- No change to model defaults in `_MODEL_METADATA`.
- No REST endpoint for role management. Roles stay internal
  infrastructure; only the CLI surface is added.
- No change to the planner prompt budget or modular markers.
- No touching `~/.kiso/roles/` content as part of M1292
  beyond what the user already gets after M1289.

### M1294 — `classifier` vs `inflight-classifier` consolidation review ✅

**Status: investigation only.** No code changes are part
of this milestone — the deliverable is a written decision
appended to this milestone, plus a follow-up milestone
*if and only if* the investigation concludes that
consolidation is the right move. If the conclusion is
"keep them separate", that decision is recorded here and
the milestone closes without code.

**Problem.** Two roles classify user messages:
`classifier.md` (initial routing into `plan`,
`investigate`, `chat_kb`, `chat`) and
`inflight-classifier.md` (in-flight routing for messages
that arrive while a task is already running:
`stop`/`update`/`independent`/`conflict`). Both default
to `gemini-2.5-flash`. They live in different files, are
loaded by different functions, and are documented in two
places. It is not obvious whether the duplication is
load-bearing or accidental.

**What to investigate.**

- Read both prompts in full. Quantify how much text is
  shared (intro framing, output schema framing, examples
  framing) vs. how much is genuinely phase-specific
  (initial categories vs. inflight categories).
- Check whether the two functions share any helper code
  in `common.py` that could become the shared substrate
  if the prompts are merged.
- Consider whether merging behind a `phase=initial|inflight`
  parameter would: (a) reduce maintenance cost, (b) risk
  cross-contamination of categories, (c) make the prompt
  too long to stay in the same model's optimal range.
- Check whether the inflight categories are stable enough
  that they belong in the same prompt, or whether they
  are likely to diverge further (e.g., new
  `pause`/`resume` categories on the inflight side that
  would have no analogue on the initial side).

**Decision criteria.**

- **Merge** if: shared framing >60% of total prompt, no
  cross-contamination risk visible in the existing
  categories, and the merged prompt fits comfortably in
  the existing budget.
- **Keep separate** if: phase-specific text dominates,
  there is plausible drift between the two category sets
  in the next cycle, or the merged prompt would push the
  classifier into a slower or more expensive model.
- **Default to "keep separate"** if the investigation is
  inconclusive. The current setup works; consolidation
  is a quality-of-life refactor, not a bug fix.

**Deliverable.** A written conclusion appended to this
milestone in the devplan, with the percentages and the
decision. If the decision is "merge", a follow-up
milestone (e.g. `M1295`) is added to v0.9-wip.md or to
the next cycle, depending on remaining v0.9 budget. The
follow-up is **not** auto-created; it is proposed to the
user first.

**Tasks.**

- [ ] Read `classifier.md` and `inflight-classifier.md`
  in full and quantify the percentage of shared framing
  vs phase-specific text.
- [ ] Review `classify_message` and `classify_inflight`
  in `kiso/brain/common.py` for shared helper code that
  could become a common substrate.
- [ ] Evaluate merge feasibility behind a
  `phase=initial|inflight` parameter against the three
  risk dimensions (maintenance, cross-contamination,
  prompt size/model fit).
- [ ] Assess drift likelihood for inflight-only
  categories (e.g. potential `pause`/`resume`) over the
  next cycle.
- [ ] Apply the decision criteria (merge if shared >60%
  and no contamination risk and prompt fits; default to
  keep separate if inconclusive).
- [ ] Append the written decision to this milestone with
  the measured percentages.
- [ ] If the decision is "merge", propose a follow-up
  milestone (e.g. `M1295`) to the user before adding it
  to the devplan.

**Test.** None. This milestone closes when the written
decision is in the devplan.

**Investigation conclusion (2026-04-09).**

Read both prompts in full and quantified the structural
overlap:

| Metric | `classifier.md` | `inflight-classifier.md` |
|---|---|---|
| Bytes | 3959 | 585 |
| Lines | 34 | 10 |
| Categories | 4 (plan / investigate / chat_kb / chat) | 4 (stop / update / independent / conflict) |
| Output format | `cat:Language` | bare `cat` |
| Language detection | yes | no |
| Entity / Known-Entities disambiguation | yes | no |
| Recent-conversation disambiguation rules | yes (5+ rules) | no (just templated injection) |
| Boundary rules (plan↔investigate, etc.) | yes (10+ rules) | no |
| Templated `{...}` substitutions | no — built in Python | yes (`{plan_goal}`, `{new_message}`, `{recent_conversation}`) |

**Word-for-word shared text: well below 5%.** The two
prompts have zero category overlap and only generic
phrases like "Classify" / "Respond with ONLY..." in
common. The shared framing budget required to consider a
merge per the milestone criteria was 60%; we are roughly
an order of magnitude below that.

**Code-level sharing.** Both functions
(`run_classifier`, `run_inflight_classifier`) call
`call_llm(config, "classifier", ...)` with the same
model key, so they already share the model assignment
via `_MODEL_METADATA` (one source of truth, set during
the M1213 upgrade). The message-construction helpers
(`build_classifier_messages` vs
`build_inflight_classifier_messages`) are separate.
There is no helper code that could become a shared
substrate beyond what is already shared at the
`call_llm` layer.

**Drift likelihood.** The initial classifier already
evolved this cycle: M1290 added the 4th category
`investigate`, M1291 added the empty-KB pre-flight
fallback. The inflight classifier categories have not
changed since they were introduced. There is plausible
future drift on both sides (e.g. inflight could grow a
`pause` / `resume` category once long-running jobs land)
but **independently** — there is no scenario where one
side's evolution constrains the other.

**Decision: KEEP SEPARATE.**

Reasons, in order of weight:

1. **Cross-contamination risk dominates.** Merging
   would mean a single prompt that the LLM has to
   route into 8 categories (4 initial + 4 inflight)
   based on a `phase` parameter. That is strictly
   worse for classifier accuracy than the current 4+4
   split, where each prompt has exactly the choices
   that apply to its phase. Fewer categories per
   call = fewer plausible mistakes.
2. **Output format divergence.** Initial returns
   `cat:Language`, inflight returns bare `cat`.
   Merging requires either a more complex output
   schema or branching post-processing — both add
   failure modes for zero benefit.
3. **Templated vs message-constructed.** Inflight
   uses string substitution (`{plan_goal}` etc.);
   initial uses Python-side message assembly. Merging
   requires picking one mechanism and rewriting the
   other.
4. **Shared framing far below the 60% threshold** that
   the milestone criteria established as the bar for
   merge.
5. **The default decision when inconclusive was "keep
   separate"**, and this analysis is not even
   inconclusive — it actively favors keeping them
   separate.

**No follow-up milestone proposed.** The two roles
deserve to stay separate. They already share the

## Phase 15 — Subprocess cancellation cleanup hygiene (M1295)

### M1295 — Reap subprocess on `wait_for(communicate(), timeout)` cleanup at 4 call sites ✅

> **Diagnosis correction (2026-04-09).** The original M1295 body
> hypothesised the bug was in `_run_subprocess`'s cancel branch
> (the `for t in pending: t.cancel()` race in
> `kiso/worker/utils.py`). When TDD red-phase tests were written
> against that hypothesis, **both the FD-leak test and the
> whitebox spy test on the cancelled `pending` set passed without
> any code change.** That falsified the original diagnosis. The
> real bug is described below; the original analysis is preserved
> verbatim further down for the audit trail.

**Problem.** Running the full unit suite under `xdist` produces a
recurring noisy traceback near the 100% mark:

```
Loop <_UnixSelectorEventLoop running=False closed=True ...> that handles pid 68407 is closed
Exception ignored in: <function BaseSubprocessTransport.__del__ ...>
  ...
  File ".../base_events.py", line 545, in _check_closed
    raise RuntimeError('Event loop is closed')
RuntimeError: Event loop is closed
```

The tests **all pass**. The traceback fires at GC time after the
event loop has been closed, so it does not break a single test, but:

- it pollutes test output and can mask real warnings;
- it indicates the asyncio child-watcher is holding stale PID
  references between tests;
- it gets worse over time as the suite grows.

**Real root cause (verified 2026-04-09 after the original
hypothesis was falsified by red tests).** The bug is **not** in
`_run_subprocess`. That function correctly calls `_kill_proc(proc)`
in the cancel branch, which does `proc.terminate()` followed by
`await proc.wait()` — the OS process is reaped and the child
watcher slot is released. Whitebox spy tests on the `pending` set
confirm the cancelled `comm_task` ends up in `done()` state by
the time `_run_subprocess` returns (the `await _kill_proc(proc)`
yields to the loop, which then drives the cancellation to
completion).

The actual bug is in **four other call sites** that wrap
`proc.communicate()` in `asyncio.wait_for(...)` and silently
swallow `TimeoutError` without killing or reaping the
subprocess:

1. `kiso/hooks.py:48-67` — `run_pre_exec_hooks`:

   ```python
   try:
       proc = await asyncio.create_subprocess_shell(...)
       stdout, stderr = await asyncio.wait_for(
           proc.communicate(context.encode()),
           timeout=_HOOK_TIMEOUT,
       )
       ...
   except asyncio.TimeoutError:
       log.warning("Pre-exec hook timed out after %ds, "
                   "allowing execution", _HOOK_TIMEOUT)
   # ← proc is NOT terminated, NOT awaited.
   #   The child process keeps running. The
   #   _UnixSubprocessTransport stays alive holding the pipes.
   #   The child watcher still has a callback registered for
   #   the PID. All three leak until process exit.
   ```

2. `kiso/hooks.py:95-107` — `run_post_exec_hooks`: same pattern,
   same swallow.

3. `kiso/tool_repair.py:61-80` — `repair_unhealthy_tools`: same
   `wait_for(proc.communicate(), timeout=...)` followed by
   `except asyncio.TimeoutError: log.warning(...)` with no
   cleanup.

4. `kiso/tool_repair.py:145-164` — `rerun_all_deps`: same pattern.

When any of these timeouts fires, `wait_for` cancels the inner
`proc.communicate()` coroutine, the StreamReaders abort their
reads, the transport survives in memory holding pipe FDs and
a registered child callback. Later, when the test (or the
process) exits and the event loop closes, the orphan transport
is GC'd; its `__del__` calls `self.close()` →
`proto.pipe.close()` → `_loop.call_soon(...)` → `_check_closed()`
→ `RuntimeError: Event loop is closed`. Same chain for the
`Loop ... that handles pid X is closed` warning emitted by
`SafeChildWatcher` for the un-released callback.

**Verified by direct code inspection** of all
`asyncio.create_subprocess_*` call sites in `kiso/`:

| Site | Pattern | Leaks? |
|---|---|---|
| `kiso/worker/utils.py:_run_subprocess` (cancel branch) | `wait` + `_kill_proc` | No — proc is killed and awaited |
| `kiso/worker/utils.py:_run_subprocess` (no-cancel branch) | direct `await proc.communicate(...)` | No — communicate awaits wait |
| `kiso/hooks.py:run_pre_exec_hooks` | `wait_for(communicate(), timeout)` + bare `except` | **Yes** |
| `kiso/hooks.py:run_post_exec_hooks` | same | **Yes** |
| `kiso/tool_repair.py:repair_unhealthy_tools` | same | **Yes** |
| `kiso/tool_repair.py:rerun_all_deps` | same | **Yes** |
| `kiso/brain/text_roles.py:run_worker` (bash -n syntax check) | direct `await proc.communicate(...)` (no wait_for) | No |

**Fix.** Introduce a small private helper in a new
`kiso/_subprocess_utils.py` module:

```python
async def communicate_with_timeout(
    proc: asyncio.subprocess.Process,
    input_bytes: bytes | None,
    timeout: float,
) -> tuple[bytes, bytes]:
    """Like ``await asyncio.wait_for(proc.communicate(input), timeout)``
    but kills + reaps the subprocess on timeout so the
    ``_UnixSubprocessTransport`` and the child watcher slot are
    released before the function returns.

    Re-raises ``asyncio.TimeoutError`` after cleanup so callers
    can handle it normally.
    """
    try:
        return await asyncio.wait_for(
            proc.communicate(input_bytes), timeout=timeout,
        )
    except asyncio.TimeoutError:
        try:
            proc.kill()
        except ProcessLookupError:
            pass
        try:
            await proc.wait()
        except BaseException:
            pass
        raise
```

A new top-level module is the cleanest home: `kiso/hooks.py` and
`kiso/tool_repair.py` cannot import from `kiso/worker/utils.py`
without risking dependency cycles, and inlining the same 8 lines
4 times is worse for maintenance.

Then update each call site:

- `kiso/hooks.py:run_pre_exec_hooks` — replace
  `await asyncio.wait_for(proc.communicate(...), timeout=...)`
  with `await communicate_with_timeout(proc, ..., _HOOK_TIMEOUT)`.
- `kiso/hooks.py:run_post_exec_hooks` — same.
- `kiso/tool_repair.py:repair_unhealthy_tools` — same.
- `kiso/tool_repair.py:rerun_all_deps` — same.

Each call site keeps its existing `except asyncio.TimeoutError:`
block — the `TimeoutError` is still raised by the helper, just
after cleanup. Behavior on the success path is identical.

**Why kill + wait, not terminate + wait_for grace + kill:** the
hook / repair scripts that timed out are already past their
budget. There is no "give it 5 more seconds to clean up
gracefully" — the budget was the budget. SIGKILL guarantees the
process exits and `proc.wait()` returns promptly without a
second timeout layer. `_run_subprocess`'s `_kill_proc` uses the
SIGTERM → grace → SIGKILL escalation because exec tasks may have
side effects worth flushing; hook/repair processes are
short-lived utilities that do not.

**Why `_run_subprocess` is NOT touched:** verified by two failed
red-phase tests (`test_cancel_does_not_leak_file_descriptors`
and `test_cancel_drains_pending_communicate_task`). The
`_kill_proc` call inside the cancel branch already drains the
cancelled `comm_task` by virtue of yielding to the loop during
`await proc.wait()`. The `_run_subprocess` cancel path is clean.

**Boundary — what this milestone does NOT do.**

- Does NOT change `_run_subprocess` or `_kill_proc`. They are
  correct as written.
- Does NOT change the public contract of `run_pre_exec_hooks`,
  `run_post_exec_hooks`, `repair_unhealthy_tools`, or
  `rerun_all_deps`. Each function still raises (and catches)
  `asyncio.TimeoutError` exactly as today; the only change is
  that the underlying subprocess is reaped before the
  `TimeoutError` reaches the `except` block.
- Does NOT add a new public API. `communicate_with_timeout` is
  module-private (no `__all__`, name not exported).
- Does NOT touch `kiso/brain/text_roles.py` — its `bash -n`
  syntax check uses direct `await proc.communicate(...)` with
  no `wait_for`, so no leak.

**Three options considered.**

- **A** — Fix only the test infrastructure: add a `populate_roles_in()`
  helper and call it from every conftest / per-test patch site that
  creates an isolated KISO_DIR. ~50 lines of test code. Preserves
  M1289 invariant exactly. Cons: every future test that creates a
  fresh KISO_DIR has to remember the helper or it fails the same
  way again. Long-term fragile.
- **B** — Restore the package fallback in `_load_system_prompt`
  (revert M1289 partially). One-file change. Tests just work. Cons:
  re-opens the "two sources of truth" trap M1289 was designed to
  close. A corrupted user file would be silently shadowed by the
  bundled default and the user would not notice. Unacceptable.
- **C — Self-heal in the loader (chosen).** When
  `_load_system_prompt` finds the user file missing or empty, copy
  the bundled default into the user dir atomically, log a WARNING,
  then read. The user dir remains the runtime source of truth — the
  bundle is the seed, not an alternate read path. Same mechanism
  `_init_kiso_dirs()` already implements at boot, just deferred to
  first access.

**Why C wins long-term.**

1. **Production resilience.** Today, if a role file is deleted or
   truncated AT RUNTIME (after the boot-time copy in
   `_init_kiso_dirs`), kiso dies on the next LLM call with a
   FileNotFoundError. The error message says "Run `kiso role reset
   NAME`" but that assumes a CLI is reachable — useless for hosted
   deployments, containers without exec access, read-only volumes,
   or sessions where a user simply cannot be at the terminal. With
   self-heal the server keeps working, the warning lands in the
   logs, and an operator can investigate at their leisure instead
   of being paged at 3 AM.
2. **Single source of truth preserved.** After self-heal, the user
   dir IS the only place the loader reads from. The bundle is the
   factory default that seeds the user dir on first access. This
   is exactly the M1289 contract — the only thing that changes is
   WHEN the seeding happens (lazy at first access vs eager at
   boot). Both paths converge on the same end state.
3. **Test infrastructure becomes trivial.** With self-heal in the
   loader, the 18 per-test patches in tests/live/* and the
   `_func_kiso_dir` functional fixture do not need to populate
   roles. The first time the loader is called during a test, it
   self-heals into whatever tmp dir is currently patched as
   `KISO_DIR`. Tests pass for the same reason production passes:
   the loader is defensive. **Zero ongoing maintenance burden** on
   future tests.
4. **Future-proof.** When a future milestone adds a new bundled
   role, the test infrastructure adapts automatically because
   self-heal copies whatever is in the bundle. Option A would
   require updating the helper every time the bundle changes.
5. **Coherence with the rest of the kiso resilience story.** The
   reset CLI, the boot-time copy, the audit trail, the
   user-dir-ownership story — all of them stay coherent. We just
   add one more layer: graceful degradation at the loader level.

**Trade-off accepted.** If a user customized a role and then
deletes the file (intentionally or by mistake), self-heal silently
restores the bundled default. The user loses their customization.
Mitigation: log a WARNING with the role name and the user dir
path. The warning is also clear about what was lost ("Any previous
customization is gone"). This is strictly better than M1289 (where
the same scenario kills the server) and strictly worse than a
hypothetical option that requires manual confirmation (which
defeats the resilience goal). Net: the warning is the right
balance.

New behavior:

```python
def _load_system_prompt(role: str) -> str:
    if role in _prompt_cache:
        return _prompt_cache[role]
    user_path = KISO_DIR / "roles" / f"{role}.md"
    if not user_path.exists() or user_path.stat().st_size == 0:
        # Self-heal: seed the user dir from the bundled default
        # (the same way _init_kiso_dirs does at boot, but
        # deferred to first access). Logs a WARNING so operators
        # see when self-heal kicks in.
        pkg_path = _ROLES_DIR / f"{role}.md"
        if not pkg_path.exists():
            raise FileNotFoundError(
                f"Role file '{role}' missing from both user dir "
                f"({user_path}) and package bundle ({pkg_path}). "
                f"The kiso installation may be corrupted; reinstall."
            )
        _self_heal_role(user_path, pkg_path, role)
    text = user_path.read_text()
    _prompt_cache[role] = text
    return text


def _self_heal_role(user_path: Path, pkg_path: Path, role: str) -> None:
    """Atomically copy pkg_path → user_path and log a warning."""
    user_path.parent.mkdir(parents=True, exist_ok=True)
    bundled_text = pkg_path.read_text(encoding="utf-8")
    tmp_path = user_path.with_suffix(user_path.suffix + ".tmp")
    tmp_path.write_text(bundled_text, encoding="utf-8")
    tmp_path.replace(user_path)  # atomic on POSIX
    log.warning(
        "Self-healed role '%s' from bundled default into %s. "
        "Any previous customization is lost; restore from backup if needed.",
        role, user_path,
    )
```

Notes:
- Atomic write via tmp + rename — concurrent self-heal from two
  workers cannot produce a partially written file.
- The empty-file case (`stat().st_size == 0`) is treated as
  "needs heal" because an empty system prompt is undefined LLM
  behavior. This matches the M1289 `_init_kiso_dirs` self-heal
  convention (`if not target.exists() or target.stat().st_size == 0`).
- Cache after self-heal contains the bundled content; subsequent
  calls hit the cache.
- The reset CLI hint in the FileNotFoundError now applies only

### M1299 — Plan→chat_kb pre-flight promotion (symmetric to M1291) ✅

**Problem.** When the classifier routes a knowledge query
to `plan` or `investigate` despite the answer already
existing in the KB, the message is forced through the
planner. This causes two distinct failures:

1. `TestMultiTurnLearning.test_learning_retention` — turn 2
   asks `che framework usa il progetto corrente?`. Curator
   has already promoted `flask` as an entity from turn 1, so
   the classifier sees it in `entity_names`, but it cannot
   make the semantic jump `framework ↔ flask` and routes to
   `plan`. The planner reads the fact from the briefer and
   produces a legitimate msg-only recall plan, which the
   validator at `kiso/brain/planner.py:417-424`
   (`_validate_plan_ordering`) rejects with
   `Plan has only msg tasks — include at least one
   exec/tool/search task`. Attempt 2 obeys and inserts an
   `exec` over `pyproject.toml` (which doesn't exist in the
   workspace) → empty output → reviewer discards learnings.
   Test asserts no exec → fail.

2. `TestF13ChatKBClassification.test_chat_kb_user_entity` —
   `cosa sai di guidance.studio?` with a pre-seeded fact.
   Classifier picks `investigate` (URL/domain rule wins
   over Known Entities), planner browses the live site, the
   messenger faithfully reports the real site content
   (`boutique di consulenza strategica con sede a Genova`),
   and the seeded fact (`SaaS platform for onboarding
   workflows`) is never loaded into the messenger context
   because the briefer→`search_facts_scored` path is only
   wired to the chat_kb fast path. Test asserts presence of
   `onboarding|saas|workflow` → fail. Crucially: the
   messenger is **not** hallucinating — it is reporting real
   browse output. The bug is upstream at the routing layer.

**Why now and why this shape.** v0.9 introduced an
asymmetry. M1290 added `investigate` as a 4th classifier
category, and M1291 added `_chat_kb_preflight_fallback` so
that an empty KB transparently demotes `chat_kb → investigate`.
The reverse direction — promoting `plan|investigate → chat_kb`
when the KB *does* have a confident match — was never
implemented. That gap is structural: there is no place in
the dispatcher where a stored fact can override a wrong
classifier verdict, even when the briefer would later
surface the same fact for a different code path.

The fix mirrors M1291 exactly: a deterministic pre-flight
keyword search at the dispatcher level, gated by query
shape, that re-routes the message through `_fast_path_chat`
when the KB has a confident match. No new schema fields, no
validator changes, no classifier prompt edits, no planner
changes — the existing chat_kb fast path already has
everything needed (briefer + entity-id/tag-aware
`search_facts_scored` + messenger). We just give the
dispatcher a way to reach it from the wrong-verdict branch.

**Why not the alternatives.**

- **Add a `kb_answer` flag to the plan schema and let the
  validator allow msg-only when set.** Rejected: requires
  schema change, planner.md edit, and would break ≥5 unit
  tests in `tests/test_brain.py` and `tests/test_worker.py`
  that exercise validator escape hatches. Also conflates
  "knowledge storage" (existing `has_knowledge` flag) with
  "knowledge recall" semantically.
- **Tighten the classifier prompt with explicit
  precedence "Known Entities > URL/domain".** Rejected:
  prompt rules are fragile against models like
  `gemini-2.5-flash-lite`, and even a corrected classifier
  cannot make the semantic leap `framework ↔ flask`
  required by TestMultiTurnLearning. The dispatcher-level
  pre-flight is deterministic and covers both cases.
- **Lower the validator strictness on msg-only.**
  Rejected: removes a real guard against lazy planner
  output and breaks unrelated tests.

**Approach.**

1. New private helper `_plan_chat_kb_preflight_promotion`
   in `kiso/worker/loop.py`, structurally symmetric to the
   existing `_chat_kb_preflight_fallback` (lines 229-304).
   Signature mirrors the fallback. Returns `True` when the
   message has been fully handled via the fast path and the
   caller should `return` immediately.
2. Insertion point: in `_process_message` right after the
   existing `_chat_kb_preflight_fallback` block
   (`kiso/worker/loop.py:2649-2654`), wrapped in a guard
   that fires only when `msg_class in ("plan", "investigate")`
   and `fast_path_enabled`.
3. Promotion gate (deterministic, no LLM):
   - Extract keywords with the same `[:10]` pattern used by
     the existing pre-flight (`content.lower().split()[:10]`)
     so behavior is symmetric and predictable.
   - Skip promotion if the message looks imperative. Reuse
     the existing `_STOP_PATTERNS`-style approach: check
     for an explicit imperative-verb prefix at the start of
     the message in EN/IT/ES (small fixed list, e.g.
     `run|install|fix|create|delete|update|write|esegui|installa|crea|cancella|aggiorna|scrivi|ejecuta|instala|crea|borra|actualiza|escribe`),
     and skip if matched. We do **not** invent a heuristic
     for "is this a question" — the absence of imperative
     prefix is sufficient because the classifier already
     gave us the strong signal that this is not stop/chat.
   - Call `search_facts_scored` with the same arguments
     the existing pre-flight uses (entity_id=None,
     tags=None, keywords=keywords, session, is_admin=False,
     project_id=session_project_id).
   - Confidence threshold: require *at least one* fact
     whose top score from `search_facts_scored` is above the
     same default that `_msg_task_impl` uses internally
     (mirror exactly — do not invent a new threshold).
     Implementation note: read `search_facts_scored` to
     confirm it returns scored results sorted, then accept
     the top-1 if its score clears the existing default.
     If the scoring shape doesn't expose a usable threshold,
     fall back to "≥1 fact returned AND query keywords have
     ≥1 alphanumeric token of length ≥4" — strictly less
     precise than the briefer, false negatives accepted.
4. On positive promotion, persist a fixed deterministic
   transition message via the same `_PROMOTE_MSGS` table
   pattern as `_CHAT_KB_FALLBACK_MSGS` (English + Italian
   minimum), then call `_fast_path_chat` exactly the way
   the existing fast-path block does, and return the
   resulting `fast_plan_id` so post-plan knowledge spawning
   stays intact.
5. The promotion is gated by `setting_bool(config.settings,
   "fast_path_enabled")` — same gate as the existing fast
   path. Operators can disable both with one switch.

### M1300 — Messenger emoji discipline: prompt + deterministic strip ✅

**Problem.** `TestMessengerQuality.test_messenger_no_emoji_no_hallucination`
fails because the messenger output contains
`🎯 💻 🌐 🛠 🔬 📚`. The current rule
`No emoji. Plain text only.` lives at
`kiso/roles/messenger.md:19` — the very last line of the
file, with weak phrasing relative to the CRITICAL Language
rule at line 1. Models like `gemini-2.5-flash` ignore
trailing soft rules, and there is no post-processing layer
to enforce the constraint deterministically. Devplan v0.5
M384 documents that the no-emoji rule was added for the
exact same reason in the past — this is a recurring
regression because prompt-only enforcement is not reliable
against this class of model.

**Why now and why this shape.** The "always deep root
cause" rule (per `feedback_deep_root_cause.md`) and the "no
heuristics, prefer structural/deterministic approaches"
rule (per `feedback_no_heuristics.md`) both point to the
same conclusion: a hard rule that the model is allowed to
violate is not actually a hard rule. The fix is to make it
a hard rule in code as well as in the prompt.

**Approach.**

1. **Prompt strengthening.** Promote the no-emoji rule to
   the top of `kiso/roles/messenger.md`, placed alongside
   the existing `CRITICAL — Language` rule. Phrasing:
   `CRITICAL — No emoji. Plain text only. Markdown is
   allowed for structure (bold, lists, code).` Remove the
   line at the bottom to avoid duplication.
2. **Deterministic strip.** In `kiso/worker/loop.py`
   `_msg_task_impl` (and any other path that finalizes
   messenger output for persistence), apply the same
   regex used by the test
   (`tests/functional/test_knowledge.py:393-395`)
   to strip emoji characters before the text is written
   to the task row, conversation history, and webhook
   payload. Centralize the regex in
   `kiso/brain/text_roles.py` (next to the messenger role
   helpers) so it has a single source of truth and can be
   imported by tests.
3. **Test regex parity.** Leave the test regex pointing at
   the same module-level constant (or assert equality in a
   small unit test) so the test cannot drift away from
   what the strip enforces.

**Why a strip and not just a prompt fix.** The agent
investigation confirmed the regex catches every emoji
emitted in the failing run, that messenger output is
persisted without sanitization today, and that this same
class of regression has historically come back after a
prompt-only fix (M384). A deterministic strip is not a
heuristic — it is a hard rule consistently applied to a
hard constraint. The prompt edit stays so the model is
biased against emitting emoji in the first place; the
strip is defense in depth.

**Files to touch.**

- `kiso/roles/messenger.md` — promote rule to top.
- `kiso/brain/text_roles.py` — define `_EMOJI_STRIP_RE`
  constant + tiny `strip_emoji(text)` helper.
- `kiso/worker/loop.py` — call `strip_emoji` on messenger
  output in `_msg_task_impl` and in `_fast_path_chat`

#### Bug A — `_fts5_query` produces invalid FTS5 syntax for any token containing `.`, `/`, or `-`

**File.** `kiso/store/knowledge.py:21-25`

```python
def _fts5_query(text: str) -> str:
    tokens = [t.strip() for t in re.findall(r"[A-Za-z0-9_./-]+", text)]
    if not tokens:
        return ""
    return " OR ".join(dict.fromkeys(tokens))
```

The regex keeps `.`, `/`, and `-` inside tokens. The
default FTS5 tokenizer (`unicode61`) treats these as
word boundaries and stores them as separators. The
resulting query (e.g. `cosa OR sai OR di OR guidance.studio`)
is invalid FTS5 syntax — `.` raises
`fts5: syntax error near "."`, `/` raises the same
error, and `-` is parsed as a NOT operator
(`-name` → `no such column: name`). The exception is
silently swallowed by `_search_facts_by_keywords:305-306`
(`except Exception: return []`), so every keyword
search containing a dotted/dashed/slashed identifier
**returns 0 facts** without surfacing any error.

**Reproduction (deterministic, no LLM).**

```text
query='guidance.studio'  -> ERROR: fts5: syntax error near "."
query='/tmp/foo'         -> ERROR: fts5: syntax error near "/"
query='dash-name'        -> ERROR: no such column: name
query='snake_case_var'   -> 1 match  (underscore is a word char in unicode61)
```

**Blast radius.** Every caller of `_fts5_query` is
affected:

- `search_facts` (knowledge.py:37) — used by the
  planner's `relevant_entities` lookup and by
  `kiso/brain/planner.py` indirectly
- `_search_facts_by_keywords` (knowledge.py:293) — used
  by `search_facts_scored` whenever `entity_id` and
  `tags` are absent (the dispatcher pre-flight path
  for both M1291 fallback and M1299 promotion)
- `search_learnings` (knowledge.py:432, untouched here
  but same regex)

Concrete observable failures:

- F13: `cosa sai di guidance.studio?` →
  pre-flight returns 0 → M1291 fallback fires
  → `msg_class="investigate"` → planner browses the
  live site → seeded fact never reaches the messenger.
- Any user query containing a file path, URL, dotted
  Python module name, version string, hyphenated
  identifier, etc. — all silently bypass the
  pre-flight today.

**Fix (structural).** Restrict `_fts5_query` to FTS5
word characters only:

```python
def _fts5_query(text: str) -> str:
    tokens = [t.strip() for t in re.findall(r"[A-Za-z0-9_]+", text)]
    if not tokens:
        return ""
    return " OR ".join(dict.fromkeys(tokens))
```

Every non-word character (including `.`, `/`, `-`,
parentheses, punctuation, accented separators) becomes
an implicit token boundary. `guidance.studio` →
`guidance OR studio`, `/tmp/foo` → `tmp OR foo`,
`v1.2.3` → `v1 OR 2 OR 3`. All resulting queries are
syntactically valid FTS5, and they match the way
unicode61 actually indexed the content (which already
split on these characters).

**Trade-off.** The change can produce more matches
per query (more OR terms, fewer rejected as invalid
syntax) but the existing scoring in
`search_facts_scored` already orders by relevance, so
false positives drop to the bottom. False negatives
(the current behavior) drop to zero for these
characters. There is no path where the new behavior
returns *fewer* matches than the old behavior.

### M1302 — Reproduce and diagnose `TestReplanFlowE2E` + `TestF30FullPipeline` timeouts

**Type.** Investigation-only milestone. No code or test
changes are committed under M1302 itself; outcome is a
written diagnosis + a follow-up milestone (or
explicit closure if no bug exists). This shape is unusual
for this project — the user explicitly authorized it
because losing the context from the 2026-04-09 full-matrix
run would force a re-investigation later from scratch.

**Problem.** Two failures from the 2026-04-09 full matrix
were excluded from M1299-M1301 because static code
inspection could not distinguish between three plausible
explanations:

1. `tests/live/test_e2e.py::TestReplanFlowE2E::test_replan_after_failed_exec`
   timed out after `LLM_TEST_TIMEOUT = 240s`. Worst-case
   LLM call count from code inspection: ~10 calls × per-call
   ceiling `llm_timeout=60s` (live) = up to 600s teoretical
   ceiling. The `_no_retry_backoff` test fixture verified
   to zero `_RATE_INITIAL_BACKOFF` (failure log confirms
   `retry 1/4 in 0.0s`), so 429 backoff is not the cause.
2. `tests/functional/test_preset_workflows.py::TestF30FullPipeline::test_browse_ocr_aider_exec`
   timed out after `LLM_MULTI_PLAN_TIMEOUT = 600s`. Plan
   has 6 tasks across 2 plans; ceiling `llm_timeout=180s`
   (functional). Last log line was a briefer for task 5
   (likely "save OCR output"). No subprocess hang
   reproduction found in code.

**Three competing hypotheses, none falsifiable from code
alone.**

- **(H1) Pure budget mismatch.** Real OpenRouter latency
  + worst-case call count > test budget. Code is correct;
  fix is a budget bump.
- **(H2) Latent code bug.** A specific LLM call hangs
  near its `llm_timeout` ceiling under particular
  conditions (e.g. SSE stream stall not catching, model
  fallback chain doubling latency, briefer call without
  outer wait_for). Code is wrong; bumping budgets would
  mask the bug.
- **(H3) Provider-side flake.** OpenRouter (or the
  specific model in use) intermittently slow/unresponsive
  in a way that is not anyone's bug. Fix is `flaky`
  marking with explicit reason.

**Approach.**

1. **Reproduction with full instrumentation.** Re-run
   each failing test in isolation **three times** with
   `KISO_LOG_LEVEL=DEBUG`, full LLM call accounting
   captured (every model name, request start/end
   timestamp, response duration, finish_reason, retry
   counts, 429 hits). Save raw output to
   `tmp/m1302/{test_name}.run{N}.log` so the data
   survives the investigation.
2. **Per-call latency histogram.** From the captured
   logs, build a per-call latency table for each run:
   `(role, model, duration_ms, retries, finish_reason)`.
   Identify the slowest 5 calls per run.
3. **Budget arithmetic.** Sum the call durations and
   compare against the test budget. Three outcomes:
   - Sum ≪ budget but `wait_for` still fired → there is a
     blocking gap *between* LLM calls. Walk the code path
     between the last visible call and the timeout.
     Likely H2.
   - Sum ≈ budget → call count × average latency simply
     exceeds budget. Likely H1.
   - Sum highly variable across the 3 runs (>2× spread)
     and individual call durations spike to ≥`llm_timeout`
     → provider-side flake. Likely H3.
4. **H2 walk (only if needed).** If H2 is suggested,
   instrument `_handle_exec_task` and `_handle_tool_task`
   with per-step timing markers and re-run once to localize
   the gap.
5. **Decision.** Based on the evidence, write a short
   diagnosis note (≤300 words) at the end of this
   milestone in the devplan, and open a follow-up:
   - **H1 confirmed** → open M1303 to bump
     `LLM_TEST_TIMEOUT` and the F30 budget to specific
     values justified by the measured worst case +
     headroom factor.
   - **H2 confirmed** → open M1303 with the specific bug
     fix; do not bump budgets.
   - **H3 confirmed** → open M1303 to add `pytest.mark.flaky`
     with an explicit comment citing the measured
     evidence.
   - **Inconclusive** → document and either re-run with
     more samples or escalate to the user.

### Phase 19: Rename tool → wrapper

The "tool" concept in kiso refers to installable subprocess wrappers
around external libraries/CLIs that operate in the session workspace.
The name "tool" is overloaded in the AI ecosystem (API-level callable,
MCP tool, etc.).  Renaming to "wrapper" clarifies the architectural
role: a wrapper adapts an external capability to kiso's session context.

No backward compatibility needed — project is pre-release.

### Phase 21: MCP client integration

47. [ ] **M1313** — MCP client core (`kiso/mcp.py`)

    **Why:** Kiso needs to connect to external MCP servers to access
    the MCP ecosystem (GitHub, Google Drive, Postgres, Slack, etc.)
    without writing dedicated wrappers.

    **Approach:** Create `kiso/mcp.py` with an `MCPClient` class that
    manages MCP server subprocess lifecycle and JSON-RPC communication.
    MCP servers are configured at instance level in `config.toml`.
    All users of the instance see the same MCP servers.

    **Design:**
    - Config: `[mcp.<name>]` sections in `config.toml` with `command`,
      `args`, `env` fields
    - `MCPClient`: spawns subprocess, sends JSON-RPC 2.0 via stdin,
      reads responses from stdout
    - Lifecycle: lazy start (on first use), auto-restart on crash,
      graceful shutdown on kiso exit
    - Methods: `initialize()`, `list_tools()`, `call_tool(name, params)`,
      `shutdown()`
    - `MCPManager`: manages pool of `MCPClient` instances, one per
      configured server

    **Tasks:**

    - [ ] Create `kiso/mcp.py` with `MCPClient` and `MCPManager`
    - [ ] Implement JSON-RPC 2.0 framing (request/response)

    - [ ] Implement lifecycle (lazy start, restart, shutdown)
