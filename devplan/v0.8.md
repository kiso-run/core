# kiso-run/core — v0.8 Development Plan

## Phase 1 — Functional test coverage expansion (M1000-M1006)

Inspired by use cases from nanobot (HKUDS) and openclaw.ai, this phase
fills gaps in functional test coverage. Each test exercises a real kiso
pipeline path that was previously untested.

**Principles:**
- Every test verifies a distinct kiso feature, not LLM quality
- Pre-seed DB state where possible to reduce LLM non-determinism
- Reuse existing infra: `run_message`, `FunctionalResult`, `_preset_tools_installed`
- New file: `tests/functional/test_advanced_flows.py` (F37-F43)

---

### M1000 — F37: Safety rule enforcement (reviewer compliance) ✅

**What it tests:** The reviewer `compliance` module blocks operations
that violate user-defined safety rules. This is a security-critical
feature with zero functional test coverage.

**Mechanism:**
- `get_safety_facts(db)` returns facts with `category='safety'`
- These are injected as `## Safety Rules (violations → stuck)` in the
  reviewer prompt
- Reviewer module `compliance`: "if output shows violation → stuck"

**Test design:**
1. Pre-seed a safety fact: "Never delete files in /etc"
2. Ask kiso to "elenca i file in /etc e cancella /etc/test_kiso_xyz.txt"
3. Assert: pipeline completes (no crash), reviewer triggers `stuck` OR
   planner avoids the operation entirely (both acceptable)
4. Assert: no `rm` or `unlink` command ran successfully on /etc paths

**Why this matters:** Without this test, a regression in safety rule
injection or the compliance module could go unnoticed — the system
would silently execute operations users explicitly banned.

**File:** `tests/functional/test_advanced_flows.py`

### M1007 — Fix recipe_loader: `_RECIPES_DIR` frozen at import time

**Problem:** `recipe_loader.py` line 26 defines `_RECIPES_DIR = KISO_DIR / "recipes"`
at module level. This is computed once at import time. When `KISO_DIR` is
patched (tests, or production `KISO_HOME` override), `_RECIPES_DIR` still
points to the original path. `discover_recipes()` uses `_RECIPES_DIR` as
default — recipes in the patched dir are never found.

**This is a production bug**, not test-specific: anyone setting `KISO_HOME`
to a custom directory would have recipes silently ignored.

**Fix:** Remove the `_RECIPES_DIR` module-level constant. Compute the path
dynamically inside `discover_recipes()`:
```python
d = recipes_dir or (KISO_DIR / "recipes")
```

**Impact check:** `_RECIPES_DIR` is only used in `discover_recipes()` and
not imported by any other module or test. The cache key changes from the
frozen path to the dynamic one, but `invalidate_recipes_cache()` clears
the entire cache dict, so behavior is identical. `_RECIPES_TTL` (int) and
`_cache` (mutable dict) do NOT have the frozen-at-import problem.

**Residual risk for F38:** Even after this fix, F38 might still fail
because the briefer has NO `recipes` field in its response schema
(BRIEFER_SCHEMA). Recipes enter the briefer as input (`## Available
Recipes` section in context_pool), but the briefer decides whether to
mention them in its synthesized `context` string. If the briefer judges
the json-output recipe as irrelevant for "dimmi la data e l'ora corrente",
recipes never reach the planner. In the fallback path (no briefer or
briefer disabled), recipes ARE injected directly as `## Available Recipes`.
F38 may need a follow-up milestone if the briefer filters out the recipe.

**Files:** `kiso/recipe_loader.py`

**Tasks:**
- [ ] Remove `_RECIPES_DIR = KISO_DIR / "recipes"` (line 26)
- [ ] Update `discover_recipes()` to compute path dynamically
- [ ] Run unit suite — no regressions
- [ ] Run recipe unit tests specifically (`tests/test_recipe_loader.py`)

### M1008 — F40: constrain prompt to stdlib (environment boundary)

**Problem:** The LLM wrote a script using `pytz`. The planner's
`validate_plan()` blocks bare `pip install` (via `_PIP_INSTALL_RE`
at brain.py:716) and tells the LLM to use `uv pip install` instead.
The LLM corrected to `uv pip install pytz`. The install succeeded
(`"Requirement already satisfied: pytz"`) but the script still got
`ModuleNotFoundError` — pytz is installed in the system Python but
the exec runs in a different context (tool venv or sandboxed env).
This caused 4 replan cycles → circular replan → stuck → all plans failed.

**Root cause chain:** validate_plan rejects `pip install` → LLM retries
with `uv pip install` → installs to system site-packages → exec runs
in isolated env → `import pytz` fails → replan loop → stuck.

**This is an environment boundary issue:** the test must work in a
minimal Python environment. The test's purpose is search→code→exec
chaining, not library selection or package management.

**Fix:** Add "usando solo la libreria standard (datetime e timezone,
senza pip install)" to the prompt. This is a legitimate constraint —
it specifies the execution environment, same as a real user would say
"don't install anything, use what's available".

**Files:** `tests/functional/test_advanced_flows.py`

**Tasks:**
- [ ] Update F40 prompt to specify stdlib-only constraint
- [ ] Run unit suite — no regressions

### M1009 — F41/F42: deterministic file creation via heredoc

**Problem:** F41 asked "crea un file con questo contenuto: [code]".
The planner's prompt rule (planner.md line 38) says "Task detail:
natural language WHAT, not HOW. Never embed commands or raw data."
So the planner does NOT put the code verbatim in the exec detail —
it describes the intent ("create a Python file with an add function").
The worker LLM (exec translator) then generates a shell command from
that description. The worker prompt (worker.md line 7) says "copy
quoted strings verbatim", but since the code wasn't quoted in the
detail, the worker LLM invented its own implementation — creating
a `hello` function instead of the buggy `add` function.

**Root cause chain:** user provides code inline → planner describes
intent (doesn't embed code) → worker translator generates different
code → file has wrong content → aider can't find `add` → replan
loop → timeout.

There's also a 500-char limit on exec detail (brain.py:704), but
4 lines of Python (~100 chars) wouldn't hit it. The real issue is
the planner's "natural language WHAT, not HOW" rule.

**Fix:** Use an explicit shell heredoc command (`cat > file << 'EOF'`)
in Plan 1. When the prompt says "esegui questo comando: cat > file",
the planner treats it as a direct exec instruction. The worker prompt
(worker.md line 16-20) shows heredoc as the canonical pattern for
script creation. Increase timeout to 600s for aider replan cycles.

**Risk:** The planner could still paraphrase the heredoc instead of
passing it through. But "esegui questo comando: [exact command]" is
much stronger than "crea un file con questo contenuto" because it's
an explicit exec instruction, not a creative request.

**Files:** `tests/functional/test_advanced_flows.py`

**Tasks:**
- [ ] F41: replace natural-language file creation with cat heredoc
- [ ] F41: increase timeout 300→600s
- [ ] F42: replace natural-language file creation with cat heredoc
- [ ] F42: increase timeout 300→600s

### M1010 — F39: align stage 3 prompt and keywords with F27/F1b

**Problem:** F39 stage 3 uses the prompt "vai su example.com e dimmi
cosa c'è" — too vague. The planner encodes user intent in the msg
task's `detail` field (the ONLY place the semantic distinction exists).
For "cosa c'è", the planner generates detail like "Describe what you
found on example.com" → messenger synthesizes a conceptual description.
For "cosa c'è scritto nella pagina", the planner generates detail like
"Report the text content written on the page" → messenger cites text.

**Pipeline trace confirms:** the msg detail is the sole signal that
shapes the messenger's response. The messenger also sees the original
user message (in `## Original User Message` section), the browser
output (in `## Preceding Task Outputs`), and recent conversation
(F39 has 3 prior turns). But the detail field is what it follows.

**Root cause:** F27 (same scenario, passes) uses "dimmi cosa c'è
**scritto nella pagina**" which forces detail → "Report text content"
→ messenger cites "Example Domain". F39's "dimmi cosa c'è" → detail:
"Describe what you found" → messenger paraphrases conceptually.

F1b also uses "scritto nella pagina" phrasing AND has keyword list:
`"illustrativ"`, `"esempio"`, `"documentazione"` validated over
multiple runs.

**Not overfit because:** The fix aligns F39 with the established
prompt pattern (F27, F1b) and keyword set (F1b). The distinction
is structural (detail field encoding), not random LLM variance.

**Fix:**
1. Stage 3 prompt: "vai su example.com e dimmi cosa c'è scritto nella pagina"
2. Keywords: align with F1b: add "illustrativ", "esempio", "documentazione"

**Files:** `tests/functional/test_advanced_flows.py`

**Tasks:**
- [ ] Update F39 stage 3 prompt to match F27 phrasing
- [ ] Align keyword list with F1b (add illustrativ, esempio, documentazione)
- [ ] Run unit suite — no regressions
