# Devplan v0.4 — Post-Install Health, Validation UX & Planner Discipline

Goal: fix the remaining failure modes from the real browser-skill trace. The skill
installs "successfully" but system deps are missing. The planner wastes ~12 LLM
calls on validation retries and ignores the reinstall rule when deps fail. The
post-install check_deps call is broken (passes empty dict).

Derived from: real trace where kiso skill install browser succeeds, but
Playwright has 70+ missing system libraries. Planner sends null args 3x despite
clear error messages. When deps fail, planner tries manual apt-get instead of
skill reinstall.

---

## Problem Inventory

### P1 — Post-install check_deps passes empty dict (always returns [])
`_plugin_install` (plugin_ops.py:249) calls `check_deps_fn({"path": str(plugin_dir)})`.
But `check_deps()` reads `skill.get("deps", {}).get("bin", [])` — the dict has
no "deps" key, so it always returns empty. The "missing binaries" warning never
fires, even when deps are genuinely missing.

### P2 — Skill args validation error doesn't show expected format
When `validate_plan` catches missing required args (brain.py:380-383), the error
says `"skill 'browser' args invalid: missing required arg: action"` but doesn't
show what the args should look like. The planner (deepseek-v3.2) sends `null`
again because it doesn't know the format. This caused 3 validation retries.

### P3 — Planner ignores broken skill reinstall rule
M180 added the rule: "if broken, plan `kiso skill remove NAME && kiso skill
install NAME`". But in the trace, the planner instead tried manual `apt-get
install` for individual libraries. The rule is too weak — it's one line among
many, and deepseek-v3.2 doesn't follow it reliably.

### P4 — Planner tries apt-get for skill deps (should be blocked)
Skill system deps should only be installed via `deps.sh` (part of `kiso skill
install`). Manual `apt-get install` for skill libraries is wrong because:
(a) the list is incomplete/guessed, (b) it doesn't match what `deps.sh` does,
(c) it often times out. The planner should be blocked from this pattern.

### P5 — Repeated identical validation errors don't escalate
The planner makes the same mistake (null args) 2-3 times in a row. Each retry
sends the same error message with no additional guidance. After 2 identical
failures, the error should include a concrete example to break the loop.

### M183 — Fix check_deps call in _plugin_install
**Problem:** P1
**Files:** `cli/plugin_ops.py` — `_plugin_install`
**Change:** After parsing the manifest and validating it, build a proper skill
info dict from the manifest for `check_deps_fn`:
```python
# Build proper info dict for check_deps (needs "deps" from manifest)
kiso = manifest.get("kiso", {})
plugin_info = {
    "path": str(plugin_dir),
    "deps": kiso.get("deps", {}),
}
missing = check_deps_fn(plugin_info)
```
When deps are missing, escalate from a warning to an actionable message:
```
warning: missing binaries after install: playwright
  The skill may not work correctly. Try: kiso skill remove NAME && kiso skill install NAME
  Or run deps.sh manually: bash /root/.kiso/skills/NAME/deps.sh
```
**Test:** Mock a skill install where kiso.toml declares `bin = ["nonexistent"]`.
Verify the warning message includes the missing binary name.

### M184 — Include args example in skill validation error
**Problem:** P2
**Files:** `kiso/brain.py` — `validate_plan`
**Change:** When skill args validation fails, include a concrete example of the
expected args format derived from the schema:
```python
if arg_errors:
    # Build example from schema
    example_args = {}
    for aname, adef in schema.items():
        atype = adef.get("type", "string")
        example_args[aname] = {"string": "value", "int": 1, "float": 1.0, "bool": True}.get(atype, "value")
    example_json = json.dumps(example_args)
    errors.append(
        f"Task {i}: skill '{skill_name}' args invalid: "
        + "; ".join(arg_errors)
        + f'. Set args to a JSON string like: \'{example_json}\''
    )
```
This gives the planner a concrete template to copy.
**Test:** Validate a plan with null args for a skill requiring `action` (string).
Verify the error contains `Set args to a JSON string like: '{"action": "value"}'`.

### M187 — Post-install health check with actionable output
**Problem:** P1 (downstream)
**Files:** `cli/plugin_ops.py` — `_plugin_install`
**Change:** After `check_deps_fn` detects missing binaries, attempt to re-run
`deps.sh` once automatically as a repair step:
```python
missing = check_deps_fn(plugin_info)
if missing and deps_path.exists() and not args.no_deps:
    print(f"Missing binaries: {', '.join(missing)} — re-running deps.sh...")
    result = subprocess.run(["bash", str(deps_path)], capture_output=True, text=True)
    # Re-check
    missing = check_deps_fn(plugin_info)
if missing:
    print(f"warning: still missing binaries after install: {', '.join(missing)}")
    print(f"  The skill may not work. Try running deps.sh manually or check the skill docs.")
```
**Test:** Mock a skill install with deps that are missing on first check but
present after re-run. Verify the auto-retry runs and the final message reflects
the result.

### M189 — Remove exec_timeout from subprocess execution
**Problem:** `exec_timeout` (300s) kills local commands (skill installs, model
downloads, builds) that are legitimately slow. The setting is overloaded: it's
used both as subprocess timeout AND as fallback LLM HTTP timeout.
**Files:** `kiso/worker/loop.py`, `kiso/worker/utils.py`, `kiso/worker/exec.py`,
`kiso/worker/skill.py`
**Change:**
- In `_run_subprocess` (utils.py): accept `timeout=None` and skip
  `asyncio.wait_for` wrapping when None (just `await proc.communicate()`).
- In `_handle_exec_task` (loop.py:867): pass `timeout=None` instead of
  `ctx.exec_timeout`.
- In `_handle_skill_task` (loop.py:765): pass `timeout=None` instead of
  `ctx.exec_timeout`.
- Remove `exec_timeout` from `_PlanExecContext` — it's no longer needed for
  subprocess calls. LLM calls already have their own dedicated timeouts.
**Keep:** `exec_timeout` setting stays in config.py as the fallback LLM timeout
(used by `call_llm` for curator/summarizer/etc.). Rename the config comment to
clarify it's for LLM calls only.
**Test:** Existing exec tests must pass. Add a test verifying that `_run_subprocess`
with `timeout=None` doesn't wrap in `wait_for`.

### M190 — Remove job_timeout from planning loop
**Problem:** `job_timeout` (900s) caps total wall-clock for processing one user
message. But `max_replan_depth=5` and `max_llm_calls_per_message=200` already
prevent infinite loops. The timeout kills legitimate long-running jobs (multi-step
install + process + verify flows).
**Files:** `kiso/worker/loop.py`, `kiso/config.py`
**Change:**
- Remove the job timeout check block in `_plan_and_execute` (loop.py ~line 1440):
  ```python
  if elapsed > job_timeout:
      ...
  ```
- Remove `job_timeout` parameter from `_plan_and_execute` and
  `_handle_new_message` signatures.
- Remove `job_timeout` from `SETTINGS_DEFAULTS` and config template in config.py.
- Keep backward compat: if someone has `job_timeout` in their config.toml, it's
  just ignored (no parse error needed).
**Test:** Existing planning loop tests must pass. Verify the timeout block is gone.
