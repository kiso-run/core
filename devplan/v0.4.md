# Devplan v0.4 — Post-Install Health, Validation UX & Planner Discipline

Goal: fix the remaining failure modes from the real browser-skill trace. The skill
installs "successfully" but system deps are missing. The planner wastes ~12 LLM
calls on validation retries and ignores the reinstall rule when deps fail. The
post-install check_deps call is broken (passes empty dict).

Derived from: real trace where kiso skill install browser succeeds, but
Playwright has 70+ missing system libraries. Planner sends null args 3x despite
clear error messages. When deps fail, planner tries manual apt-get instead of
skill reinstall.

---

## Problem Inventory

### P1 — Post-install check_deps passes empty dict (always returns [])
`_plugin_install` (plugin_ops.py:249) calls `check_deps_fn({"path": str(plugin_dir)})`.
But `check_deps()` reads `skill.get("deps", {}).get("bin", [])` — the dict has
no "deps" key, so it always returns empty. The "missing binaries" warning never
fires, even when deps are genuinely missing.

### P2 — Skill args validation error doesn't show expected format
When `validate_plan` catches missing required args (brain.py:380-383), the error
says `"skill 'browser' args invalid: missing required arg: action"` but doesn't
show what the args should look like. The planner (deepseek-v3.2) sends `null`
again because it doesn't know the format. This caused 3 validation retries.

### P3 — Planner ignores broken skill reinstall rule
M180 added the rule: "if broken, plan `kiso skill remove NAME && kiso skill
install NAME`". But in the trace, the planner instead tried manual `apt-get
install` for individual libraries. The rule is too weak — it's one line among
many, and deepseek-v3.2 doesn't follow it reliably.

### P4 — Planner tries apt-get for skill deps (should be blocked)
Skill system deps should only be installed via `deps.sh` (part of `kiso skill
install`). Manual `apt-get install` for skill libraries is wrong because:
(a) the list is incomplete/guessed, (b) it doesn't match what `deps.sh` does,
(c) it often times out. The planner should be blocked from this pattern.

### P5 — Repeated identical validation errors don't escalate
The planner makes the same mistake (null args) 2-3 times in a row. Each retry
sends the same error message with no additional guidance. After 2 identical
failures, the error should include a concrete example to break the loop.

### M189 — Remove exec_timeout from subprocess execution
**Problem:** `exec_timeout` (300s) kills local commands (skill installs, model
downloads, builds) that are legitimately slow. The setting is overloaded: it's
used both as subprocess timeout AND as fallback LLM HTTP timeout.
**Files:** `kiso/worker/loop.py`, `kiso/worker/utils.py`, `kiso/worker/exec.py`,
`kiso/worker/skill.py`
**Change:**
- In `_run_subprocess` (utils.py): accept `timeout=None` and skip
  `asyncio.wait_for` wrapping when None (just `await proc.communicate()`).
- In `_handle_exec_task` (loop.py:867): pass `timeout=None` instead of
  `ctx.exec_timeout`.
- In `_handle_skill_task` (loop.py:765): pass `timeout=None` instead of
  `ctx.exec_timeout`.
- Remove `exec_timeout` from `_PlanExecContext` — it's no longer needed for
  subprocess calls. LLM calls already have their own dedicated timeouts.
**Keep:** `exec_timeout` setting stays in config.py as the fallback LLM timeout
(used by `call_llm` for curator/summarizer/etc.). Rename the config comment to
clarify it's for LLM calls only.
**Test:** Existing exec tests must pass. Add a test verifying that `_run_subprocess`
with `timeout=None` doesn't wrap in `wait_for`.

### M192 — Planner: navigate-vs-search rule & install-then-replan guard
**Problem:** The planner uses `search` when the user asks to "navigate to" or "visit"
a URL. Search queries a search engine — it doesn't render or interact with web pages.
When the user says "go to guidance.studio", Perplexity returns info about a
completely different site (guidestudio.com). Additionally, the planner repeatedly
tries to use a skill in the same plan where it installs it (3 validation retries).
**Files:** `kiso/roles/planner.md`
**Change:** Add two rules:

1. **Navigate-vs-search** — In the "Web interaction" section, strengthen the
existing guidance:
```
- When the user asks to visit, navigate, browse, or interact with a specific URL,
  the browser skill is REQUIRED — do NOT use search. Search queries search engines
  and may return unrelated results. The browser skill visits the actual page.
  If the browser skill is not installed, install it first (exec + replan).
```

2. **Install-then-replan guard** — In the "Skills efficiency" section:
```
- You CANNOT use a skill that is not listed in the Skills section below. If you
  need an uninstalled skill, your plan MUST be: (1) exec task to install it,
  (2) replan task. The skill becomes available only after install completes.
  NEVER put a skill task for an uninstalled skill in the same plan as its install.
```

**Test:** Read planner.md, verify both rules are present. Unit test: build planner
messages and verify the prompt contains "browser skill is REQUIRED" and
"CANNOT use a skill that is not listed".

### M193 — Stronger "skill not installed" validation message
**Problem:** When `validate_plan` rejects a plan with an uninstalled skill, the
error message says "Use an exec task with `kiso skill install NAME` to install it
first, then replan to use it." But deepseek-v3.2 ignores this and retries with the
same structure. The message needs to be more explicit about what the corrected plan
should look like.
**Files:** `kiso/brain.py` — `validate_plan`
**Change:** Make the error message include a concrete plan structure example:
```python
errors.append(
    f"Task {i}: skill '{skill_name}' is not installed. "
    f"Available skills: {available}. "
    f"You CANNOT use '{skill_name}' in this plan. Remove the skill task. "
    f"Correct structure: "
    f'[{{"type": "exec", "detail": "Install the {skill_name} skill", ...}}, '
    f'{{"type": "replan", "detail": "Use {skill_name} after install", ...}}]'
)
```
**Test:** Validate a plan with an uninstalled skill. Verify the error contains
"CANNOT use" and the corrected structure example.

### M194 — Reviewer: cross-check search domain against requested URL
**Problem:** When a search task's `detail` contains a specific URL
(e.g. "visit https://guidance.studio"), but the search output talks about a
different domain (guidestudio.com), the reviewer marks it "ok" because the `expect`
is vaguely satisfied. The reviewer should detect domain mismatches.
**Files:** `kiso/roles/reviewer.md`
**Change:** Add a rule for search tasks with explicit URLs:
```
- Search domain check: if the task detail contains a specific URL/domain and the
  search output describes a DIFFERENT site (different domain, different company),
  mark as replan with reason "search returned results for wrong domain". The user
  asked about a specific site — results from a different site are not useful.
```
**Test:** Build reviewer messages with detail containing "https://guidance.studio"
and output mentioning "guidestudio.com". Verify the prompt contains the domain
check rule.

### M195 — Auto-correct plan when planner repeats same validation error
**Problem:** When the planner makes the same validation error 3 times (e.g. using
an uninstalled skill), M186 adds an escalation message but the planner still
fails. After `max_validation_retries`, the plan fails entirely and the user gets
an error. Instead, the system should auto-correct the plan.
**Files:** `kiso/brain.py` — `_retry_llm_with_validation` or `run_planner`
**Change:** When `validate_plan` exhausts retries and the last error is "skill not
installed", auto-correct the plan:
- Remove the offending skill task
- Replace with: exec `kiso skill install NAME` + replan
- Log a warning: "Auto-corrected plan: replaced uninstalled skill task with
  install + replan"

This is a targeted fix for the most common validation failure pattern. Other
validation errors (missing expect, wrong ordering) are still hard failures.
**Test:** Mock a planner that always returns a plan with an uninstalled skill.
Verify that after max retries, `run_planner` returns an auto-corrected plan with
exec install + replan instead of raising PlanError.
