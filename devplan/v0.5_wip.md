# Devplan v0.5 — Learning Quality Pipeline + Verbose & UX Quality

## Problem Statement

The reviewer→curator learning pipeline produces low-quality, fragmented, context-less
learnings that waste tokens and pollute the knowledge base. A single page view of
guidance.studio generated 26 learnings — most duplicated, lacking subject context,
containing ephemeral data (browser element indices), or splitting one concept into
per-field items.

In verbose mode, IN panels for LLM calls are rendered twice for msg tasks (P6-P7, fixed
in M324-M326). Broader end-to-end browser sessions reveal further UX issues: the
messenger speaks in third person, the planner re-navigates redundantly, the reviewer
false-rejects form fills, replan notifications are in English regardless of user language,
learnings don't show tags in CLI, the curator doesn't generate entity-scoped tags, pending
tasks render as "started" before execution, and the planner hallucinates skill arg names.

A second guidance.studio session (contact form submission) revealed deeper replan-loop
pathologies (P16-P20): the reviewer triggers replan on CAPTCHA-blocked tasks it cannot
solve, circular replan detection is too narrow (only compares consecutive pairs),
extend_replan is granted during stuck loops, learning dedup misses paraphrases (50+
learnings for one form), and the planner omits required skill args causing validation
retries. Total: ~165K tokens and ~20 minutes for an impossible task.

## Problems

### P1 — Reviewer over-fragments learnings

The reviewer splits a single observation into up to 5 separate learn items (one per
form field, one per link, one per UI element). Each is a contextless fragment:
`"A submit button [14] is available for the contact form."` — which form? which site?

Root cause: reviewer.md says `max 5` but has no rule against fragmentation. Cheap
models interpret "array of facts" as "list every detail separately".

### P2 — Learnings lack subject context

Learnings like `"The 'Name' field is an input element with type 'text'."` only make
sense in the task context that produced them. As standalone knowledge they're useless.

Root cause: reviewer.md doesn't require learnings to be self-contained. The reviewer
sees the full task context but produces fragments assuming the reader has the same.

### P3 — Ephemeral data in learnings

Browser element indices (`[8]`, `[9]`, `[14]`), "X loaded successfully",
"X installed successfully" — all transient states that end up as "durable facts".

Root cause: reviewer.md says "never transient info" but this is too vague for cheap
models. No explicit examples of what counts as transient.

### P4 — No learning deduplication

`save_learning` does a raw INSERT with no duplicate check. The same fact
("guidance.studio has a contact form") is saved 5 times across different tasks.

Root cause: each reviewer call is independent and doesn't know what was already
learned. No dedup at save time either.

### P5 — Tag explosion from fragmented learnings

7 learnings about one website produced 12 tags. Each fragment gets its own tag set,
creating narrow, non-reusable tags instead of a few broad ones.

Root cause: downstream consequence of P1. More fragments → more tags → diluted
semantic retrieval.

### P6 — msg task verbose path bypasses dedup

`_render_msg_task` (cli/__init__.py:669) calls `render_llm_calls_verbose()` directly.
This function in render.py renders ALL IN+OUT panels unconditionally — it has no access
to `state.inflight_in_shown` and cannot skip already-shown IN panels.

All other task types use `_emit_verbose_calls` → `_print_verbose_panels`, which checks
`inflight_in_shown` and skips IN panels already rendered by inflight.

### P7 — msg tasks excluded from intermediate verbose rendering

The intermediate update path (cli/__init__.py:886) has `ttype != "msg"`, which blocks
verbose panel rendering for msg tasks while they're still running. This means when a
msg task runs briefer→messenger sequentially, the briefer OUT panel only appears when
the entire msg task completes (together with all other panels), instead of appearing
as soon as the briefer call finishes.

For exec tasks, OUT panels appear one by one during execution. For msg tasks, they all
dump at the end. This inconsistency makes the output confusing.

### P8 — Messenger voice breaks user illusion

Voice rules instruct the messenger to use third-person for upcoming actions ("The browser
skill will be installed"). The user sees Kiso as a single entity and expects first-person
("I'll install the browser skill"). This creates a jarring disconnect.

### P9 — Planner re-navigates redundantly

The planner web module has no rule about browser state persistence. Every new plan starts
with navigate + snapshot even when the browser is already on the target page. Browser
state (cookies, localStorage) persists between skill calls in the same session.

### P10 — Reviewer false-rejects form fills

The browser skill's `fill` action returns `Filled '[N]'. URL: ...` without echoing the
filled value. The reviewer interprets `expect` literally and demands to see the text in
the output, causing false replans on successful fills. (Primary fix is in the browser
skill — echoing value. This milestone adds a tolerance rule as defense-in-depth.)

### P11 — Planner hallucinates skill arg names

The planner (deepseek-v3.2) sometimes uses `selector` instead of `element` or `text`
instead of `value` for the browser fill action. The current `validate_skill_args` catches
missing required args but doesn't auto-correct common aliases.

### P12 — Pending tasks shown as "started"

When a plan is created, all tasks get rendered via `render_task_header` with their type
icon (⚡ for skill, 💬 for msg) even while still "pending". This creates a wall of
"started" tasks before any execution begins, looking like consecutive INs without OUTs.

### P13 — Replan notifications in English

Replan messages ("Replanning (attempt 1/3): ...") are hardcoded in English in
`_execute_loop`. The user's language is known from the original message but not used.

### P14 — Learning tags invisible in CLI

The curator assigns tags to promoted facts, but `render_review` only shows the fact text.
Tags are useful for the user to understand how facts are categorized.

### P15 — No entity-scoped tags

The curator generates generic tags (homepage, contact, form) but never entity-specific
tags. When the user interacts with guidance.studio, facts should be tagged
`entity:guidance.studio` for scoped retrieval.

### P16 — Reviewer replans on unsolvable failures

The reviewer has only two statuses: `ok` and `replan`. When a task fails for a reason
outside the system's control (CAPTCHA requiring human verification, authentication,
rate limiting), the reviewer still says `replan` — because `expect` wasn't met. This
triggers the full replan machinery, wasting tokens and time on an impossible goal.

In a guidance.studio session, the reviewer said "replan" 6 times for the same CAPTCHA
problem. Each replan triggered briefer+planner+reviewer calls, consuming ~165K tokens
total for a task that should have stopped after the first failed submit attempt.

Root cause: the reviewer prompt only knows `ok` (succeeded) and `replan` (failed). It
has no way to express "failed AND not fixable". The planner receives replan context and
tries again, often with trivially different selectors or redundant form-fill sequences.

### P17 — Circular replan detection too narrow

The circular replan detector compares only the last 2 entries: failure word overlap >60%
OR strategy fingerprint Jaccard >50%. But failures cycle through different wordings
("name field not found" → "button not enabled" → "button not found" → "invalid selector")
that all stem from the same root cause (CAPTCHA blocks submit). The planner also varies
its strategy enough each time (navigate→snapshot→fill→click vs snapshot→fill→click) to
evade fingerprint detection.

Root cause: detection only looks at consecutive pairs ([-2] vs [-1]), not the full
history. Different failure *symptoms* map to the same unsolvable root *cause*.

### P18 — extend_replan granted when stuck

The planner can request `extend_replan` even during circular/stuck situations. In the
guidance.studio session, the planner extended the budget from 3 to 6 attempts on an
unsolvable CAPTCHA problem, doubling the wasted tokens.

Root cause: `extend_replan` is granted purely based on the global cap
(`_MAX_EXTEND_REPLAN=3`), with no check for stuck patterns or repeated failures.

### P19 — Learning dedup misses paraphrases

The 70% word-overlap dedup catches near-duplicates but misses semantic paraphrases.
"guidance.studio has a CAPTCHA" and "guidance.studio contact form has CAPTCHA detection"
have <70% word overlap but convey the same fact. The session produced ~50 learnings,
most semantically duplicated.

Root cause: word-level Jaccard without normalization is too coarse. Function words
(has/a/the/is/on/in/for/with) inflate the denominator, deflating the ratio below 0.7
even for semantically identical facts.

### P20 — Planner omits required skill args entirely

M330 handles wrong arg *names* (selector→element). But the planner (deepseek-v3.2)
frequently produces `args: null` or `args: "{}"` for browser skill tasks, omitting
the required `action` field entirely. Each occurrence triggers a validation retry
(re-sending the full planner prompt, ~5-7K tokens).

In the session, this happened at least 3 times, each requiring 1-2 validation retries
at 13-30s each. The validation error message already includes an example, but by then
the retry cost is incurred.

Root cause: the planner prompt shows args format in the task type section but doesn't
emphasize that skill tasks MUST always have a complete args JSON string with all required
fields. The model occasionally "forgets" and relies on a downstream translator that
doesn't exist for skills.

   def auto_correct_skill_args(args: dict, args_schema: dict) -> dict:

### P21 — Entity is a tag prefix, not a first-class concept

Facts about named entities (websites, tools, companies) are tagged with an `entity:`
prefix convention (M334). But there's no entity *model*: no registry, no dedup across
entities, no entity-scoped retrieval, no way to browse "everything we know about X".

The curator assigns `entity:guidance.studio` as a tag string — it's not validated,
not normalized, and has no relationship structure. Two learnings about the same entity
can produce `entity:guidance.studio` and `entity:www.guidance.studio` with no dedup.

Every fact is about something (a subject). If the curator can't identify the subject,
the fact is either too generic (discard) or poorly written (rewrite). Entity should be
NOT NULL — its absence is a quality signal.

Root cause: M334 was a lightweight first step (tag prefix). A proper entity model
requires schema changes, curator schema updates, and briefer integration.

### P22 — Curator doesn't see existing facts for same entity

When curating new learnings about guidance.studio, the curator doesn't know what facts
already exist for that entity. It promotes "guidance.studio has a contact form" even
though that fact was already promoted 3 sessions ago.

Root cause: `build_curator_messages` receives available *tags* but not existing facts.
The curator has no way to check for redundancy against the knowledge base.

without unifying with existing ones.

### P59 — Entity backfill uses substring match (false positives)

`backfill_fact_entities` checks `if entity["name"] in content_lower` — the
entity "java" matches "javascript", "sql" matches "sqlite"/"mysql"/"postgresql",
"go" matches "google"/"going".  First matching entity wins (`break`).

Root cause: M382 implemented quick substring check for backfill.  Should use
word-boundary matching.

### P60 — Briefer skill filter uses substring match (false positives)

"dockerize" passes because "docker" is a substring.

Root cause: M368 used substring matching for speed; should use word-boundary
or exact skill-name matching.

### P61 — `find_or_create_entity` ignores kind mismatch

If entity exists with kind="tool" and curator promotes with kind="framework",
the existing entity is reused without updating kind.  The first caller's kind
wins permanently.

Root cause: `find_or_create_entity` only checks name, ignores kind on reuse.

Functional tests (F1-F8) only cover browser navigation, SSH key, git+aider,

- chat_kb fast path (classifier → briefer → messenger, no planner)

### P66 — No way to cancel a running job

Once a plan starts executing, there is no user-facing mechanism to stop it. The
`cancel_event` exists in the worker loop and `main.py` sets it on session delete,
but there is no dedicated API endpoint or CLI command to cancel a running job
without destroying the session. If the bot starts an unintended destructive
operation (e.g., `rm -rf /data`), the user must kill the process or delete the
entire session.

Root cause: cancel was designed as an internal cleanup mechanism, not a
user-facing control.

### P67 — New messages during active jobs are silently queued

When a job is running, incoming messages go into the asyncio queue and execute
sequentially after the current job finishes. There is no classification of
whether the new message is:
- A stop/cancel command ("ferma tutto", "stop")
- An update to the current job ("actually, use port 8080 instead")
- An independent request that can wait
- A conflicting request that should cancel the current job

This caused real-world issues: the OpenClaw incident where a harmless-sounding
instruction triggered unintended behavior that could not be interrupted.

Root cause: the queue is FIFO with no priority or classification.

### P68 — No persistent safety rules

Critical operational constraints (e.g., "never delete /data", "always ask before
`rm`", "production DB is read-only") can only be expressed as transient
learnings that may be decayed or compressed. There is no "pinned" fact category
that is always injected into the planner regardless of briefer selection.

Root cause: all facts are equal in the knowledge store — no priority tier.

### P69 — No CLI for managing safety rules

Even if safety facts existed (P68), there is no CLI to list/add/remove them.
Admins must interact via chat to teach the bot safety constraints, which is
error-prone and not auditable.

Root cause: safety rules don't exist yet as a concept.

### P70 — README doesn't document runtime safety features

The README's Philosophy section mentions sandbox, prompt injection defense,
review gates, and curated knowledge — but nothing about:
- Job cancellation (`kiso cancel`, REST endpoint)
- In-flight message classification (stop/update/conflict detection)
- Safety rules (`kiso rules`, always-inject, reviewer compliance)
- Scored fact retrieval (entity + tag + keyword ranking)

These are core differentiators vs. other agent bots and should be prominently
documented for both end users and wrapper/integration developers.

**Problem:** P58

**Files:** `kiso/store.py` — new unified search function

**Change:**

1. Add `search_facts_scored(db, entity_id?, tags?, keywords?, limit=50)`:
   ```python
   async def search_facts_scored(
       db: aiosqlite.Connection,
       entity_id: int | None = None,
       tags: list[str] | None = None,
       keywords: list[str] | None = None,
       session: str | None = None,
       is_admin: bool = True,
       limit: int = 50,
   ) -> list[dict]:
       """Score and rank facts by relevance.

       Score = (entity_match × 10) + (tag_overlap_count × 3) + (keyword_hit × 1)
       Returns top `limit` facts ordered by score desc, then last_used desc.
       """
   ```
2. Implementation strategy — single SQL query with scoring:
   ```sql
   SELECT f.*,
       CASE WHEN f.entity_id = ? THEN 10 ELSE 0 END AS entity_score,
       COALESCE(t.tag_count, 0) * 3 AS tag_score
   FROM facts f
   LEFT JOIN (
       SELECT fact_id, COUNT(*) AS tag_count
       FROM fact_tags
       WHERE tag IN (?, ?, ...)
       GROUP BY fact_id
   ) t ON t.fact_id = f.id
   WHERE (f.entity_id = ? OR t.tag_count > 0)
   ORDER BY (entity_score + tag_score) DESC,
            COALESCE(f.last_used, f.created_at) DESC
   LIMIT ?
   ```
3. Keyword scoring done in Python post-query (FTS5 MATCH for filtering,
   then count keyword hits for ranking).
4. Session scoping: non-admin users only see facts where
   `session IS NULL OR session = ?`.

- Tags-only query: fact with 3 matching tags scores 9, 1 tag scores 3
- Combined: entity + 2 tags = 10+6 = 16, beats entity-only (10) and tags-only (6)
- Keywords boost: among same-score facts, keyword matches rank higher
- Limit respected
- Session scoping for non-admin

### M417 — README: document runtime safety and intelligence features

**Problem:** P70

**Files:** `README.md`

**Change:**

1. Add to **Philosophy** section, after the existing "Review gates" paragraph:

   **Runtime kill switch.** Any running job can be cancelled instantly —
   via CLI (`kiso cancel`), REST API (`POST /sessions/{sid}/cancel`), or
   programmatically from a wrapper. No session destruction required, no
   process kill. The bot confirms what was completed, what was cancelled.

   **In-flight message triage.** New messages arriving during an active job
   aren't blindly queued. A fast-path catches stop commands ("ferma",
   "STOP", "cancel") in milliseconds without LLM calls. Everything else is
   classified: updates modify the running plan, conflicts replace it,
   independent requests wait their turn with an immediate ack.

   **Safety rules.** Persistent, admin-defined constraints (`kiso rules add
   "never delete /data"`) that are always injected into the planner — not
   gated by the briefer, not subject to decay or compression. The reviewer
   flags violations as stuck, blocking execution.

2. Add to **Usage** section, after the existing commands:
   ```
   kiso cancel               # cancel the active job in current session
   kiso cancel <session>     # cancel a specific session's job
   kiso rules list           # show safety rules
   kiso rules add "..."      # add a safety rule
   kiso rules remove <id>    # remove a safety rule
   ```
