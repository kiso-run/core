# Devplan v0.5 — Learning Quality Pipeline

## Problem Statement

The reviewer→curator learning pipeline produces low-quality, fragmented, context-less
learnings that waste tokens and pollute the knowledge base. A single page view of
guidance.studio generated 26 learnings — most duplicated, lacking subject context,
containing ephemeral data (browser element indices), or splitting one concept into
per-field items.

## Problems

### P1 — Reviewer over-fragments learnings

The reviewer splits a single observation into up to 5 separate learn items (one per
form field, one per link, one per UI element). Each is a contextless fragment:
`"A submit button [14] is available for the contact form."` — which form? which site?

Root cause: reviewer.md says `max 5` but has no rule against fragmentation. Cheap
models interpret "array of facts" as "list every detail separately".

### P2 — Learnings lack subject context

Learnings like `"The 'Name' field is an input element with type 'text'."` only make
sense in the task context that produced them. As standalone knowledge they're useless.

Root cause: reviewer.md doesn't require learnings to be self-contained. The reviewer
sees the full task context but produces fragments assuming the reader has the same.

### P3 — Ephemeral data in learnings

Browser element indices (`[8]`, `[9]`, `[14]`), "X loaded successfully",
"X installed successfully" — all transient states that end up as "durable facts".

Root cause: reviewer.md says "never transient info" but this is too vague for cheap
models. No explicit examples of what counts as transient.

### P4 — No learning deduplication

`save_learning` does a raw INSERT with no duplicate check. The same fact
("guidance.studio has a contact form") is saved 5 times across different tasks.

Root cause: each reviewer call is independent and doesn't know what was already
learned. No dedup at save time either.

### P5 — Tag explosion from fragmented learnings

7 learnings about one website produced 12 tags. Each fragment gets its own tag set,
creating narrow, non-reusable tags instead of a few broad ones.

Root cause: downstream consequence of P1. More fragments → more tags → diluted
semantic retrieval.

---

### M318 — Reviewer learn rules: self-contained, consolidated, no ephemera

**Problem:** P1, P2, P3

**Files:** `kiso/roles/reviewer.md` — add specific learn rules

**Change:**

1. Change `max 5` to `max 3` in the learn field description (forces consolidation)
2. Add explicit learn rules after the existing `learn: only durable facts...` line:
   - Each learning MUST be self-contained: include subject (site name, project, tool).
     Bad: `"has a contact form"`. Good: `"guidance.studio has a contact form at /venture-launchpad"`.
   - Consolidate related observations into one learning. Never split per-field/per-element.
     Bad: 5 items for each form field. Good: 1 item describing the form.
   - Never include ephemeral data: browser element indices `[N]`, internal IDs,
     session-specific paths, temporary file locations.
   - `"X installed/loaded/ran successfully"` is a transient state, not a durable fact → null.

**Test:**
- `test_brain.py`: verify reviewer prompt contains the new rules (substring assertions)
- `test_brain.py`: verify `max 3` in the learn description
- Existing reviewer tests must still pass

### M319 — Learning dedup at save time

**Problem:** P4

**Files:** `kiso/store.py` — add similarity check in `save_learning`

**Change:**

1. Before INSERT in `save_learning`, query recent learnings (same session, status='pending')
   for substring overlap: if new content is ≥70% contained in an existing learning
   (or vice versa), skip the save and return 0
2. Use simple normalized comparison: lowercase, strip whitespace, check if shorter
   string is a substring of longer, or word-overlap ratio ≥ 0.7
3. Log a debug message when dedup triggers: `"Learning deduped against id=%d"`

**Test:**
- `test_store.py`: saving the same learning twice returns 0 on second call
- `test_store.py`: saving a near-duplicate (subset of words) returns 0
- `test_store.py`: saving a genuinely different learning returns a valid id
- `test_store.py`: learnings in different sessions are NOT deduped against each other

### M320 — Reviewer learn validation in brain.py

**Problem:** P1, P2, P3

**Files:** `kiso/brain.py` — add learn validation in `validate_review`

**Change:**

1. In `validate_review`, add soft validation for `learn` items (warnings, not errors —
   don't block the review, just filter bad items):
   - Filter out items shorter than 15 chars (too fragmentary)
   - Filter out items matching ephemeral patterns: bare `[N]` element indices
     (regex: `\[\d+\]` appearing 2+ times), "installed successfully",
     "loaded successfully", "ran successfully"
2. Add `_clean_learn_items(items: list[str]) -> list[str]` function that applies these
   filters. Call it in `_run_review_step` (loop.py) before saving learnings.
3. Log filtered items at debug level

**Test:**
- `test_brain.py`: `_clean_learn_items` filters short items
- `test_brain.py`: `_clean_learn_items` filters "X installed successfully"
- `test_brain.py`: `_clean_learn_items` filters items with 2+ `[N]` indices
- `test_brain.py`: `_clean_learn_items` preserves valid learnings
- `test_brain.py`: items at boundary (exactly 15 chars) kept

### M321 — Curator prompt: explicit output format + consolidation rule

**Problem:** P5 + curator clarity

**Files:** `kiso/roles/curator.md` — strengthen prompt

**Change:**

1. Add consolidation rule: "If multiple learnings describe the same subject, evaluate
   them as a group — produce ONE evaluation with a consolidated fact, not N separate ones.
   Set learning_id to the first in the group."
2. Add explicit note: "Each evaluation must include learning_id matching the `[id=N]`
   from the input."
3. Strengthen discard examples: add `"X loaded/installed successfully"`,
   `"page has element type text/email"`, `"form has N fields"` (per-field details)

**Test:**
- `test_brain.py`: curator prompt contains "consolidat" substring
- `test_brain.py`: curator prompt contains "learning_id"
- Existing curator tests must still pass

---

### M322 — Curator validation: consolidation awareness

**Problem:** P5

**Files:** `kiso/brain.py` — relax `validate_curator` expected_count check

**Change:**

1. In `validate_curator`: change the expected_count check from exact match to
   `len(evals) > expected_count` (error only if MORE evaluations than learnings,
   fewer is fine — consolidation reduces count)
2. Add validation: promoted facts must be ≥ 10 chars (reject trivially short facts)

**Test:**
- `test_brain.py`: curator with fewer evaluations than learnings passes validation
- `test_brain.py`: curator with more evaluations than learnings fails validation
- `test_brain.py`: promoted fact with < 10 chars fails validation
- Existing curator tests must still pass
