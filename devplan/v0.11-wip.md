# kiso-run/core — v0.11 Development Plan

## Headline

**Post-launch polish: clearer install UX and naming consistency
across LLM roles.**

v0.10 closed with the wrapper subsystem fully retired (Phase 4),
the model migration to DeepSeek V4 complete (Phase 14), the
default preset trimmed to MCPs-and-skills only, and disk/test
parity verified against a clean slate. v0.11 picks up the
loose ends visible in the first user-facing flows: small wording
problems in `install.sh` and one role name (`mcp_sampling`) that
breaks the otherwise single-word convention shared by every
other role.

These are not behavioral changes — they are clarity changes.
None affects the wire format of any LLM role, the schema of any
plan, or the persistence shape of any DB row. They sharpen what
the user sees and what a contributor reads in the codebase.

## Detailed milestones

### M1567 — `install.sh` user-prompt clarity

**Problem.** The interactive Linux-user prompt in `install.sh`
(at `core/install.sh:249-337`, function `ask_username`) lists
"available users" by filtering `/etc/passwd` for `UID >= 1000`
and excluding `nobody`. On Ubuntu and most modern distros this
filter is too loose: snap packaging assigns UIDs in the same
range, so users see a list like:

```
Available users: snap_daemon, snapd-range-524288-root, ymx1zq
```

In addition, when the user types a name that doesn't exist the
installer responds with `Create it first: sudo useradd -m <name>`
— a hint that reads as "we'll create one if you ask" but actually
just loops the prompt until you give an existing user. The
installer never creates users.

The combined effect is that the prompt looks like an open-ended
"who should this be" picker when in fact it is a "pick one of
the real existing users on this box" picker, with a default
that is almost always correct (`whoami`).

**Approach.**

1. **Filter the available-users list more aggressively.** Drop
   names matching well-known system-package patterns:
   - leading underscore (`_kiso`-style placeholders, `_lxd`, etc.)
   - `snap_*`, `snapd-*` (Ubuntu snap)
   - `lxd`, `pollinate`, `landscape`, `messagebus` (Ubuntu
     daemon users that happen to land in the >=1000 range)
   - `systemd-*`, `dbus`, `gnome-*` (rare but seen)
   The filter is a denylist of substrings/regex applied to the
   awk output. Keep the list narrow — this is a UX hint, not a
   security boundary, and we still validate via `id` later.
2. **Reword the prompt.** New text:
   ```
   Which existing Linux user should own Kiso's files and run tasks?
     Default: <whoami> (you, current user)
     Other existing users: <filtered-list>
   Press Enter to use the default, or type another existing username.

   User [<whoami>]:
   ```
   This makes three things explicit: (a) "existing", (b) the
   default is "you", (c) the list is informational not
   exhaustive. The prompt remains a single line for non-tty
   parity.
3. **Drop the misleading `useradd` hint.** When the user types
   a non-existent name, replace
   ```
   Create it first:  sudo useradd -m <name>
   Then enter the name again, or press Ctrl+C to abort.
   ```
   with
   ```
   That user does not exist on this system.
   Use one of the existing users above, or press Ctrl+C to abort.
   ```
   The installer never invokes `useradd`, so the prior hint
   was misleading. If the operator genuinely wants a new user,
   they will create it and re-run.
4. **Same fixes apply when `--user <name>` is passed via flag**
   (the non-interactive branch at `install.sh:258-271`): the
   error message there also says `Create it first: sudo useradd`
   and should be aligned with the new wording. Keep the exit
   code (currently `1`) — flag mode is non-interactive, so
   "exit and let the operator fix it" is correct.

**Test plan.**

- New cases in `core/tests/test_install_sh.py` (BATS-style
  pytest harness around `install.sh` execution):
  - `test_system_users_filtered_out` — set up a temp passwd
    fixture with `snap_daemon`, `_lxd`, `ymx1zq` at UID>=1000;
    assert the available-users list contains only `ymx1zq`.
  - `test_nonexistent_user_rejected_no_useradd_hint` — drive
    the interactive prompt with input `nope`, assert stderr
    contains `does not exist` and does NOT contain
    `useradd`.
  - `test_flag_user_nonexistent_message_aligned` — invoke
    `install.sh --user nope`, assert exit code 1 and stderr
    contains the new wording.
  - Pre-existing happy-path tests (default user accepted,
    valid user accepted) should keep passing untouched.
- No live-LLM or Docker integration test is required; this
  is shell-only.

**Files.**

- `core/install.sh` (function `ask_username`, ~80 lines)
- `core/tests/test_install_sh.py` (3 new test cases)

**Out of scope.**

- The TAB-completion logic (`bind -x` callback) stays as-is.
- Other prompts in `install.sh` (bot name, instance name,
  provider, models) are unchanged.
- No change to flag semantics: `--user`, `--bot-name`,
  `--instance` keep their current contracts.

### M1568 — Rename role `mcp_sampling` → `sampler`

**Problem.** Every LLM role in Kiso uses a single-word lowercase
identifier:

```
briefer, classifier, planner, reviewer, curator, worker,
summarizer, paraphraser, messenger, consolidator, mcp_sampling
```

`mcp_sampling` is the odd one out: it has an underscore, a
prefix, and reads as a *protocol verb* rather than a *role*.
The naming made sense when the feature shipped (M1529, "MCP
`sampling/createMessage` support"), because it tracked the
protocol method 1:1. But from the user's perspective — sitting
at the install prompt scrolling through the roles list — it
looks like an MCP-internal control rather than "the LLM that
fulfils sampling requests from MCP servers". The role's *job*
is sampling on behalf of MCP servers; calling it `sampler` says
that without coupling to the wire protocol.

**Approach.**

This is a pure rename. No semantics change. No backward
compat — pre-release codebase, no users have `mcp_sampling` in
their `~/.kiso` configs that we can't migrate by re-running
`kiso init` (and any operator who already has a config can
update one key).

1. **File rename.** `core/kiso/roles/mcp_sampling.md` →
   `core/kiso/roles/sampler.md`. Body header (the human-readable
   "You are X" sentence) updated to "You are a sampler".
2. **Config tuple.** In `core/kiso/config.py`:
   - `_MODEL_METADATA` — the entry
     `("mcp_sampling", "google/gemini-2.5-flash", "fulfils sampling/createMessage requests from MCP servers")`
     becomes `("sampler", "google/gemini-2.5-flash", "fulfils sampling/createMessage requests from MCP servers")`.
     The description stays — it correctly explains what the role
     does without depending on the role *name* matching the
     protocol verb.
   - `CONFIG_TEMPLATE` — the line
     `mcp_sampling = "google/gemini-2.5-flash"` becomes
     `sampler = "google/gemini-2.5-flash"`.
   - `REASONING_DEFAULTS` — currently has no entry for this
     role; if any reference exists, update.
3. **Call sites.** Wherever core asks "which model handles
   sampling requests", replace the role lookup key. Expected
   sites (verify before editing):
   - `core/kiso/mcp/sampling.py` — the dispatcher that turns an
     incoming `sampling/createMessage` into an LLM call.
   - `core/kiso/mcp/http_client.py` — the HTTP MCP transport
     that handles bidirectional sampling (M1544).
   - `core/kiso/api/runtime.py` — only if it surfaces the role
     externally (e.g. in `/health` or `/config`).
4. **Tests.** Search-and-replace `mcp_sampling` → `sampler`
   across `core/tests/`. Likely-affected files:
   - `test_mcp_sampling.py` (and rename to `test_sampler.py`
     for consistency, if the existing name pins the protocol
     verb).
   - any `test_config*.py` that asserts `MODEL_DEFAULTS` shape.
   - `test_health.py` if it checks the role list.
   - `test_v010_pipeline_live.py` and similar full-stack tests.
5. **Docs.** Search `docs/` for `mcp_sampling`:
   - `docs/llm-roles.md` (or equivalent)
   - `docs/mcp.md`
   - `docs/architecture.md` if it lists roles
   Replace with `sampler` and update one-line descriptions where
   appropriate.
6. **Trust / config-checks.** `core/kiso/config_checks.py`
   (M1542) validates per-user role allowlists; check whether
   `mcp_sampling` is enumerated there explicitly, and align.

**Test plan.**

- Pure unit-suite job. Run `uv run pytest` from `core/`; expect
  all 4500+ tests to remain green after the rename. No new
  behavioral test is needed — the change is a rename of a
  string identifier.
- Add one targeted regression test in `test_config.py` (or
  equivalent): `test_no_legacy_mcp_sampling_key` — assert that
  `MODEL_DEFAULTS`, `CONFIG_TEMPLATE`, and `_MODEL_METADATA` do
  not contain `"mcp_sampling"` anywhere. This lock prevents
  silent partial reverts.
- BATS test (if any) that runs `install.sh` should be re-run to
  confirm the prompt still renders all 11 roles, with `sampler`
  in place of `mcp_sampling`.

**Files (estimated, verify on entry).**

- `core/kiso/roles/mcp_sampling.md` → `sampler.md` (rename + header)
- `core/kiso/config.py` (3 sites in `_MODEL_METADATA` /
  `CONFIG_TEMPLATE` block)
- `core/kiso/mcp/sampling.py`
- `core/kiso/mcp/http_client.py` (if referenced)
- `core/kiso/config_checks.py`
- `core/tests/test_*` — search-and-replace
- `core/docs/*.md` — search-and-replace

**Out of scope.**

- The `sampling/createMessage` protocol verb stays as-is — that
  is owned by the MCP spec and we cannot rename it.
- Behavior of the sampler is unchanged: same prompt, same model
  default, same dispatcher.
- No deprecation alias for `mcp_sampling` — pre-release, single
  cut.

### M1569 — `install.sh` flow + layout polish

**Problem.** Beyond M1567 (user-prompt wording), `install.sh`
has accumulated smaller flow / layout / wording rough edges that
each take 5-30 lines to fix and together meaningfully sharpen
the first-run experience. None changes installer semantics or
what gets written to `config.toml`.

Bundled into one milestone because they all touch the same file
and share the same test surface
(`core/tests/test_install_sh.py`); split per-item at the test
level for granularity.

**Approach — ten sub-changes.**

1. **Move the `Rebuild and restart?` prompt to AFTER config
   write.** Today:
   ```
   Container exists. Rebuild and restart? [Y/n]    ← here
   old container removed
   Configuring...
   [27 prompts]
   Write this config? [Y/n]
   ```
   The user is asked to destroy the running container *before*
   configuration. If they change their mind mid-config, the
   container is already gone. Correct order:
   ```
   Configuring...
   [27 prompts]
   Write this config? [Y/n]
   Container exists. Restart with new config? [Y/n]    ← here
   ```
   The "Y" branch removes-and-rebuilds; "n" leaves the running
   container alone (the new config takes effect on next manual
   restart). Single `restart_container()` call after write.

2. **Indent consistency: all field prompts at 2 spaces.** Today
   some prompts indent with 2 spaces
   (`  Bot name [Kiso]:`,  `  RAM in GB [4]:`), some with 0
   (`User [ymx1zq]:`, `Provider name [openrouter]:`,
   `Identifier [kiso]:`, `LLM provider URL [...]:`,
   `Provider name [openrouter]:`). One pass over `install.sh`
   to normalize every interactive prompt to a 2-space prefix.
   No structural change — only alignment.

3. **External URL wording.** Today:
   ```
   External URL (for file download links):
   ```
   The URL is also used for callbacks and shown by MCPs. New
   wording:
   ```
   Public URL where this Kiso is reachable
     (used in download links, webhook callbacks, MCP outputs):
   ```

4. **Network warning made actionable.** After choice 2
   (Public), today:
   ```
   ⚠  API exposed over HTTP (no encryption).
   Make sure port 8333 is reachable from outside ...
   For production use, set up HTTPS with a reverse proxy.
   See: docs/https.md — we recommend Caddy + Let's Encrypt.
   ```
   Add one closing line:
   ```
   You can switch back to local-only anytime by editing
   ~/.kiso/instances/<name>/config.toml (set host = "127.0.0.1").
   ```

5. **Persona post-prompt hint.** After the user types/accepts
   `Bot persona`, emit one line before continuing:
   ```
   (To change later: edit `bot_persona` in
    ~/.kiso/instances/<name>/config.toml and `kiso restart`.)
   ```
   Surfaces the escape hatch without making the prompt itself
   an editor (the user explicitly asked to keep the single-line
   prompt unchanged).

6. **Decorative consistency.** The `─────...` divider appears
   only around the config preview block. Drop it — the preview
   is already clearly delimited by its `Config preview` header
   and the trailing `Write this config? [Y/n]` prompt. The
   `Config preview` header line stays.

7. **Final prompt: add `edit` as third option.** Today
   `Write this config? [Y/n]` is all-or-nothing — saying `n`
   discards every answer and exits. Change to `[Y/n/edit]`:
   - `Y` (default) → write
   - `n` → discard, exit
   - `edit` → loop back to the first prompt with all current
     answers as defaults; the user re-walks the flow and
     re-confirms at the end. No file is written until `Y`.

8. **Move `Bot persona` next to `Bot name`.** Today persona
   is asked between `User` and `Provider`, in the middle of
   what should be the Instance section. Move the prompt to
   appear immediately after `Bot name`, so the user answers
   both bot-identity questions in sequence before moving on
   to instance / provider. No change to prompt text or default
   value. Pure reorder of the function-call sequence in
   `install.sh:main`.

9. **API key inline in Provider block, drop the pre-flight
   announcement.** Today `install.sh` emits, very early in
   the flow:
   ```
   /home/.../.env not found — will ask for API key.
   ```
   …and then asks the API key only ~30 prompts later, after
   the config preview. The early notice plants an expectation
   that takes a long time to land, and the prompt itself is
   logically detached from the provider it credentials.

   Two changes:
   - Remove the pre-flight `will ask for API key` line entirely.
   - Move the `ask_api_key` call so it runs immediately after
     the provider name + base URL prompts, as the third field
     of the Provider block. The `ENV_FILE` write at the end
     of `install.sh` (currently lines 965-985) stays unchanged —
     only the time of the prompt moves, not the destination of
     the value.

   **Hard requirement: the entered API key MUST NOT be echoed
   to the terminal at any point.** The current implementation
   uses `safe_read -rsp` (`-s` flag) at `install.sh:378` —
   preserve that flag in the new location. Add an explicit
   regression test that asserts the key value never appears in
   stdout/stderr during the flow. Also assert the key does not
   appear in the config preview block (it should already be
   absent since the preview reads from `config.toml`, which
   never contains the key — it lives in `.env` only).

10. **Provider URL auto-fill for known providers.** Today
    the user is asked twice for tightly-coupled fields:
    ```
    Provider name [openrouter]:
    LLM provider URL [https://openrouter.ai/api/v1]:
    ```
    The URL default is hardcoded for OpenRouter; if the user
    types `anthropic` as the provider name, the URL default
    stays wrong and the user has to overwrite it manually.

    Introduce a small lookup map at the top of `ask_provider`:
    ```bash
    declare -A KNOWN_PROVIDERS=(
        [openrouter]="https://openrouter.ai/api/v1"
        [anthropic]="https://api.anthropic.com/v1"
        [ollama]="http://localhost:11434/v1"
    )
    ```
    Behavior:
    - Ask `Provider [openrouter]:` only.
    - If the answer is in `KNOWN_PROVIDERS`, use the mapped URL
      silently. The user does not see a second prompt.
    - If the answer is NOT in `KNOWN_PROVIDERS`, ask
      `Base URL [https://...]:` as a follow-up prompt.

    The single-provider case (the 99% path: `openrouter`)
    drops from two prompts to one. Custom providers are
    unaffected. The TOML `[providers.<name>]` key
    naming convention is preserved unchanged.

**Test plan.**

Extend `core/tests/test_install_sh.py` with one test per
sub-change, keeping per-item granularity:

- `test_rebuild_prompt_after_write` — drive `install.sh`,
  assert the `Restart with new config?` prompt is emitted
  strictly after `Write this config?` has accepted.
- `test_indent_consistency_all_prompts_two_spaces` — static
  scan of `install.sh`: every interactive `read -p ...` /
  `safe_read -p ...` line emits a prompt starting with two
  spaces.
- `test_external_url_wording_includes_callbacks` — assert
  the new wording mentions "webhook" and "MCP".
- `test_network_warning_includes_revert_hint` — assert the
  warning mentions `host = "127.0.0.1"`.
- `test_persona_post_prompt_hint_emitted` — drive through
  the persona prompt, assert the next line mentions
  `bot_persona` and `kiso restart`.
- `test_no_decorative_divider_in_install_sh` — `grep -c '─'`
  on `install.sh` returns 0.
- `test_final_prompt_accepts_edit` — drive the final prompt
  with input `edit`, assert the loop restarts at the first
  prompt with previous answers as defaults; with `n`, exit
  without writing; with `Y`, file written.
- `test_persona_prompted_immediately_after_bot_name` —
  assert the `Bot persona` prompt appears in the call sequence
  before `Identifier`, `User`, and `Provider`.
- `test_no_preflight_api_key_announcement` — grep
  `install.sh` for the literal `will ask for API key` and
  assert it is absent.
- `test_api_key_prompted_after_provider_block` — drive the
  flow, assert `OPENROUTER_API_KEY:` (or equivalent)
  appears between the provider URL prompt and the models
  block, NOT after the config preview.
- `test_api_key_never_echoed` — drive the flow with input
  `secret-test-key-1234`, assert the literal
  `secret-test-key-1234` never appears in captured
  stdout/stderr.
- `test_api_key_absent_from_config_preview` — drive the
  flow with a non-empty key, assert the config preview
  block does not contain the key value.
- `test_known_provider_skips_url_prompt` — answer
  `openrouter` to `Provider:`, assert the URL is not
  re-prompted and the resulting `[providers.openrouter]`
  block in the preview has the mapped URL.
- `test_unknown_provider_asks_url` — answer `my-custom`
  to `Provider:`, assert the `Base URL` prompt is emitted
  next.

**Files.**

- `core/install.sh` (~120 lines touched)
- `core/tests/test_install_sh.py` (14 new test cases)

**Out of scope (rejected by user).**

### M1570 — `install.sh` unified section template

**Problem.** Today five different visual patterns introduce a
block of prompts in `install.sh`:

```
1. Bot name                              ← header maiuscolo
     What do you want to call your bot?       + descr + indent
     Bot name [Kiso]:

2. Instance identifier (used for ...):   ← descr inline,
     Identifier [kiso]:                       no header

3. Provider name [openrouter]:           ← prompt nudo

4. Models — press Enter to keep default: ← em-dash header
     briefer       selects ...
                  [deepseek/...]:

5. Network access:                       ← colon header +
     1) Local only — ...                     numbered options
     2) Public — ...
     Choice [2]:
```

Five styles means the eye does not learn a pattern. Every
section reads as "start over". This is independent of M1569
(which addresses individual prompt content); M1570 is purely
the visual frame around them.

**Approach.** Replace the five styles with one fixed template:

```
## <Section name>
  <one-line description, optional>

  <field> [<default>]:
  <field> [<default>]:
```

Six sections in this exact order:

- `## Bot identity` → `bot_name`, `bot_persona`
  (description: `How Kiso presents itself in conversation.`)
- `## Instance` → `instance_id`, `user`
  (description: `Container name and data directory on this host.`)
- `## Provider` → `provider`, `base_url` (conditional, M1569
  sub-change 10), `OPENROUTER_API_KEY` (M1569 sub-change 9)
  (description: `Where Kiso routes LLM calls.`)
- `## Models` → 11 role prompts, one per line
  (description: `Press Enter to keep the default. See
  docs/llm-roles.md for context.`)
- `## Resources` → `ram_gb`, `cpus`, `disk_gb`, `max_pids`
  (description: `Container limits.`)
- `## Network` → `mode`, `external_url`
  (description: `How the API is exposed.`)

**Network options change.** Today the Network block uses
numbered choices (`1) Local only ... 2) Public ...
Choice [2]:`). This is the only prompt in the installer that
uses numbered options; replace with a word prompt for
consistency:

### M1571 — `install.sh` inline change indicator (re-install only)

**Problem.** When re-installing on an existing instance,
`install.sh` loads the current `config.toml` values and
re-prompts every field with those values as defaults. The
user accepts most with Enter and changes a few. Today, after
27 prompts, the user sees the full ~70-line config preview
and has to compare from memory to figure out what they
actually changed.

A diff helps — but accumulating changes into an end-of-flow
block is more machinery than the problem deserves, and
forces the user to look back when each change has just been
confirmed seconds earlier on screen. An inline indicator
under each prompt where the value differs from the loaded
"current" gives the same information at the point where it
is most relevant, with much less code.

**Approach.** Single inline marker emitted immediately after
any prompt where the entered value differs from the loaded
"current" value (the value that was already in the existing
`config.toml` when this re-install started).

```
bot_name [Kiso]: Bobi
  ✓ changed (was: "Kiso")

bot_persona [a friendly...]: 
                                    ← (Enter accepted; no marker)
```

The marker is two spaces in, leading checkmark, parenthetical
showing the previous value as a quoted literal.

For first install (no pre-existing `config.toml`), there is
no "current" reference value to compare against, so the
marker never appears (graceful degradation, zero noise).

**Implementation outline.** A small wrapper in `install.sh`:

```bash
ask_field_with_change_indicator() {
    local key=$1 default=$2 prompt_text=$3
    local current="${EXISTING[$key]:-}"   # set by load_existing_config
    safe_read -rp "$prompt_text [$default]: " val
    val=${val:-$default}
    if [[ -n "$current" && "$val" != "$current" ]]; then
        printf '  ✓ changed (was: "%s")\n' "$current"
        CHANGE_COUNT=$((CHANGE_COUNT + 1))
    fi
    eval "$key=\"\$val\""
}
```

The existing `load_existing_config` helper already populates
defaults; extend it to also populate an `EXISTING` associative
array keyed on field name. `CHANGE_COUNT` is a process-global
counter used by sub-change "post-write summary" below.

**Post-write summary line.** Two states:

- After a successful write with `CHANGE_COUNT > 0`:
  ```
  Wrote N changes to config.toml.
  ```
  (replaces the current bare `Configuration written.` line, if
  any).

- On re-install with `CHANGE_COUNT == 0`: short-circuit the
  final `Write this config? [Y/n]` prompt entirely. Print:
  ```
  No changes — config matches current state.
  ```
  …and exit the configuration block without re-writing the
  file. The container restart prompt (M1569 sub-change 1)
  still runs after this branch — the user might want to
  restart even without config changes (e.g. to pick up a
  new image).

On first install (no `EXISTING`), neither branch fires; the
existing post-write line is unchanged.

**Test plan.**

- `test_clean_install_no_change_indicator` — fresh install
  (no prior `config.toml`), assert no `✓ changed` line ever
  appears in output.
- `test_reinstall_unchanged_prompts_no_indicator` —
  re-install accepting all defaults with Enter, assert no
  `✓ changed` line and the final summary says
  `No changes — config matches current state.`
- `test_reinstall_changed_prompt_emits_indicator` —
  re-install changing `bot_name` from `Kiso` to `Bobi`,
  assert `  ✓ changed (was: "Kiso")` appears immediately
  after the bot_name prompt.
- `test_reinstall_no_changes_skips_write_prompt` —
  re-install accepting all defaults, assert
  `Write this config?` is NOT prompted.
- `test_reinstall_with_changes_post_summary` — re-install
  with 2 changes, assert post-write line is
  `Wrote 2 changes to config.toml.`
- `test_reinstall_no_changes_still_runs_restart_prompt` —
  re-install with no changes, assert the container restart
  prompt (M1569 sub-change 1) still fires.

**Files.**

- `core/install.sh` (~30 lines: helper function +
  `EXISTING` array population in `load_existing_config` +
  branching in the post-write block)
- `core/tests/test_install_sh.py` (6 new test cases)

**Dependency.** Soft dependency on M1569 sub-change 1
(restart prompt moved after write) — the
"no-changes-still-restarts" behavior in test 6 above only
makes sense if the restart prompt is decoupled from the
config write. If M1569 has not landed yet, this test
becomes `test_reinstall_no_changes_does_not_remove_container`
instead.

**Out of scope.**

- A formatted diff block at the end (`Changes: bot_name X →
  Y, models.planner A → B`). Inline replaces it.
- Showing changes for derived/computed values
  (e.g. `external_url` defaulting from public IP detection):
  the change indicator only fires when the user actively
  types a non-default value. If detection produces a
  different IP between runs, that is not "the user changed
  something" — silent.
- Coloring the marker (`✓` is plain ASCII, no `tput`
  escape codes).

### M1574 — Surface non-pytest errors in `run_tests.sh` recap

**Problem.** When a suite fails *before* pytest produces

```
✗ Docker tests             done (1s)
```

     `failed:`, `KeyError`, `OSError`, lines starting with
     `Error:` or `[ERROR]`.

   becomes:

## Phase 3 architectural framing (load-bearing for M1578-M1592)

**Decided 2026-04-27 with the user (refactored 2026-04-27):**

The post-v0.10 Kiso is **a broker of the user's installed MCPs**, not a
toolkit with built-in solutions. The classifier, planner, and chat_kb
pipeline must reflect this. The seven decisions below are load-bearing
for every Phase 3 milestone:

1. **Capability-missing policy = "ask-first":** when a request needs
   a capability not in the catalog, the planner emits a msg that
   asks the user *"do you have a specific URL in mind, or should I
   search for one?"*. The planner NEVER autonomously decides to
   search and NEVER invents URLs. If the user picks "search":
   - With a search MCP installed → planner uses it, presents result,
     asks for explicit approval before installing.
   - Without a search MCP → planner replies "I can't search without
     a search MCP installed; please give me a URL or install
     `kiso-run/search-mcp` (or another search MCP of your choice)".

2. **Default preset = installed by default, opt-out allowed:**
   `kiso init` runs `--preset default` which installs the 7 bundled
   MCPs + 1 skill. Users can choose `--preset minimal`/`--preset none`
   or remove individual servers later. The CANONICAL test scenario
   assumes the default preset is present; tests that simulate
   "browser missing" exercise an off-the-default-path edge case.

3. **chat / chat_kb / plan ontology (clean):**
   - **chat**: small talk + LLM general knowledge answers (no DB
     query, no shell, no MCP call). Stateless from KB perspective.
   - **chat_kb**: query the user's stored KB (facts/entities). Returns
     what's known. Does NOT search externally.
   - **investigate**: read live system state (read-only shell,
     read-only file ops). Stays unchanged from M1290.
   - **plan**: any action — including searching for external info,
     installing capabilities, file ops, executing commands.
   The trigger phrase "what do you know about X" alone does NOT
   force chat_kb — the topic must be a stored entity. General
   knowledge questions about public topics (Flask, Python, etc.) are
   `chat`. M1291's "0 facts → investigate" fallback is replaced:
   chat_kb with 0 matches → messenger admits "not in KB, want me to
   search?", treating it as a graceful degradation, NOT a re-route
   to shell investigation.

4. **Trust escalation via chat (no terminal prompt):** the planner
   shows trust info (source key + tier + risk factors) inside the
   `needs_install` msg. User approves with explicit positive text
   ("install" / "yes" / "vai" — not generic "ok"). Next turn has
   `Install Status` set; planner emits exec
   `kiso skill install --from-url <url> --yes` (the `--yes` flag
   bypasses the CLI-level prompt because chat-level approval has
   replaced it). Tier-1 installs (kiso-run/* / anthropic/*) skip
   the risk-factor display since trust is hardcoded; untrusted
   installs require the explicit per-source display.

5. **Multi-instance is an active feature**, not legacy. Multiple
   `kiso-<name>` containers on different ports, isolated data dirs
   in `~/.kiso/instances/<name>/`, registry in
   `~/.kiso/instances.json`. `install.sh` supports it; `kiso-host.sh`
   supports it; tests must too.

6. **Generalist-first, anti-overfitting:** every prompt rule, plan
   schema field, and validation logic must work for ANY MCP / skill
   the user installs, not just the ones in the default preset. The
   planner prompt MUST NOT contain hardcoded MCP names or example
   URLs from kiso-run/* (those are fixed in the trust-tier code,
   not in the role prompts). Tests must include parametrized
   capability cases over arbitrary names ("transcribe",
   "render-graph", "qux-9000") to verify the broker behavior is
   capability-agnostic. Discovered as a principle on 2026-04-27
   when the user noted: *"deve funzionare sempre"*.

7. **Smart-test principle:** every test that exercises a capability
   that lives in an MCP MUST guarantee the corresponding MCP is
   installed (real or mock) before running the flow. Tests that
   expect "Kiso does X via MCP Y" without setting up Y are testing
   the degraded fallback path (planner improvises with `exec`),
   NOT the positive functionality. This principle is operationalized
   by Block C infra (M1580 mock framework, M1581 `requires_mcp`
   marker, M1582 session isolation). Tests for "capability missing
   → planner asks for URL/search" are deliberately *negative
   tests* and are explicitly distinguished from positive
   functionality tests.

These seven decisions were enumerated with the user before opening
Phase 3. Every milestone below cites which decision(s) it
operationalizes.

### M1582 — Functional session isolation audit + fix

**Why.** The recent run showed cross-test contamination:
`TestF1BrowserInstall` runs first, leaves "browser wrapper
installed" in the KB. `TestF2WikipediaPython` runs next, classifier
sees the leftover fact, and the messenger response says
*"Il wrapper del browser Chromium è già installato"* instead of

exposing that they aren't (or the test fixtures don't reset

exposes a deployment-relevant invariant: session/KB isolation
must be guaranteed.

**Approach.**

1. **Investigate** where the contamination happens:
   - Does the functional test fixture share a single KISO_DIR
     across the test class? Across the whole module? Across
     the whole session?
   - Is the DB reset between tests? (Likely no — tests in a
     class share a `seeded_db` fixture from conftest.)

     question.)

   query test in session B (different `clean_session`),
   asserts B does NOT see A's fact.

- Lock test described above.
- After applying `clean_session` to F1/F2: F2 must NOT see

### M1591 — V4-Flash output schema robustness (full consumer audit)

**Why.** The recent live run showed
`TestCuratorLive.test_curator_v4_flash_verdict_mix` failing
because V4-Flash returned `learning_id` as the string `"1"`
instead of the integer `1`. Schema validation passed (loose
JSON typing), but the test (and likely downstream consumers)
assumed int. This is a CLASS of bug — V4-Flash's json_object
mode doesn't strictly enforce JSON schema types. **Finding 1
bug strongly implies more latent bugs in other consumers.**

This milestone is a comprehensive audit + defensive coercion
across **all V4-Flash output consumers**, not just curator.

**Approach.**

**Part A — Audit (the wider scope):**

1. **Enumerate all V4-Flash consumers** (roles that use
   `deepseek/deepseek-v4-flash` or similar V4 models per
   `MODEL_DEFAULTS`):
   - `briefer`: parses output for `modules`, `skills`,
     `mcp_methods` lists, `output_indices` (integer list),
     `relevant_tags`, `relevant_entities`.
   - `classifier`: parses category + language string. Bare
     output, less risk.
   - `planner`: parses `tasks` (list of objects with `type`,
     `detail`, `args`, `expect`, `group` integer, `server`,
     `method`), `needs_install` (list), `awaits_input`
     (bool — new from M1579a), `kb_answer` (bool),
     `extend_replan` (int).
   - `curator`: parses `evaluations` with `learning_id`
     (int), `verdict`, `entity_kind`, etc.
   - `messenger`: parses output (mostly free text, low risk).
   - `worker`: parses commands / shell args.

2. **For each consumer**, identify fields with strict type
   expectations and inject defensive coercion at parse time:
   - Integer fields (`learning_id`, `output_indices`,
     `group`, `extend_replan`) → `int(value)` with
     defensive str→int.
   - Boolean fields (`awaits_input`, `kb_answer`) →
     `bool(value)` if str ("true"/"false"/"1"/"0").
   - List fields with item types → coerce per-item.

3. **Coerce at parse time, NOT at schema validation level.**
   Rationale: schema validation is a contract verification;
   coercion is a defensive translation. Keep them separate.

4. **Tighten schema where possible**: where the field SHOULD
   be int, write `{"type": "integer"}` (not `oneOf:
   [int, string]`). The strict schema + lenient parser pattern
   catches drift early without breaking on it.

**Part B — Lock tests:**

5. `tests/test_v4_flash_schema_robustness.py` — parametrized
   over each consumer + each typed field:
   - Feed parser a synthetic response with the field as
     "wrong type" (string where int expected, int where
     bool, etc.).
   - Assert parser produces correctly-typed value, no
     downstream TypeError.
   - Assert deviation is silently coerced (or logged at
     DEBUG, not ERROR).

**Test plan.**

- Lock tests across ~15 typed-field × consumer
  combinations (curator learning_id, planner task index,
  briefer output_indices items, etc.).

### M1592 — `kiso doctor` broker-model invariants

**Why.** A forward-looking observability addition. The
broker model's correctness depends on the alignment between
the planner prompt and the planner's actual behavior. Drift
can happen (model upgrades, prompt edits, schema changes).
A `kiso doctor` invariant check surfaces drift in production
before users hit it as bugs.

**Approach.**

Extend `kiso doctor` (CLI subcommand) with a new section:
`Broker model invariants`:

- **Check 1:** of the last 100 plans in the DB, count those
  that are msg-only without `needs_install` / `kb_answer` /
  `awaits_input` set. Threshold: `> 5%` → WARN. `> 20%` →
  ERROR. (These are validation breaches that retry-passed
  via planner pivot to exec — a sign of prompt drift.)
- **Check 2:** of the last 100 plans, count those that
  used `exec` when the briefer reported empty MCP catalog
  AND the user message had a capability-style intent.
  Threshold: `> 0` → WARN.

  **Intent-matching strategy (decided 2026-04-27): heuristic
  with optional LLM fallback.**
  - **Primary**: a small static keyword list (~30 verbs)
    covering common capability intents: "transcribe",
    "search", "screenshot", "OCR", "summarize", "translate",
    "render", "compile", "lint", "deploy", "fetch", "scrape",
    "convert", "compress", "encode", "decode", etc.
    Generalist by design — no MCP-specific names. Match is
    word-boundary-aware (avoids false positives on
    "transcript" matching "transcribe").
  - **Fallback** (only if primary returns no match AND the
    plan ran exec with non-trivial output AND the user
    message is >5 words): invoke a single LLM call with
    a tight prompt asking *"did the user request a
    capability that an MCP would normally provide?"*. Cost:
    ~1 LLM call per `kiso doctor` invocation (rare —
    `kiso doctor` is run on demand, not continuously).
    The LLM fallback catches the long-tail capabilities
    not in the keyword list (decision 6 — generalist).
  - The keyword list lives in `kiso/api/doctor.py` (or
    extracted to a `_capability_keywords.py` module). New
    additions are PRs, no auto-discovery — keeps the heuristic
    auditable.
- **Check 3:** count `awaits_input=true` plans
  followed by `exec` plans within the same session — should
  be 0 (the user's reply should reclassify, not silently
  proceed). Threshold: `> 0` → WARN.

Output format follows existing `kiso doctor` conventions
(green/yellow/red sections per check).

**Test plan.**

- Unit tests for each check function:
  - Synthetic DB rows that trigger each threshold.
  - Verify warn/error level emitted as expected.
- Integration:
  - Real `kiso doctor` invocation against a test DB.

**Files.**
- `kiso/api/doctor.py` (or wherever doctor logic lives) —
  new section (~80 lines)
- `kiso/roles/doctor.md` (if applicable) — documentation
- `tests/test_doctor_broker_invariants.py` — new (~120 lines)
- `docs/cli.md` — update doctor doc

**Out of scope.**
- Telemetry to a remote endpoint — purely local check.
- Auto-remediation — doctor reports, doesn't fix.

### M1602 — Trust persistence multi-turn live test (closes M1588 case 4)

**Why.** M1588 covered the canonical untrusted-install
approve/reject toggle; case 4 (trust persistence after chat
approval) was deferred. The design contract — *first install
prompts with risk factors; second install of the same source
shows tier=custom, no risk-factor re-prompt* — is implicit in
`planner.md:82` but no test verifies the persistence end-to-end.
Without coverage, a regression to "always-prompt" is silent.

**Approach.** Multi-turn live test in
`tests/live/test_trust_chat.py::TestFlowGTrustPersistence`:
- Turn 1: user asks to install
  `random-org/cool-mcp from <url>`. Verify needs_install msg
  shows `tier=untrusted` + risk factors.
- Turn 2: user replies "yes install it". Verify exec install fires
  AND `~/.kiso/trust/skills.json` (or wherever the user-trust list
  lives) records the source as `tier=custom-user-approved`.
- Turn 3: user asks to install the same source again. Verify the
  needs_install msg now shows `tier=custom` and does NOT re-list
  risk factors.

- [ ] Test: live — the 3 turns + the trust-store assertion
- [ ] Test: unit — extract the trust-record format into a small
      lock so the test_live test can read it without depending
      on the messenger output
- [ ] Run with `--llm-live --live-network`
- [ ] Commit & push

**Done when:** the multi-turn flow passes live AND the second
