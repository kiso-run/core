# Devplan v0.4 — Context Intelligence Layer (Briefer Architecture)

## Problem Statement

kiso's pipeline treats context as a resource-management problem (character budgets,
mechanical truncation) instead of an information problem (what does this specific
consumer need?). Every LLM call receives a static, monolithic context dump regardless
of the specific task. This causes:

1. **Context overload** — models receive irrelevant rules and data, start ignoring
   instructions (especially weaker models like deepseek-v3.2)
2. **Blind truncation** — `_truncate_output`, `_smart_truncate`,
   `_format_plan_outputs_for_msg` cut at N chars, destroying information
   asymmetrically (may keep boilerplate, lose actual answers)
3. **Same context for all consumers** — messenger, replanner, and exec translator
   all receive the same `plan_outputs` formatted the same way, even though they
   need different information from it

## Architecture: the Briefer

A new LLM role called **briefer** sits at each stage boundary. It reads the full
context pool and produces a focused briefing tailored to the specific consumer
and task.

**Why "briefer"**: follows the existing -er naming convention (planner, reviewer,
messenger, curator, summarizer, paraphraser). The metaphor is a military briefer
who prepares a focused pre-mission report for an operative — exactly what this
component does for each LLM role.

### Context flow (before → after)

**Before** (current):
```
All available context → character-count truncation → monolithic prompt → LLM
```

**After** (with briefer):
```
All available context → Briefer LLM (selects + compresses) → focused prompt → LLM
```

### System prompt structure (per role)

Each role prompt has two layers:

1. **Fixed core** (identity + output format + core rules) — always present, never
   changes. Defines WHO the role is and HOW it responds.
2. **Conditional modules** (situational rules) — injected based on briefer signals.
   Pre-written, curated rule blocks that the briefer selects.
3. **Dynamic context** (from briefer output) — selected skills, relevant facts,
   focused history, synthesized task outputs.

The planner already does keyword-based appendix injection (planner-plugin-install,
planner-kiso-commands, planner-user-mgmt). The briefer generalizes this: instead of
simple keyword matching, an LLM evaluates which modules and context are relevant.

### Briefer schema (output)

```json
{
  "modules": ["web", "data_flow"],
  "skills": ["browser: navigate, screenshot, text — browser automation"],
  "context": "User wants to visit gazzetta.it and get news. Browser is installed. No replan history. No relevant facts.",
  "output_indices": [2, 3]
}
```

- `modules`: which prompt modules to inject into the consumer's system prompt
- `skills`: relevant skill descriptions only (for planner)
- `context`: synthesized briefing replacing raw summary + facts + history
- `output_indices`: which plan_output entries to include (for messenger/worker)

### DB context extraction

DB queries stay unchanged (fast, <1ms). The full context pool is gathered:
- Session summary, known facts, pending items, recent messages (from DB)
- Installed skills, connectors, system env (from config/filesystem)
- Plan outputs, replan context (from in-memory pipeline state)

The briefer reads the full pool and selects. Optimization for high-volume sessions
(1000+ facts, 500+ messages) can come later with DB-level filtering.

### Per-role model selection

With the briefer reducing context for every role, each role receives focused input.
This enables smarter models for critical roles and faster/cheaper models for
repetitive roles.

Default model recommendations (based on cost/quality/speed analysis):

| Role | Model | t/s | AAII | Coding | Input $/M | Rationale |
|------|-------|-----|------|--------|-----------|-----------|
| briefer | qwen-3.5-flash | ~90 | ~40 | ~82 | ~0.20 | 1M context, fast, cheap — reads everything to select |
| classifier | step-3.5-flash | ~85 | ~38 | ~86 | 0.10 | Simple binary classification, speed matters |
| planner | glm-4.7 | ~130 | ~42 | ~85 | 0.38 | Critical role: huge upgrade from deepseek-v3.2 (AAII 32→42, coding 59→85) |
| worker | step-3.5-flash | ~85 | ~38 | ~86 | 0.10 | Best coding/cost ratio for command translation |
| reviewer | step-3.5-flash | ~85 | ~38 | ~86 | 0.10 | Fast evaluation, called per-task |
| messenger | kimi-k2.5 | ~100 | ~47 | ~85 | 0.45 | User-facing: highest AAII for language quality |
| curator | step-3.5-flash | ~85 | ~38 | ~86 | 0.10 | Simple classification |
| summarizer | qwen-3.5-flash | ~90 | ~40 | ~82 | ~0.20 | Large context for reading many messages |
| paraphraser | step-3.5-flash | ~85 | ~38 | ~86 | 0.10 | Simple rewriting, speed matters |
| searcher | perplexity/sonar | — | — | — | — | Specialized, unchanged |

Estimated cost per request: ~$0.008 (vs ~$0.004 current). Delta: ~$0.004.

### M242 — Briefer role: prompt, schema, LLM function

**Problem:** No mechanism exists to intelligently select context for downstream LLM
calls. All context is dumped mechanically.

**Files:** `kiso/roles/briefer.md` (new), `kiso/brain.py`, `kiso/config.py`

**Change:**

1. Create `kiso/roles/briefer.md` — system prompt for the briefer:
   - Input: consumer role name, consumer task description, full context pool
   - Output: structured JSON with `modules`, `skills`, `context`, `output_indices`
   - Rules: select only what's relevant, compress don't truncate, preserve specific
     values (names, paths, URLs, numbers), never invent data

2. Add to `brain.py`:
   - `BRIEFER_SCHEMA` — JSON schema for structured output
   - `build_briefer_messages(consumer_role, task_description, context_pool)` —
     assembles the briefer prompt from the context pool dict
   - `run_briefer(config, consumer_role, task_description, context_pool, session)` —
     calls the LLM, validates output, returns the briefing dict
   - `BrieferError` exception class

3. Add `"briefer"` to `MODEL_DEFAULTS` in `config.py` (default: same as current
   default, will be updated in M251)

**Test:** Unit tests for `build_briefer_messages` (correct prompt assembly),
`run_briefer` (valid schema output, handles LLM errors gracefully).

### M243 — Planner prompt modularization

**Problem:** `planner.md` is a monolithic 61-line prompt. All rules are always
present regardless of whether they're relevant.

**Files:** `kiso/roles/planner.md`, `kiso/brain.py`

**Change:**

1. Split `planner.md` into clearly marked sections with module markers:
   ```
   <!-- MODULE: core -->
   [identity, output format, task types, core rules — always present]

   <!-- MODULE: web -->
   [web interaction rules: navigate vs search, browser skill, download]

   <!-- MODULE: replan -->
   [anti-loop, strategy diversification, confirmed facts, retry hints]

   <!-- MODULE: scripting -->
   [heredoc rules, one-liner block, script file writing]

   <!-- MODULE: skill_recovery -->
   [broken skill reinstall procedure, CRITICAL deps.sh rule]

   <!-- MODULE: data_flow -->
   [file-based data flow, truncation awareness, large output handling]
   ```

2. Add `_load_modular_prompt(role, modules)` to `brain.py` that:
   - Reads the full prompt file
   - Parses module markers
   - Returns core + selected modules concatenated
   - Falls back to full prompt if no markers found (backward compat)

3. The existing keyword-based appendix injection (planner-plugin-install, etc.)
   remains unchanged — those are ADDITIONAL appendices, not modules within the
   main prompt.

**Test:** `_load_modular_prompt("planner", ["core", "web"])` returns only core +
web sections. `_load_modular_prompt("planner", ["core"])` returns minimal prompt.
Full module list returns same content as current monolithic prompt.

### M244 — Briefer integration for planner

**Problem:** `build_planner_messages` dumps all context (summary, facts, recent
messages, ALL skills, ALL connectors, system env) into one prompt regardless of
the user's request.

**Files:** `kiso/brain.py` — `build_planner_messages()`

**Change:**

1. Before building the planner prompt, call `run_briefer()` with:
   - `consumer_role = "planner"`
   - `task_description = new_message` (the user's request)
   - `context_pool = {summary, facts, recent, skills, connectors, sys_env, pending,
     paraphrased_context, replan_context (if any)}`

2. Use the briefer's output to:
   - Select prompt modules via `_load_modular_prompt("planner", briefing["modules"])`
   - Include only `briefing["skills"]` in the Skills section (not all installed)
   - Replace raw summary + facts + recent with `briefing["context"]` (synthesized)
   - Keep: system env (small, semi-static), new message (always needed), appendices
     (already conditional)

3. Add a config setting `briefer_enabled` (default: `true`) to allow disabling
   the briefer for debugging or cost-sensitive deployments. When disabled, fall
   back to current behavior.

4. The briefer call runs with a short timeout (15-30s) and on failure falls back
   to current behavior (dump everything). The briefer is an optimization, not a
   gate.

**Test:** Integration tests comparing planner prompt with/without briefer. Verify
briefer-produced prompt is smaller. Verify fallback on briefer failure. Verify
web-related request gets web module, non-web request doesn't.

### M248 — Fact tagging: schema, storage, and retrieval

**Problem:** Facts have only a coarse `category` field (project/user/tool/general)
and FTS5 keyword search on `content`. This means:
- "naviga su un sito" won't match a fact about "Playwright browser automation"
  because there's no word overlap
- No way to ask "all facts about the browser" unless the word "browser" appears
  in the content text
- As facts accumulate (100+), imprecise retrieval wastes context budget

**Files:** `kiso/store.py`, `kiso/brain.py` (curator schema + briefer integration)

**Change:**

1. Add a `fact_tags` junction table:
   ```sql
   CREATE TABLE IF NOT EXISTS fact_tags (
       fact_id INTEGER REFERENCES facts(id) ON DELETE CASCADE,
       tag     TEXT NOT NULL,
       PRIMARY KEY (fact_id, tag)
   );
   CREATE INDEX IF NOT EXISTS idx_fact_tags_tag ON fact_tags(tag);
   ```

2. Add store functions:
   - `save_fact_tags(db, fact_id, tags: list[str])` — insert tags for a fact
   - `get_all_tags(db) -> list[str]` — returns `SELECT DISTINCT tag FROM fact_tags`
   - `search_facts_by_tags(db, tags: list[str], session, is_admin) -> list[dict]`
     — returns facts that have ANY of the given tags, ranked by tag overlap count
   - Update `save_fact()` and `save_facts_batch()` to accept optional `tags` param

3. Update `save_fact` to also insert into `fact_tags` when tags are provided.

4. Schema migration: add `fact_tags` table in `_SCHEMA`. Existing facts get no
   tags (backward compatible — tagless facts are still found by FTS5).

**Test:** CRUD for fact_tags. `search_facts_by_tags` returns correct facts.
`get_all_tags` returns distinct tags. Facts without tags still work via FTS5.

### M250 — Briefer uses tags for fact retrieval

**Problem:** The briefer (M244) currently receives all facts from FTS5 search.

### Phase 5: Deep Context Filtering — The Briefer as True Funnel

---

#### Problem Analysis

The briefer (Phase 1) was wired into the planner, but it controls only a fraction
of what the planner receives. Measured on a real request ("vai su guidance e dimmi
di che si occupa l'azienda"):

- **Planner system prompt**: 10832 chars / ~2708 tokens. The `core` module alone
  is 44 lines (~2500 chars) of mixed rules for ALL use cases. For a simple web
  search, the planner doesn't need skills efficiency rules, scripting rules, replan
  strategy, kiso-native-first, etc.
- **Keyword-triggered appendixes** (`planner-kiso-commands`, `planner-user-mgmt`,
  `planner-plugin-install`): injected via dumb keyword matching, bypassing the
  briefer entirely. `"install" in msg_lower` fires even for "install guidance on
  my phone".
- **Unconditional includes**: `sys_env` goes to the planner always, even for
  "tell me a joke". `capability_gap` block is injected regardless of briefer.
- **Other roles get zero filtering**: worker, reviewer, messenger, curator,
  summarizer — the briefer does nothing for them (or minimal output_indices for
  messenger).

**Core insight**: the briefer model (gemini-flash-lite) has 1M context and costs
$0.10/M input. It's designed to ingest everything and output a focused brief. The
downstream models (glm-4.7, step-3.5-flash) have smaller contexts, are slower, and
more expensive. The briefer should be doing HEAVY filtering so every downstream
model gets the minimum it needs.

**Target**: reduce planner input from ~2700 tokens to ~800-1200 for typical requests
(~60% reduction).

### M256 — Decompose planner core into granular modules

**Problem:** The `core` module in `planner.md` is 44 lines of mixed rules that
always go to the planner. It contains skills efficiency rules, kiso-native-first
policy, replan caveats, public files rules, and more — most irrelevant for a
typical request.

**Files:** `kiso/roles/planner.md`, `kiso/brain.py` (`BRIEFER_MODULES`)

**Change:**

1. Split the current `core` module into:

   | Module | Content | When needed |
   |---|---|---|
   | `core` | Identity + JSON format + task type definitions + last-task-must-be-msg/replan + empty-tasks rule (~15 lines) | **Always** |
   | `planning_rules` | expect rules, detail-is-natural-language, recent-messages-are-background, multi-step-intermediate-msg, public-files, no-fabrication | Most requests (skip for trivial lookups) |

### Phase 6: Runtime Fixes — Model Compat, Messenger Identity, Briefer Precision

---

#### Problem Analysis

Runtime testing of a real request ("vai su guidance.studio e dimmi cosa dice la
pagina") revealed **7 problems**, from fatal (infinite replan loop) to cosmetic
(duplicate inflight indicator). Root causes:

1. **P1 — Reviewer model incompatibility**: `stepfun/step-3.5-flash` rejects
   `response_format: {"type": "json_schema"}` with 400. Every review fails →
   forced replan → loop until exhaustion. Tested via OpenRouter: StepFun only
   accepts `json_object`, not `json_schema`. Additionally, step-3.5-flash is a
   reasoning model that spends ~900 tokens of chain-of-thought for simple tasks
   (6x more expensive, 4x slower than gemini-2.5-flash-lite).

2. **P2 — Messenger identity crisis**: The messenger (qwen3.5-flash) says
   "non posso navigare su siti web" — it thinks it IS the system instead of
   describing what the system will do. The prompt never clarifies this distinction.
   The intent msg detail says "what you're about to do" which the LLM reads as
   "what I (the LLM) can do".

3. **P3 — Briefer oversharing with messenger**: Briefer returns modules and
   skills for the messenger role, but messenger has no modular prompt and ignores
   both fields. Wastes briefer output tokens and may confuse the briefer's own
   context filtering.

4. **P4 — Web module implies browser without checking installation**: Briefer
   selects `web` module → planner reads "use browser skill to visit URLs" → tries
   `skill: browser` → validation rejects (not installed) → 31s retry. The web
   module should not imply browser availability unless it's installed.

5. **P5 — Briefer propagates messenger hallucination**: The messenger's wrong
   output ("I can't navigate websites") enters replan context as a task output.
   The briefer copies it into its synthesized context, spreading the false claim
   to the next planner cycle. This is resolved by M264 (fix messenger identity)
   — once the messenger stops hallucinating, the false claim won't enter the
   replan context.

6. **P6 — Duplicate inflight indicator on validation retry**: When the planner
   validation retry triggers a second LLM call, the CLI shows two
   `⏳ planner → glm-4.7 (waiting...)` lines.

**Measured model performance** (OpenRouter, 2026-03-08):

| Model | json_schema | json_object | Review latency | Review cost | Worker latency |
|-------|-------------|-------------|----------------|-------------|----------------|
| gemini-2.5-flash-lite | ✅ | ✅ | **1.8s** | **$0.00004** | **1.3s** |
| qwen3.5-flash | ✅ | ✅ | 10.3s | $0.0005 | 1.5s |
| step-3.5-flash | ❌ (400) | ✅ (inconsistent) | 8.2s | $0.0003 | 1.9s |

### M276 — Classifier: add conversation context for follow-up detection

**Problem:** The classifier receives only the raw message, with no conversation

**Files:** `kiso/roles/classifier.md`, `kiso/brain.py` (classifier message builder)

**Change:**

   If a "## Recent Context" section is provided, use it to disambiguate.
   A short follow-up message that references a previous action

   implies further action on a previous plan. It is "chat" only if the user
   is asking about the *output* already delivered.

2. In `brain.py`, pass the last 1-2 messages and last plan goal as context
   when building the classifier prompt. Keep it minimal (< 200 chars) to
   avoid token waste.

3. Add rule: "If the message is fewer than 5 words and a recent plan exists,
   default to 'plan' — short follow-ups are almost always action requests."

### M277 — Messenger: enforce third-person system voice

**Problem:** The messenger prompt says "You speak directly to the user" and
"you are announcing system actions" but doesn't explicitly ban first-person "I"
for system actions. Models sometimes write "I searched the web" instead of
"Here's what was found". The prompt also lacks explicit guidance on
distinguishing between the assistant's voice (warm, direct) and system action
reporting (third-person, factual).

**Files:** `kiso/roles/messenger.md`

**Change:**

1. Add explicit voice rule:
   ```
   Voice rules:
   - For conversational responses (greetings, opinions, explanations): speak
     naturally as {bot_name} using first person ("I think...", "Sure!").
   - For reporting system actions and results: use passive or third-person
     ("The search found...", "3 files were created", "Installation completed").
     NEVER say "I ran", "I installed", "I searched" — you did not perform
     these actions, the system did.
   - For upcoming actions: "The system will now install..." or simply describe
     what happens next ("Next step: installing the browser skill").
   ```

2. Remove the ambiguous "You speak directly to the user" line and replace
   with the clearer voice rules above.

**Test:** Verify the prompt contains "NEVER say I ran". Verify existing
messenger tests still pass.

### M280 — Reviewer: handle truncated output and partial success

**Problem:** The reviewer prompt doesn't address two common scenarios:
(1) output is truncated (`[truncated]` or `[Full output saved to...]`) — the
reviewer may judge based on incomplete data; (2) partial success where the
command succeeded but output is incomplete or has warnings that matter.

**Files:** `kiso/roles/reviewer.md`

**Change:**

1. Add truncation awareness rule:
   ```
   - Truncated output: if the output contains "[truncated]" or "[Full output
     saved to ...]", the visible output is incomplete. If the visible portion
     satisfies `expect`, mark "ok". If you cannot determine success from the
     visible portion, mark "ok" with learn noting the truncation — do NOT
     replan just because output was truncated. The data may be complete in
     the saved file.
   ```

2. Add partial success boundary rule:
   ```
   - Partial success: if the command exits 0 and produces useful output but
     also shows warnings or incomplete sections, mark "ok" if `expect` is
     met. Include warnings in `summary` so downstream stages are aware.
     Only replan if the missing part is essential to `expect`.
   ```

**Test:** Verify the prompt contains truncation and partial success rules.

### M281 — Briefer: clarify fast-path boundaries and conflict handling

**Problem:** The briefer prompt says "For simple requests: modules=[], skills=[],
context=''" but doesn't define what counts as "simple." Models may over-filter
or under-filter. Additionally, there's no guidance on handling contradictory
facts in the context pool.

**Files:** `kiso/roles/briefer.md`

**Change:**

1. Replace the vague "simple requests" line with explicit examples:
   ```
   - Fast-path (modules=[], skills=[], context=""): greetings, thanks, small
     talk, simple knowledge questions, "what is X?", opinions, jokes. These
     need ONLY the core role rules.
   - Needs context (but few/no modules): information retrieval ("search for X"),
     single-skill tasks ("take a screenshot"), env lookups. Include the
     relevant skill and/or context, but skip modules unless multi-step.
   - Needs modules: multi-step plans, replan scenarios, error recovery, skill
     installation. Add only the specific module(s) required.
   ```

2. Add conflict handling:
   ```
   - Conflicting facts: if two facts contradict each other, include the more
     recent one (later in the context pool). Flag the conflict in the context
     string: "Note: conflicting facts about X — using most recent."
   ```

**Test:** Verify the prompt contains fast-path examples and conflict handling.

### M282 — Curator: tag reuse enforcement and conflict resolution

**Problem:** The curator prompt tells the curator to "reuse existing tags when
possible" but doesn't enforce it strongly. In practice, curators create
near-duplicate tags (e.g., "web-browser" vs "browser") or promote facts that
contradict existing ones without checking.

**Files:** `kiso/roles/curator.md`

**Change:**

1. Strengthen tag reuse:
   ```
   - Tag reuse (CRITICAL): before creating a new tag, check if an existing
     tag covers the same concept. Prefer broad tags over narrow ones. NEVER
     create a tag that is a synonym of an existing tag (e.g., "web-browser"
     when "browser" exists, "dependencies" when "deps" exists). If unsure,
     use the existing tag.
   ```

2. Add conflict resolution:
   ```
   - Contradicting facts: if a new learning contradicts an existing fact
     (visible in the Existing Facts section), promote the new learning
     with a note: "Supersedes: [old fact summary]". The fact consolidator
     will handle the merge. Do NOT discard a contradicting learning — newer
     observations take precedence.
   ```

**Test:** Verify the prompt contains tag reuse and contradiction rules.
