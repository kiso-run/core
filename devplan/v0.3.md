# Devplan v0.3 — Logic Flaws & Flow Fixes

Goal: fix the systemic logic flaws that prevent kiso from completing non-trivial
multi-step tasks (web browsing, plugin install chains, data processing, etc.).

Derived from a full trace of every flow and prompt against realistic scenarios
like: *"vai su guidance.studio, dimmi cosa fa l'azienda e mandami uno screenshot"*
starting from a factory-reset state (no skills, no connectors).

---

## Problem Inventory

### P1 — `python -c` blocked but recommended by planner prompt
The deny list (`security.py`) blocks `python3 -c`, `node -e`, `perl -e`, `ruby -e`.
But `planner.md` line 40 explicitly tells the planner to use `python3 -c` as a
strategy diversification for HTML/JSON parsing. The worker translates to a
`python3 -c '...'` command, it gets denied, the reviewer replans, and the planner
tries the same blocked approach again — burning 2+ replan cycles on an impossible
strategy.

### P2 — No scripting escape hatch documented
Since one-liners are blocked, the only way to run Python/Node logic is to write a
script file and execute it. But no prompt or rule tells the planner this pattern.

### P3 — Plugin-install chain costs too many replans
The mandatory `planner-plugin-install.md` sequence is:
1. `curl registry` → replan (1)
2. `curl kiso.toml` → replan (2) to read env requirements
3. If env vars needed: `msg` user → replan (3)
4. `kiso env set` + `kiso skill install` → replan (4) to use the skill
5. Actually use the skill + `msg`

With `max_replan_depth=3`, steps 4-5 are unreachable. Even the happy path
(no env vars needed) burns 2 replans just for discovery, leaving only 1 for
the actual task + any retries.

### P4 — Plugin-install appendix injection too narrow
The keyword gate (`_plugin_kw = {"install", "installa", "plugin", "add"}`) only
fires on explicit install requests. The M129 fallback (inject when zero skills)
helps on factory reset, but once ANY skill is installed, the appendix disappears
for messages like "fammi uno screenshot" — the planner has no guidance on how to
install the missing `browser` skill.

### P5 — No web-interaction guidance in planner prompt
The planner doesn't know the difference between:
- "understand what a website is about" → use `search` task (Sonar visits the URL)
- "download raw content from URL" → use `exec curl -o file`
- "interact with a website / take screenshot" → needs `browser` skill
Without this, the planner defaults to `exec curl` for everything web-related,
gets raw HTML it can't parse (see P1), and wastes replan cycles.

### P6 — `search` task underspecified in planner prompt
The `search` type is described as "web search" but Perplexity Sonar can also
visit a specific URL and summarize its content. The planner doesn't know this,
so it never uses `search` with a specific URL target like
"visit guidance.studio and describe what the company does".

### P7 — No proactive registry check rule
When the planner encounters a task requiring capabilities beyond basic shell
(screenshot, browser automation, code editing), there's no rule saying "check
the registry first". The planner relies on the plugin-install appendix (P4) or
the generic "Kiso-native first" rule, which is too vague to trigger reliably.

### P8 — Information loss through multi-hop truncation
Data flows through: raw output → `_LARGE_OUTPUT_THRESHOLD` (4096 chars, saved to
file with 500-char preview) → reviewer summary (500 chars) → replan context
(prefers summary, 2000-char limit per task). By the 3rd replan, the planner has
lost most details from early tasks. For web content extraction, the HTML is
essentially gone by the time the planner needs to decide what to do with it.

### P9 — `max_worker_retries=1` too low for trivial fixes
When the reviewer says "replan" with a retry_hint (wrong flag, missing `-L`,
wrong path), the exec task gets ONE retry before escalating to a full replan.
Many commands need 2 attempts for minor adjustments (curl redirect, jq syntax,
grep pattern).

### P10 — Circular replan detection too simple
Detection is >60% word overlap between last 2 failure reasons. But the planner
can repeat the same failing *strategy* with completely different wording:
- "curl returned empty" → "wget received no content"
These have low word overlap but are the same underlying problem.

### P11 — `PROBE_BINARIES` missing common tools
The list doesn't include: `chromium`, `chromium-browser`, `playwright`, `lynx`,
`w3m`, `html2text`, `rg` (ripgrep), `bat`, `less`, `tee`, `wc`, `sort`, `uniq`,
`cut`, `tr`, `head`, `tail`, `diff`, `patch`, `file`, `stat`, `du`.
The planner can't know these exist even if they're installed.

### M149 — Remove `python -c` contradiction from planner prompt
**Problem:** P1
**Files:** `kiso/roles/planner.md`
**Change:** Remove `python3 -c` from the strategy diversification examples.
Replace with: "write a Python/Node script to a file then execute it".
**Test:** Verify planner.md no longer mentions `python -c` or `node -e`.
Read-only check.

---

### M150 — Add scripting-via-file guidance to planner prompt
**Problem:** P2
**Files:** `kiso/roles/planner.md`
**Change:** Add a new rule in the Skills efficiency section:
> Scripting: `python -c` and `node -e` one-liners are blocked by security policy.
> For data processing (HTML parsing, JSON manipulation, CSV analysis), use two
> exec tasks: the first writes a script file (e.g. `parse.py`), the second
> executes it (`python3 parse.py`). Keep scripts short and focused.
**Test:** Read planner.md, verify the guidance is present and no blocked commands
are recommended.

---

### M151 — Add web-interaction patterns to planner prompt
**Problem:** P5, P6
**Files:** `kiso/roles/planner.md`
**Change:** Add a new section "Web interaction patterns" to planner.md:
> - **Understand a website's content:** use a `search` task targeting the specific
>   URL (e.g. detail="visit https://example.com and describe what the company does").
>   The search engine visits the page and returns a synthesis.
> - **Download raw files from a URL:** use `exec` with curl/wget to save to a file.
> - **Browser automation / screenshots / form interaction:** requires the `browser`
>   skill. If not installed, check the registry and install it first.
> - Never use `exec curl` to understand page content — raw HTML is not useful
>   without parsing. Use `search` or the `browser` skill instead.
**Test:** Read planner.md, verify the web interaction section exists with all
4 bullet points.

### M152 — Collapse plugin-install chain: allow install without intermediate replan
**Problem:** P3
**Files:** `kiso/roles/planner-plugin-install.md`
**Change:** Rewrite the mandatory sequence to be more efficient:
1. User says "la skill X" or "il connector X" → step 3.
2. Ambiguous request: exec `curl <registry_url>` to discover. Then:
   - If registry result is clear (single match, no env vars based on the name),
     proceed to step 3 in the same plan with a replan at the end.
   - If ambiguous (multiple matches), replan to let the user clarify.
3. exec `curl kiso.toml` to read env requirements.
4. If ALL required env vars are already set OR no env vars needed:
   proceed directly to `kiso skill install` in the same plan. No replan needed.
5. If required env vars are MISSING: msg user asking for values → replan.
6. After user provides values: `kiso env set` + `kiso skill install` + `kiso
   connector run` if connector.
7. After install: replan to use the newly installed skill (planner needs fresh
   skill list).

Key change: steps 3+4+6 can be in ONE plan when env vars aren't needed.
The only mandatory replan points are: (a) after registry discovery when ambiguous,
(b) after asking user for env vars, (c) after install to get fresh skill list.
**Test:** Read planner-plugin-install.md, verify that the sequence allows
combining kiso.toml fetch + install in one plan when no env vars are missing.

### M153 — Widen plugin-install appendix injection
**Problem:** P4
**Files:** `kiso/brain.py` — `build_planner_messages`
**Change:** Add a capability-gap heuristic. After building the skill list, if
the message implies a capability not covered by installed skills, inject the
plugin-install appendix. Heuristic: maintain a small keyword→capability map:
```python
_CAPABILITY_KEYWORDS = {
    "screenshot": "browser",
    "browser": "browser",
    "navigate": "browser",
    "click": "browser",
    "form": "browser",
    "webpage": "browser",
    "website": "browser",  # when combined with action verbs
    "code": "aider",
    "refactor": "aider",
    "edit code": "aider",
}
```
If any keyword matches AND the mapped skill name is NOT in `installed_names`,
inject the plugin-install appendix.
Also keep the existing M129 fallback (zero skills → inject).
**Test:** `test_build_planner_messages` — when message contains "screenshot" and
browser skill is not installed, verify plugin-install appendix is in the system
prompt. When browser IS installed, verify it's NOT injected (unless other
triggers fire).

### M154 — Add proactive registry check rule to planner prompt
**Problem:** P7
**Files:** `kiso/roles/planner.md`

### M157 — Improve circular replan detection with strategy fingerprinting
**Problem:** P10
**Files:** `kiso/worker/loop.py` — `_run_planning_loop`
**Change:** In addition to word overlap, compare the task *types and detail
prefixes* of the last 2 plan attempts. If >50% of tasks have the same
type+first-30-chars-of-detail, flag as circular even if failure reasons differ.

Store a `strategy_fingerprint` (sorted set of `f"{type}:{detail[:30]}"`) in
each replan_history entry. Compare fingerprints with Jaccard similarity.
**Test:** Two plans with same task types/details but different failure wording
→ stuck_detected=True. Two plans with genuinely different strategies →
stuck_detected=False.

### M158 — Increase max_replan_depth default to 5
**Problem:** P3 (complementary to M152)
**Files:** `kiso/config.py`, `kiso/sysenv.py` (plan limits text)
**Change:** Change `"max_replan_depth": 3` → `"max_replan_depth": 5`.

- 1 replan for registry discovery
- 1 replan after skill install
- 3 replans for actual task execution/retries
**Test:** Verify default is 5, plan limits text updated.

### M160 — Improve replan context: carry forward full confirmed facts
**Problem:** P8
**Files:** `kiso/worker/utils.py` — `_build_replan_context`
**Change:** Currently `_extract_confirmed_facts` only looks at task outputs for
patterns. Add: also extract the reviewer `summary` field as a confirmed fact
when the review status is "ok" and summary is non-null. This means successful
task summaries automatically become confirmed facts in the replan context,
reducing information loss.
Cap at 15 facts (up from 10) and prioritize reviewer summaries over raw
output pattern matching.
**Test:** Mock completed tasks with reviewer_summary → verify summaries appear
in confirmed facts section. Verify cap is 15.

### M161 — Worker prompt: add file-reading guidance for large outputs
**Problem:** P8 (complementary)
**Files:** `kiso/roles/worker.md`
**Change:** The worker prompt already says to use `cat`/`grep` on saved files.
Strengthen: add a rule that when the task detail says "extract" or "parse" or
"read" and preceding task outputs reference a saved file, the command MUST
operate on that file (not re-fetch the data).
**Test:** Read worker.md, verify the rule is present.

### M162 — Planner prompt: explicit file-based data flow for web content
**Problem:** P8, P5
**Files:** `kiso/roles/planner.md`
**Change:** Strengthen the existing "File-based data flow" rule:
> When fetching web content with `exec curl`, ALWAYS save to a file
> (`curl -L -o page.html <url>`). Never let HTML/JSON output flow through
> stdout — it will be truncated and you lose the data. Subsequent tasks
> can then `grep`, `head`, or run a script on the saved file.
Move this rule higher in the prompt (currently buried at the end of Rules).
**Test:** Read planner.md, verify the rule is prominent and mentions curl -o.
