# Devplan v0.6 ‚Äî Plugin Taxonomy Redesign

## Problem Statement

The current naming convention is inconsistent with the broader AI ecosystem. "Skills" in
kiso map to what Claude Code, MCP, and most agent frameworks call "tools" ‚Äî executable
packages that perform actions. The term "skill" is overloaded and confusing for new users.

Additionally, there's no lightweight concept for prompt-only instructions that guide the
planner without requiring an executable package.

## New Taxonomy

- **plugin** ‚Äî generic umbrella term for any installable extension
  - **tool** ‚Äî executable package (browser, aider, search) ‚Äî what was called "skill"
  - **skill** ‚Äî lightweight `.md` file with planner instructions (new concept)
  - **connector** ‚Äî external integration (slack, telegram) ‚Äî unchanged

## Design Principles

1. **tool** aligns with Claude Code, MCP, and function-calling conventions
2. **skill** becomes a simpler, lower-friction concept ‚Äî just a markdown file
3. **connector** stays unchanged ‚Äî already well-defined
4. **plugin** provides the grouping term for CLI (`kiso plugin list`)

## Scope of Rename

The skill ‚Üí tool rename touches:

**Core modules:**
- `kiso/skills.py` ‚Üí `kiso/tools.py` (file rename + all functions/classes)
- `kiso/brain.py` ‚Äî `TASK_TYPE_SKILL‚ÜíTASK_TYPE_TOOL`, schema fields, validators, prompt builders
- `kiso/store.py` ‚Äî `tasks.skill` column ‚Üí `tasks.tool`, `TaskDict.skill‚Üítool`, `create_task` param
- `kiso/worker/skill.py` ‚Üí `kiso/worker/tool.py` ‚Äî `_skill_task‚Üí_tool_task`
- `kiso/worker/loop.py` ‚Äî all skill imports, installed_skills, user_skills, task dispatch
- `kiso/skill_repair.py` ‚Üí `kiso/tool_repair.py`
- `kiso/security.py` ‚Äî `PermissionResult.skills‚Üítools`, access control
- `kiso/config.py` ‚Äî `User.skills‚Üítools`
- `kiso/main.py` ‚Äî `user_skills‚Üíuser_tools`

**CLI:**
- `cli/skill.py` ‚Üí `cli/tool.py` ‚Äî `kiso skill‚Üíkiso tool`, `OFFICIAL_PREFIX="tool-"`
- `cli/__init__.py` ‚Äî parser registration, subcommands

**Registry + repos:**
- `registry.json` ‚Äî `"skills"‚Üí"tools"`, repo prefix `skill-‚Üítool-`
- External repos: `skill-browser‚Üítool-browser`, `skill-search‚Üítool-search`, etc. (user renames)

**Prompts:**
- `kiso/roles/planner.md` ‚Äî task type `skill‚Üítool`, all skill references
- `kiso/roles/briefer.md` ‚Äî `skills` field, module descriptions
- `kiso/roles/worker.md` ‚Äî skill venv, short names
- `kiso/roles/reviewer.md` ‚Äî skill references

**Manifest format:**
- `[kiso.skill]` ‚Üí `[kiso.tool]` in tool kiso.toml files
- `[kiso.skill.args]` ‚Üí `[kiso.tool.args]`
- Env vars: `KISO_SKILL_{NAME}_{KEY}` ‚Üí `KISO_TOOL_{NAME}_{KEY}`

**Directory paths:**
- `~/.kiso/skills/` ‚Üí `~/.kiso/tools/`

**DB migration:**
- `tasks.skill` column ‚Üí `tasks.tool` (ALTER TABLE RENAME COLUMN)

**Tests (12+ files):**
- `test_skills.py‚Üítest_tools.py`, `test_cli_skill.py‚Üítest_cli_tool.py`, etc.
- All mocks: `patch("kiso.worker.loop.discover_skills"‚Üí"discover_tools")`, etc.

### Phase 2: MD-based skills (new concept)

### M449 ‚Äî Skill loader: discover and parse .md skill files

**Files:** `kiso/skill_loader.py` (new)

**Change:**

1. New module `kiso/skill_loader.py`:
   - `discover_md_skills(skills_dir: Path | None = None) -> list[dict]`
   - Default dir: `~/.kiso/skills/` (distinct from `~/.kiso/tools/`)
   - Each `.md` file in the directory is a skill.
   - Parse frontmatter (YAML between `---` markers):
     ```yaml
     ---
     name: data-analyst
     summary: Guides planner for data analysis tasks
     ---
     ```
   - Body is the planner instruction text.
   - Return list of `{"name": str, "summary": str, "instructions": str, "path": str}`.
   - No trigger matching ‚Äî all discovered skills go to the briefer's context pool,
     consistent with how tools are handled. The briefer decides relevance.
   - TTL cache like `discover_tools()` to avoid repeated filesystem scans.
2. `build_planner_skill_list(skills: list[dict]) -> str`:
   - Format skill list for planner context, same pattern as `build_planner_tool_list()`.
   - Each entry: `- {name} ‚Äî {summary}` + instructions block.

**Test:** Unit tests with temp .md files.

### M450 ‚Äî Inject skills into briefer context pool

**Files:** `kiso/brain.py`, `kiso/worker/loop.py`

**Change:**

1. `_gather_planner_context()` calls `discover_md_skills()`.
2. All discovered skills go into the briefer's `context_pool` as a `"md_skills"`
   entry, same pattern as tools go into `context_pool` as `"skills"`/`"tools"`.
3. The briefer decides which MD skills are relevant and includes their instructions
   in the planner context. No pre-filtering ‚Äî consistent with tool handling.
4. Skills are additive ‚Äî they don't replace tool descriptions, they augment
   the planner's behavior for specific task types.
5. Add `"md_skills"` to `_BRIEFER_MODULE_DESCRIPTIONS` so the briefer knows
   what this context pool entry contains.

**Test:** Assert skill instructions appear in planner messages when briefer selects them.
