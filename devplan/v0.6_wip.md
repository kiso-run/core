# Devplan v0.6 ÔÇö Plugin Taxonomy Redesign

## Problem Statement

The current naming convention is inconsistent with the broader AI ecosystem. "Skills" in
kiso map to what Claude Code, MCP, and most agent frameworks call "tools" ÔÇö executable
packages that perform actions. The term "skill" is overloaded and confusing for new users.

Additionally, there's no lightweight concept for prompt-only instructions that guide the
planner without requiring an executable package.

## New Taxonomy

- **plugin** ÔÇö generic umbrella term for any installable extension
  - **tool** ÔÇö executable package (browser, aider, search) ÔÇö what was called "skill"
  - **skill** ÔÇö lightweight `.md` file with planner instructions (new concept)
  - **connector** ÔÇö external integration (slack, telegram) ÔÇö unchanged

## Design Principles

1. **tool** aligns with Claude Code, MCP, and function-calling conventions
2. **skill** becomes a simpler, lower-friction concept ÔÇö just a markdown file
3. **connector** stays unchanged ÔÇö already well-defined
4. **plugin** provides the grouping term for CLI (`kiso plugin list`)

## Scope of Rename

The skill Ôćĺ tool rename touches:

**Core modules:**
- `kiso/skills.py` Ôćĺ `kiso/tools.py` (file rename + all functions/classes)
- `kiso/brain.py` ÔÇö `TASK_TYPE_SKILLÔćĺTASK_TYPE_TOOL`, schema fields, validators, prompt builders
- `kiso/store.py` ÔÇö `tasks.skill` column Ôćĺ `tasks.tool`, `TaskDict.skillÔćĺtool`, `create_task` param
- `kiso/worker/skill.py` Ôćĺ `kiso/worker/tool.py` ÔÇö `_skill_taskÔćĺ_tool_task`
- `kiso/worker/loop.py` ÔÇö all skill imports, installed_skills, user_skills, task dispatch
- `kiso/skill_repair.py` Ôćĺ `kiso/tool_repair.py`
- `kiso/security.py` ÔÇö `PermissionResult.skillsÔćĺtools`, access control
- `kiso/config.py` ÔÇö `User.skillsÔćĺtools`
- `kiso/main.py` ÔÇö `user_skillsÔćĺuser_tools`

**CLI:**
- `cli/skill.py` Ôćĺ `cli/tool.py` ÔÇö `kiso skillÔćĺkiso tool`, `OFFICIAL_PREFIX="tool-"`
- `cli/__init__.py` ÔÇö parser registration, subcommands

**Registry + repos:**
- `registry.json` ÔÇö `"skills"Ôćĺ"tools"`, repo prefix `skill-Ôćĺtool-`
- External repos: `skill-browserÔćĺtool-browser`, `skill-searchÔćĺtool-search`, etc. (user renames)

**Prompts:**
- `kiso/roles/planner.md` ÔÇö task type `skillÔćĺtool`, all skill references
- `kiso/roles/briefer.md` ÔÇö `skills` field, module descriptions
- `kiso/roles/worker.md` ÔÇö skill venv, short names
- `kiso/roles/reviewer.md` ÔÇö skill references

**Manifest format:**
- `[kiso.skill]` Ôćĺ `[kiso.tool]` in tool kiso.toml files
- `[kiso.skill.args]` Ôćĺ `[kiso.tool.args]`
- Env vars: `KISO_SKILL_{NAME}_{KEY}` Ôćĺ `KISO_TOOL_{NAME}_{KEY}`

**Directory paths:**
- `~/.kiso/skills/` Ôćĺ `~/.kiso/tools/`

**DB migration:**
- `tasks.skill` column Ôćĺ `tasks.tool` (ALTER TABLE RENAME COLUMN)

**Tests (12+ files):**
- `test_skills.pyÔćĺtest_tools.py`, `test_cli_skill.pyÔćĺtest_cli_tool.py`, etc.
- All mocks: `patch("kiso.worker.loop.discover_skills"Ôćĺ"discover_tools")`, etc.

### Phase 2: MD-based skills (new concept)

### M449 ÔÇö Skill loader: discover and parse .md skill files

**Files:** `kiso/skill_loader.py` (new)

**Change:**

1. New module `kiso/skill_loader.py`:
   - `discover_md_skills(skills_dir: Path | None = None) -> list[dict]`
   - Default dir: `~/.kiso/skills/` (distinct from `~/.kiso/tools/`)
   - Each `.md` file in the directory is a skill.
   - Parse frontmatter (YAML between `---` markers):
     ```yaml
     ---
     name: data-analyst
     summary: Guides planner for data analysis tasks
     ---
     ```
   - Body is the planner instruction text.
   - Return list of `{"name": str, "summary": str, "instructions": str, "path": str}`.
   - No trigger matching ÔÇö all discovered skills go to the briefer's context pool,
     consistent with how tools are handled. The briefer decides relevance.
   - TTL cache like `discover_tools()` to avoid repeated filesystem scans.
2. `build_planner_skill_list(skills: list[dict]) -> str`:
   - Format skill list for planner context, same pattern as `build_planner_tool_list()`.
   - Each entry: `- {name} ÔÇö {summary}` + instructions block.

**Test:** Unit tests with temp .md files.

### Phase 11: LLM resilience ÔÇö fix transient failure cascade (6/30 functional tests)

**Root cause analysis**: 6 functional test failures all stem from transient `httpx.RequestError` in
`call_llm` that cascade into plan failures. Four compounding issues:

1. `httpx.RequestError` at `llm.py:388` uses `str(e)` which is empty for some subclasses
   (RemoteProtocolError, ReadError) Ôćĺ opaque `"LLM request failed: "` log lines
2. `call_llm` immediately raises on `httpx.RequestError` ÔÇö no retry at the transport level
3. `run_messenger` (`brain.py:1905`) calls `call_llm` directly with zero retries ÔÇö a single
   network blip kills the msg task (unlike structured roles which get 3 retries via
   `_retry_llm_with_validation`)
4. `_handle_msg_task` (`loop.py:915`) returns `stop=True, stop_success=False` on any
   LLM/MessengerError ÔÇö even when all prior exec tasks succeeded and only the summary msg
   fails, the entire plan is marked failed

**Affected tests** (all functional):
- `test_fibonacci_script`: all 3 exec tasks succeeded, but final msg task failed Ôćĺ plan "failed"
- `test_learning_retention`: messenger LLM error, fast-path messenger also failed
- `test_curator_entity_creation_and_recall`: LLM error during search + messenger
- `test_search_synthesize_publish`: LLM error during search
- `test_ssh_key_display`: LLM errors + plan validation failure
- `test_chat_kb_self_hostname`: 60s timeout (LLM stall)
