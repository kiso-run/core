# Devplan v0.6 — Plugin Taxonomy Redesign

## Problem Statement

The current naming convention is inconsistent with the broader AI ecosystem. "Skills" in
kiso map to what Claude Code, MCP, and most agent frameworks call "tools" — executable
packages that perform actions. The term "skill" is overloaded and confusing for new users.

Additionally, there's no lightweight concept for prompt-only instructions that guide the
planner without requiring an executable package.

## New Taxonomy

- **plugin** — generic umbrella term for any installable extension
  - **tool** — executable package (browser, aider, search) — what was called "skill"
  - **skill** — lightweight `.md` file with planner instructions (new concept)
  - **connector** — external integration (slack, telegram) — unchanged

## Design Principles

1. **tool** aligns with Claude Code, MCP, and function-calling conventions
2. **skill** becomes a simpler, lower-friction concept — just a markdown file
3. **connector** stays unchanged — already well-defined
4. **plugin** provides the grouping term for CLI (`kiso plugin list`)

## Scope of Rename

The skill → tool rename touches:

**Core modules:**
- `kiso/skills.py` → `kiso/tools.py` (file rename + all functions/classes)
- `kiso/brain.py` — `TASK_TYPE_SKILL→TASK_TYPE_TOOL`, schema fields, validators, prompt builders
- `kiso/store.py` — `tasks.skill` column → `tasks.tool`, `TaskDict.skill→tool`, `create_task` param
- `kiso/worker/skill.py` → `kiso/worker/tool.py` — `_skill_task→_tool_task`
- `kiso/worker/loop.py` — all skill imports, installed_skills, user_skills, task dispatch
- `kiso/skill_repair.py` → `kiso/tool_repair.py`
- `kiso/security.py` — `PermissionResult.skills→tools`, access control
- `kiso/config.py` — `User.skills→tools`
- `kiso/main.py` — `user_skills→user_tools`

**CLI:**
- `cli/skill.py` → `cli/tool.py` — `kiso skill→kiso tool`, `OFFICIAL_PREFIX="tool-"`
- `cli/__init__.py` — parser registration, subcommands

**Registry + repos:**
- `registry.json` — `"skills"→"tools"`, repo prefix `skill-→tool-`
- External repos: `skill-browser→tool-browser`, `skill-search→tool-websearch`, etc. (user renames)

**Prompts:**
- `kiso/roles/planner.md` — task type `skill→tool`, all skill references
- `kiso/roles/briefer.md` — `skills` field, module descriptions
- `kiso/roles/worker.md` — skill venv, short names
- `kiso/roles/reviewer.md` — skill references

**Manifest format:**
- `[kiso.skill]` → `[kiso.tool]` in tool kiso.toml files
- `[kiso.skill.args]` → `[kiso.tool.args]`
- Env vars: `KISO_SKILL_{NAME}_{KEY}` → `KISO_TOOL_{NAME}_{KEY}`

**Directory paths:**
- `~/.kiso/skills/` → `~/.kiso/tools/`

**DB migration:**
- `tasks.skill` column → `tasks.tool` (ALTER TABLE RENAME COLUMN)

**Tests (12+ files):**
- `test_skills.py→test_tools.py`, `test_cli_skill.py→test_cli_tool.py`, etc.
- All mocks: `patch("kiso.worker.loop.discover_skills"→"discover_tools")`, etc.

### Phase 2: MD-based skills (new concept)

### M449 — Skill loader: discover and parse .md skill files

**Files:** `kiso/skill_loader.py` (new)

**Change:**

1. New module `kiso/skill_loader.py`:
   - `discover_md_skills(skills_dir: Path | None = None) -> list[dict]`
   - Default dir: `~/.kiso/skills/` (distinct from `~/.kiso/tools/`)
   - Each `.md` file in the directory is a skill.
   - Parse frontmatter (YAML between `---` markers):
     ```yaml
     ---
     name: data-analyst
     summary: Guides planner for data analysis tasks
     ---
     ```
   - Body is the planner instruction text.
   - Return list of `{"name": str, "summary": str, "instructions": str, "path": str}`.
   - No trigger matching — all discovered skills go to the briefer's context pool,
     consistent with how tools are handled. The briefer decides relevance.
   - TTL cache like `discover_tools()` to avoid repeated filesystem scans.
2. `build_planner_skill_list(skills: list[dict]) -> str`:
   - Format skill list for planner context, same pattern as `build_planner_tool_list()`.
   - Each entry: `- {name} — {summary}` + instructions block.

**Test:** Unit tests with temp .md files.

```python
# kiso/registry.py
import json
import logging

import httpx

log = logging.getLogger(__name__)

REGISTRY_URL = (
    "https://raw.githubusercontent.com/kiso-run/core/main/registry.json"
)

_registry_cache: dict | None = None
_registry_ts: float = 0.0
_REGISTRY_TTL: float = 300.0  # 5 min cache


def fetch_registry() -> dict:
    """Fetch registry, cached for 5 min. Returns {} on error."""
    import time
    global _registry_cache, _registry_ts
    now = time.monotonic()
    if _registry_cache is not None and (now - _registry_ts) < _REGISTRY_TTL:
        return _registry_cache
    try:
        resp = httpx.get(REGISTRY_URL, timeout=10.0, follow_redirects=True)
        resp.raise_for_status()
        _registry_cache = json.loads(resp.text)
        _registry_ts = now
        return _registry_cache
    except Exception as exc:
        log.warning("Failed to fetch registry: %s", exc)
        return _registry_cache or {}


def search_entries(entries: list[dict], query: str | None) -> list[dict]:
    """Filter registry entries: match name first, then description."""
    if not query:
        return entries
    q = query.lower()
    by_name = [e for e in entries if q in e["name"].lower()]
    if by_name:
        return by_name
    return [e for e in entries if q in e.get("description", "").lower()]
```

  python3 script.py

### Phase 40: Install approval — server-side detection replaces keyword heuristic

### M615 — Detect install proposals from validation state, not keyword matching

**Problem:** `session_has_install_proposal()` uses a fragile keyword heuristic to
detect whether the previous plan proposed a tool installation.  It checks the msg
task `detail` for substrings like `"would you like"`, `"shall i"`, `"permission"`.
This fails when:

- The LLM uses a slightly different phrasing (e.g. `"Ask the user whether they
  would like to install..."` — `"would you like"` is not a substring because the
  subject is `"they"`, not `"you"`)
- The detail is in Italian or another language (no Italian keywords are checked)
- The detail is generic (e.g. `"Answer in Italian. Inform about browser tool
  installation."`) and the actual proposal text only appears in the messenger
  `output` field, which is not checked at all

Consequence: `install_approved=False` on the second turn → validation tells the
planner "ask the user" again → infinite loop → no install ever happens.  This is
the root cause of F1 and F2 functional test failures.

**Approach:** Server-side deterministic detection — no LLM cooperation needed.

We already **know** when a plan is an install proposal: `_validate_plan_tasks()`
generates "tool not installed" errors, the LLM retries, and eventually produces a
single-msg plan.  This info exists in the retry loop — we just need to propagate it.

The detection lives in `run_planner`: after `_retry_llm_with_validation` returns a
valid plan, check if (a) validation saw uninstalled-tool errors during any retry,
AND (b) the final plan is msg-only.  If both true → it's an install proposal.
Store it on the plan row.

Zero dependence on LLM text, works in any language, 100% deterministic.

**Files:** `kiso/brain.py`, `kiso/store.py`, `kiso/worker/loop.py`,
`tests/test_install_confirm.py`

**Changes:**

1. **`_retry_llm_with_validation`** (`brain.py`): track whether any validation
   attempt produced an "is not installed" error.  After the loop exits
   successfully, attach a `_saw_uninstalled_tool: bool` attribute on the
   returned dict (same pattern as `exc.last_errors` on exceptions).

2. **`run_planner`** (`brain.py`): after getting the valid plan dict, check:
   ```python
   is_install_proposal = (
       getattr(plan, "_saw_uninstalled_tool", False)  # retry loop flagged it
       and all(t.get("type") == "msg" for t in plan["tasks"])
   )
   ```
   If true, set `plan["_install_proposal"] = True` (internal marker, not in
   JSON schema — stripped before return).  Return the marker alongside the plan.

   **Simpler alternative**: `run_planner` returns the plan dict and we add
   `install_proposal` as a regular key.  `_process_message` reads it.

3. **`create_plan`** (`store.py`): add `install_proposal TEXT` column to plans
   table (nullable).  Add migration for existing DBs.  Accept optional
   `install_proposal` parameter.

4. **`session_has_install_proposal`** (`store.py`): replace keyword heuristic:
   ```python
   cur = await db.execute(
       "SELECT install_proposal FROM plans "
       "WHERE session = ? AND status IN ('done', 'failed') "
       "ORDER BY id DESC LIMIT 1",
       (session,),
   )
   row = await cur.fetchone()
   return bool(row and row["install_proposal"])
   ```

5. **`_process_message`** (`worker/loop.py`): after `run_planner` returns,
   persist `install_proposal` on the plan row.

6. **No schema changes** — `PLAN_SCHEMA` stays the same.  No prompt changes.
   No LLM cooperation needed.

**Detection logic (in `_retry_llm_with_validation` or `run_planner`):**

```python
# In _retry_llm_with_validation: track uninstalled-tool errors
_TOOL_NOT_INSTALLED_MARKER = "is not installed"
saw_uninstalled = False
# ... inside the loop, after validate_fn returns errors:
if any(_TOOL_NOT_INSTALLED_MARKER in e for e in errors):
    saw_uninstalled = True
# ... after loop exits with valid result:
result["_saw_uninstalled_tool"] = saw_uninstalled

# In run_planner: detect install proposal
saw_uninstalled = plan.pop("_saw_uninstalled_tool", False)
is_proposal = saw_uninstalled and all(
    t.get("type") == "msg" for t in plan["tasks"]
)
```

**Test:**
- Existing `TestSessionHasInstallProposal` tests: rewrite to use new column
- New: plan with `install_proposal` set → `session_has_install_proposal` True
- New: plan without → returns False
- New: `_retry_llm_with_validation` sets `_saw_uninstalled_tool` when validation
  sees uninstalled-tool errors
- New: `run_planner` detects install proposal (single msg + saw_uninstalled)
- New: `run_planner` does NOT flag as proposal when plan has exec tasks
- Different session isolation still works

**Impact:** Fixes F1/F2 functional tests.  Eliminates all keyword heuristics.
Works in any language.  Zero LLM dependence — purely server-side detection.

## Phase 41: Testing Architecture — Connectors, Plugins & Interactive

### Problem Statement

Three critical testing gaps:

1. **Supervisor lifecycle** has zero test coverage — crash/backoff/SIGTERM/gave_up are untested.
2. **Connector protocol** (session registration → msg → webhook delivery → polling fallback) is only tested as isolated webhook unit tests, never as an end-to-end flow.
3. **Plugin tests live in plugin repos** but there's no way to run them from kiso core, and no way to run live connector tests against a real platform.
4. **Interactive scenarios** (CAPTCHA, external verification, OAuth) require a real human and have no test infrastructure at all.

### Architecture

```
┌──────────────────────────────────────────────────────────────┐
│  L4: Interactive          --interactive                      │
│  Real human in the loop — CAPTCHA, Moltbook signup, OAuth    │
│  Terminal prompt pauses test, human acts, test resumes        │
├──────────────────────────────────────────────────────────────┤
│  L3: Plugin tests         kiso tool test / kiso connector test│
│  Run each plugin's own test suite from kiso CLI              │
│  --live variant runs integration tests against running kiso  │
├──────────────────────────────────────────────────────────────┤
│  L2: Integration          tests/integration/                 │
│  Stub connector + mock LLM against real kiso ASGI app        │
│  Protocol compliance, multi-turn flows, webhook delivery     │
├──────────────────────────────────────────────────────────────┤
│  L1: Unit                 tests/ (existing)                  │
│  Supervisor mock, webhook, CLI, validate_plan                │
│  Always runs, no flags needed                                │
└──────────────────────────────────────────────────────────────┘
```

#### Responsibility boundaries

| What to test                          | Where it lives             | Who owns it |
|---------------------------------------|----------------------------|-------------|
| Connector internal logic (mapping, parsing, platform SDK) | `connector-*/tests/` | Plugin repo |
| Connector integration vs kiso API     | `connector-*/tests/test_integration.py` | Plugin repo |
| Supervisor lifecycle (spawn, backoff, SIGTERM, gave_up) | `tests/test_supervisor.py` | kiso core |
| Webhook delivery (retry, HMAC, truncation) | `tests/test_webhook.py` | kiso core (done) |
| Connector CLI (install, list, search) | `tests/test_cli_connector.py` | kiso core (done) |
| Connector protocol e2e (stub connector ↔ kiso API) | `tests/integration/` | kiso core |
| Multi-turn flows (approval, stuck→replan) | `tests/integration/` | kiso core |
| Run plugin's own tests from kiso      | `kiso tool test` / `kiso connector test` | kiso core (orchestration) |
| Real human interaction (CAPTCHA, signup) | `tests/interactive/` | kiso core |

#### Interactive test flow

Interactive tests handle scenarios where the agent **cannot proceed without
a real human performing an action outside the system**.  Examples:

- **CAPTCHA**: agent takes screenshot, shows it to user, user types solution
- **Moltbook signup**: agent says "tweet X to verify", human tweets, says "done"
- **OAuth**: agent says "click this link and authorize", human does it

The mechanism:

```
test starts → agent pipeline runs → agent sends msg asking human to act
    → test framework intercepts msg → prints it to terminal
    → waits for human input (stdin prompt or marker file)
    → feeds human response as next user message
    → agent pipeline continues → test verifies final result
```

This is NOT for flows where the human response can be predicted and mocked
(install approval, simple yes/no).  Those belong in L2 integration tests.

#### Plugin test orchestration

`kiso connector test <name>` and `kiso tool test <name>` are convenience
commands that run the plugin's own test suite from the kiso CLI:

```bash
# Unit tests (no kiso instance needed)
kiso connector test discord
# → cd ~/.kiso/connectors/discord && uv run --group dev pytest tests/ -v \
#     --ignore=tests/test_integration.py

# Integration tests (requires running kiso)
kiso connector test discord --live
# → cd ~/.kiso/connectors/discord && uv run --group dev pytest tests/test_integration.py -v

# All installed plugins
kiso plugin test-all
kiso plugin test-all --live
```

The `run_full_tests.sh` gains `--plugins` and `--plugins-live` flags.

   - **`interactive` pytest marker** — gated by `--interactive` CLI flag
   - **`human_relay` fixture**: intercepts agent msg-type task outputs and
     presents them to the terminal operator, then collects their response:
     ```python
     class HumanRelay:
         """Bridges agent ↔ human during interactive tests."""

         def __init__(self, run_message_fn):
             self._run = run_message_fn
             self.transcript: list[dict] = []

         async def send(self, content: str, **kw) -> FunctionalResult:
             """Send a message, intercept if agent asks for human action."""
             result = await self._run(content, **kw)
             self.transcript.append({"role": "agent", "content": result.msg_output})
             if self._needs_human_action(result):
                 human_input = self._prompt_human(result.msg_output)
                 self.transcript.append({"role": "human", "content": human_input})
                 result = await self._run(human_input, **kw)
                 self.transcript.append({"role": "agent", "content": result.msg_output})
             return result

         def _needs_human_action(self, result: FunctionalResult) -> bool:
             """Detect if the agent is waiting for human action."""
             for task in result.tasks:
                 if task.get("stuck_reason"):
                     return True
             return False

         def _prompt_human(self, agent_msg: str) -> str:
             """Show agent message, wait for human input."""
             print("\n" + "=" * 60)
             print("AGENT SAYS:")
             print(agent_msg)
             print("=" * 60)
             return input("Your response (or 'done' if you completed the action): ")
     ```

## Phase 42: Functional Test Robustness — Planner Compliance & Transport Resilience

### Problem Statement

Four functional tests (F1, F2, F7, F8) fail systematically due to three
distinct classes of problems:

- **A — Atomic CLI decomposition**: The planner breaks single CLI commands
  (`kiso tool install browser`) into multi-step plans with analytical exec
  tasks ("check the content of kiso.toml") that the exec translator cannot
  convert to shell commands.  This generalizes: any kiso CLI operation can be
  over-decomposed.

- **B — Inline artifacts**: The planner embeds file content in msg tasks
  instead of creating files via exec, even when the user explicitly requests
  a file.  Auto-publish never fires because no file is written to disk.
  Generalizes to any artifact type (markdown, CSV, scripts, images).

- **C — Transport error cascade**: When OpenRouter is degraded, every LLM
  call in the pipeline wastes time on doomed retries.  A 10-call pipeline
  with 3 attempts each = 30 wasted requests, burning the entire timeout
  budget.  No mechanism detects provider-level degradation.

### M629 — LLM transport circuit breaker

**Problem:** When OpenRouter is degraded, every `call_llm()` in the pipeline
does up to 3 attempts (1 + 2 retries × 1-2s backoff).  A typical plan cycle
involves ~10 LLM calls (classifier, briefer, planner, exec translator,
reviewer, messenger, summarizer, curator).  With 30 wasted requests and
backoff delays, the pipeline timeout (180-300s) is consumed by doomed retries.

No global state tracks provider health across calls.  Affects F8 (and any
test running during provider degradation).

**Files:** `kiso/llm.py`, `tests/test_llm.py`

**Change:**

1. Add module-level circuit breaker state:

   ```python
   # Circuit breaker — protects against provider-wide degradation
   _CB_FAILURE_THRESHOLD = 5       # open after 5 consecutive transport failures
   _CB_COOLDOWN = 30.0             # stay open for 30s
   _cb_consecutive_failures = 0
   _cb_open_until = 0.0            # monotonic timestamp
   _cb_lock = asyncio.Lock()       # protect state in concurrent calls

   async def _cb_record_failure() -> bool:
       """Record transport failure.  Returns True if circuit just opened."""
       ...

   async def _cb_record_success() -> None:
       """Reset failure counter on success."""
       ...

   def _cb_is_open() -> bool:
       """Check if circuit is open (should fail fast)."""
       return time.monotonic() < _cb_open_until
   ```

2. In `call_llm()`, before transport retry loop:
   ```python
   if _cb_is_open():
       raise LLMError("Circuit breaker open — provider degraded, failing fast")
   ```

3. After successful response: `await _cb_record_success()`.

4. After transport error (in retry handler): `await _cb_record_failure()`.
   If circuit just opened, log warning with provider name.

5. Half-open: when `_cb_is_open()` returns False after cooldown, allow one
   probe request.  If it succeeds → closed.  If it fails → re-open.

6. **Test reset**: add `_cb_reset()` helper, call it from the autouse
   `_no_retry_backoff` fixture so tests start with a clean circuit.

**Test:**
- 5 consecutive transport errors → circuit opens → next call raises immediately
- Success after failures → counter resets
- Circuit open → wait cooldown → half-open → probe success → closed
- Circuit open → wait cooldown → half-open → probe failure → re-open
- Concurrent calls don't corrupt state (asyncio.Lock)

### M644 — `stdin=DEVNULL` in `_run_subprocess`

**Problem:** `_run_subprocess` in `kiso/worker/utils.py` doesn't set `stdin`
on the subprocess.  It inherits from the parent process, which may be a real
terminal.  Subprocesses like `deps.sh` → `sudo apt-get install` see the TTY
and prompt for passwords, blocking execution.

Even inside Docker (where root doesn't need sudo), this is a security issue —
exec tasks should never read from stdin.

**Files:** `kiso/worker/utils.py` (`_run_subprocess`), `tests/test_worker.py`

**Change:**

1. In `_run_subprocess`, add `stdin=asyncio.subprocess.DEVNULL`:

   ```python
   kwargs: dict = dict(
       cwd=cwd,
       env=env,
       stdin=asyncio.subprocess.DEVNULL,  # exec tasks must not be interactive
       stdout=asyncio.subprocess.PIPE,
       stderr=asyncio.subprocess.PIPE,
   )
   ```

2. This makes any interactive prompt (sudo, apt confirmation, etc.) fail
   immediately instead of blocking.

**Test:**
- Subprocess that tries to read stdin → gets EOF immediately
- Existing exec tests still pass
