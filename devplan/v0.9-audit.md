## test_worker.py

130 test classes by surface area: subprocess execution, exec/msg/tool
task handlers, runWorker loop, review task path, plan execution,
plan output handling, curator/learning side effects, post-install
rescan, webhook delivery, large output handling, replan context
building, sanitization, tool venv, etc. Each class targets a distinct
production function or scenario class.

**Candidates considered for cuts:** none with high confidence at this
audit depth. Class names are descriptive and non-overlapping. The
size reflects breadth of coverage on a runtime-critical module, not
accumulated dead weight.

**What WOULD be needed for a deeper prune:** per-class line audit to
find scenario duplication within the larger classes (e.g.
`TestRunWorker` and `TestExecutePlan` may have multi-line scenarios
that could be merged). This is multi-hour work and should be a
maintainer-driven follow-up rather than an autonomous batch.

**Recommendation:** keep as-is for v0.9. Open a v0.10 follow-up for
"deep audit of the 5 largest test classes in test_worker.py with
explicit duplication report".

## test_brain.py

129 classes by role (planner/reviewer/curator/messenger/summarizer/
paraphraser/exec_translator/classifier/searcher/briefer) and by
helper (_clean_learn_items, learning_contradicts_output,
validate_plan, load_system_prompt, build_*_messages, etc.). Each
role and each helper has its own test class. The structure mirrors
the production module layout.

**Candidates considered for cuts:** none with high confidence. Tests
covering fallback_model are exhaustive (8+ in this file alone) and
all referenced from M1271's deferral note — they exist for a reason.

**Recommendation:** keep as-is for v0.9. Same v0.10 follow-up note as
test_worker.py.

## test_safety_rules.py vs tests/functional/test_safety_rules.py

- `tests/test_safety_rules.py` (279 lines, 12 tests, 3 classes):
  `TestFRulesPathDisclosure`, `TestFRulesBehaviorCitation`,
  `TestFRulesLifecycle`. These are **unit-tier** tests of the safety
  rule lifecycle (CRUD, in-memory enforcement, path disclosure
  helpers).
- `tests/functional/test_safety_rules.py` (139 lines, 6 tests):
  pre-seeded safety rule blocks dangerous operations through the
  full pipeline with a real LLM.

**Verdict:** intentional boundary, NOT duplication. Unit covers the
helper logic, functional covers the runtime enforcement under real
planner/reviewer behavior. Keep both.

## test_security.py

8 classes: `TestCheckCommandDenyList`, `TestBuildSecretVariants`,
`TestSanitizeOutput`, `TestSanitizeValue`, `TestCollectDeploySecrets`,
`TestFenceContent`, `TestRevalidatePermissions`,
`TestNoDoubleMasking`. These cover the **secret/sanitization
infrastructure** in `kiso.security`, distinct from the safety rule
enforcement covered by `test_safety_rules.py`.

**Verdict:** distinct domain. Keep.

## Files explicitly NOT recommended for deletion (re-confirming
M1268 audit findings)

- `tests/test_rename_completeness.py` (54 lines, 4 tests): runtime
  invariants for the post-rename module layout. **Keep.**
- `tests/test_import_surfaces.py` (86 lines, 4 tests): public
  surface guard for `kiso.brain` / `kiso.main` / `kiso.worker`.
  **Keep.**

## Summary

No cuts proposed at this audit depth. The largest files are
well-structured by class and cover distinct production surfaces.
The size reflects real coverage, not accumulated cruft. A deeper
per-class scenario review (multi-hour, maintainer-driven) is the
appropriate next step if the team wants further reduction; that
work belongs to a v0.10 follow-up milestone, not autonomous
execution.
