# kiso-run/core — v0.10 Development Plan

## Headline

**Convergence to skills + MCP. Kiso/core stops shipping or maintaining tools.**

Today kiso has four overlapping extension primitives — recipes,
wrappers, prompt modules, and MCP — with fuzzy routing rules
between them. The briefer
(`kiso/brain/common.py:1382`) has to disambiguate per task, the
planner prompt has to explain the difference, and contributors
have to learn four concepts where two would suffice.

v0.10 collapses the model to **two orthogonal primitives**
(plus exec as the fallback):

- **Skills** — Markdown units with frontmatter, optionally
  bundling scripts/templates, describing *how to think* about a
  class of problem. Replaces recipes 1:1.
- **MCP** — Standardized protocol for *what to call*. Replaces
  every wrapper without exception.
- **Exec** — Pure shell commands. Unchanged.

Prompt modules (`<!-- MODULE: xxx -->` blocks in `planner.md`,
hardcoded list in `kiso/brain/common.py:1023-1043`) are NOT in
this convergence — they are prompt fragments, not capabilities,
and stay where they are.

**Kiso/core does not ship or maintain any tool code.** Capability
gaps in the public MCP ecosystem (codegen, transcription, OCR,
local document parsing) are filled by **four standalone MCP
servers published under the `kiso-run` GitHub org and distributed
via PyPI**. They are maintained on the same release cadence as
`kiso/core` itself but live in their own repos, identical in
status to any third-party MCP server from the user's point of
view. There is no "privileged" tool layer inside `kiso/core`.

## Prerequisites from v0.9

This plan assumes the v0.9 Phase 19 milestones (M1367-M1372)
have landed. In particular:

- **M1367** — `npx` is part of the runtime baseline. `uvx` is
  also available because `uv` is already installed for kiso
  itself, no extra work.
- **M1368** — planner routes Node tooling through `npx`, not
  global installs.
- **M1370** — the briefer already sees MCP methods as a
  first-class category in its input pool. Without this milestone
  the v0.10 refactor degenerates into "introduce a new concept"
  instead of "rename and merge".

## Non-goals (explicit)

- **No skill discovery infrastructure.** Skills are loaded from
  `~/.kiso/skills/*.md` at briefer-input assembly time, the
  briefer LLM picks. The discovery mechanism is the existing
  briefer call. No retrieval, no embeddings, no scoring.
- **No registry, no catalog runtime.** Kiso sees only what is
  installed in `mcp.json`. If an MCP is not installed, it does
  not exist for kiso. The planner prompt already contains this
  rule (`kiso/roles/planner.md:74`).
- **No marketplace, no service.** The "ecosystem" is curated
  Markdown pages in the docs.
- **No tools in `kiso/core`.** Not as wrappers, not as builtin
  modules, not as in-tree MCP servers. Anything that does work
  for the user lives in a separate repo.
- **No "wrapper feature porting" milestone.** Of the five
  wrapper-side features that today live in `kiso/wrappers.py`
  (session/workspace injection, `consumes` file routing, arg
  alias normalization, role filtering, healthy reporting), four
  do not need porting at all (MCP env-at-spawn, tool description
  quality, modern schema validation, native MCP handshake) and
  the fifth (role filtering) is ~20 lines folded into M1502.

## Trust rule for the default preset

The default preset shipped with `kiso init --preset default`
contains **only MCP servers maintained by trustworthy upstreams**:

- **Tier 1**: organizations with public release pipelines and
  verifiable track records (Anthropic `@modelcontextprotocol/*`,
  Microsoft `@playwright/mcp`, GitHub `@github/*`).
- **Tier 2**: `kiso-run/*-mcp` servers maintained by the same
  team that maintains `kiso/core`. Listed at the same trust
  level only because the kiso project itself is the upstream.

Explicitly excluded from the default preset:
- Single-developer experiments, however popular.
- Forks of enterprise servers.
- Servers requiring paid API keys *for basic operation* (free
  tier OK; required paid key not OK in default).
- Community servers from public registries (`mcp.so`,
  `glama.ai`, `smithery.ai`) without org backing.

These can still be **recommended in docs** under
`docs/recommended-mcps.md` with explicit disclaimers about
provenance and maintenance status — but they are not in the
default preset, ever.

Every future PR that modifies `kiso/presets/default.json` must
demonstrate that the new entry passes this rule. The rule lives
in the `Done when` of M1503 and in the docs page itself.

# Track A — `kiso/core` cleanup

Six milestones inside this repo. Each lands independently and
the order matters because later milestones depend on earlier
ones (skill loader before briefer schema, briefer schema before
wrapper retirement, wrapper retirement before final prompt
cleanup).

## Phase 1 — Skill foundation (M1500-M1501)

### M1500 — `kiso/skill_loader.py` and `~/.kiso/skills/` discovery

**Why.** Skills replace recipes as the knowledge primitive. They
need a loader symmetric to `kiso/recipe_loader.py:32` — same TTL
cache (30s), same frontmatter parsing, same invalidation hook on
install/remove — but extended to support optional bundled assets
(scripts, templates) sitting alongside the `.md` file.

**Approach.** Copy the `recipe_loader.py` shape; new
`~/.kiso/skills/` root; each skill is either a single
`<name>.md` or a directory `<name>/` containing `skill.md` plus
arbitrary files. Frontmatter: `name`, `description`,
`when_to_use`, `bundled_files` (optional). The loader returns a
list of `Skill` dataclasses with `name`, `description`, `body`,
`bundled_root`.

**Tasks**:
- [ ] `kiso/skill_loader.py` with `discover_skills()`, TTL cache,
      `invalidate_skills_cache()`.
- [ ] `Skill` dataclass.
- [ ] Frontmatter parser shared with recipe loader where possible.
- [ ] Unit tests: single-file skill, directory skill with
      bundled assets, malformed frontmatter, cache invalidation.
- [ ] `kiso/config.py` exposes `SKILLS_DIR` constant under
      `KISO_DIR`.
- [ ] `kiso/main.py` ensures the skills dir is precreated at
      init (mirroring M1288's pattern).
- [ ] Commit & push.

**Done when**:
- A skill at `~/.kiso/skills/foo.md` (or
  `~/.kiso/skills/foo/skill.md`) is discovered, parsed, and
  returned by `discover_skills()`.
- Bundled assets are accessible via `Skill.bundled_root`.

## Phase 2 — Briefer schema convergence (M1502)

### M1502 — Briefer schema: drop `wrappers`/`exclude_recipes`, add `skills`

**Why.** Today the briefer JSON output
(`kiso/brain/common.py:994`, `kiso/roles/briefer.md:3-12`) has
8 fields covering 4 categories. After convergence the briefer
reasons about two orthogonal primitives: skills and MCP methods.
Its output schema simplifies accordingly. The fewer categories
the briefer has to disambiguate, the fewer routing errors.

This milestone also folds in the *only* wrapper-side feature
worth preserving: per-user role filtering. Today
`build_planner_wrapper_list` (`kiso/wrappers.py:246`) filters
the wrapper catalog by `user_role`. The same filter is applied
to the MCP method catalog at briefer-input assembly time
(~20 lines). The other four wrapper-side features
(session/workspace injection, `consumes` routing, arg alias
normalization, healthy reporting) are *not* ported — they
correspond to MCP capabilities that already exist natively or
are no longer needed (see "Non-goals" above).

**Approach.**
- New schema: `skills`, `mcp_methods`, `modules`, `context`,
  `output_indices`, `relevant_tags`, `relevant_entities`. Drop
  `wrappers` and `exclude_recipes`.
- Update `kiso/brain/common.py:994` (response schema), `:1237`
  (input section assembly: replace "Available Wrappers" /
  "Available Recipes" with "Available Skills"; "Available MCP
  Methods" already present from M1370), `:1412-1428`
  (post-filter), and the consumers in
  `kiso/brain/planner.py:1321-1353`.
- Add user-role filtering on the MCP catalog presented to the
  briefer (mirror of `build_planner_wrapper_list:246`).
- Rewrite `kiso/roles/briefer.md` rules: skills + MCP categories,
  drop the wrapper/recipe disambiguation paragraph.
- `kiso/worker/message_flow.py:185` (messenger briefer call)
  already forces wrappers/modules empty — minimal impact.

**Tasks**:
- [ ] Update briefer JSON schema (`common.py:994`).
- [ ] Update input section assembly (`common.py:1237`).
- [ ] Update post-filter (`common.py:1412-1428`).
- [ ] Update planner consumer (`planner.py:1321-1353`).
- [ ] Add MCP catalog role filtering at briefer input.
- [ ] Rewrite `briefer.md`.
- [ ] Update messenger briefer call site.
- [ ] Migrate every existing test that mocks the briefer to the
      new schema.
- [ ] Commit & push.

**Done when**:
- Briefer returns the new schema.
- All mocked-briefer tests updated.
- `wrappers` and `exclude_recipes` fields no longer appear
  anywhere in the briefer codepath.
- Per-user role filtering on the MCP catalog works (admin sees
  all, restricted user sees only allowed servers).

## Phase 3 — Default preset (M1503)

### M1503 — `kiso/presets/default.json` and `kiso init --preset` shim

**Why.** With no registry and no catalog runtime, new users need
*something* on first init that gives them working capabilities
out of the box. The answer is a curated default preset: a
static JSON file shipped in `kiso/core` that `kiso init` copies
into `~/.kiso/mcp.json` at first install.

The preset is bootstrap-only. After `init`, kiso never reads
the preset file again — it reads the user's `mcp.json`, which
the user is free to modify. There is no "preset upgrade" path;
when we ship a new preset version it affects only fresh
installs.

**Approach.**
- Create `kiso/presets/default.json` with the eight servers
  listed in the "Default preset" section above.
- Extend `kiso/main.py` (or `cli/__init__.py`) `init` command
  with `--preset <name>` flag. Default behavior: `kiso init`
  with no flag still applies `default` preset.
- `kiso init --preset none` for users who want an empty
  `mcp.json`.
- Document required env vars (`OPENROUTER_API_KEY`,
  `GEMINI_API_KEY`) in the post-init message.
- Add `docs/default-preset.md` describing each server, its
  upstream, and its trust tier (Tier 1 / Tier 2 from the trust
  rule above).
- Add `docs/recommended-mcps.md` listing community MCP options
  that are NOT in the default, with disclaimers.
- Encode the trust rule in a CI check or test that fails if
  `default.json` contains a server whose `command` is not on
  the allowlist of approved upstreams (`@modelcontextprotocol/*`,
  `@playwright/*`, `@github/*`, `kiso-*`). This is the
  enforcement mechanism for the trust rule.

**Tasks**:
- [ ] `kiso/presets/default.json` written.
- [ ] `init --preset` flag implementation.
- [ ] Post-init message listing required env vars.
- [ ] `docs/default-preset.md` written.
- [ ] `docs/recommended-mcps.md` written.
- [ ] CI check / test for trust rule allowlist.
- [ ] Functional test: `kiso init --preset default` produces the
      expected `mcp.json` and the briefer can see the eight
      servers as MCP methods (assumes M1370 landed).
- [ ] Commit & push.

**Done when**:
- A fresh `kiso init` produces the default preset `mcp.json`.
- The trust-rule CI check passes for the shipped default and
  fails on a deliberately-bad PR.
- Both docs pages exist and are linked from `docs/index.md` (or
  whatever the docs entry point is).

## Phase 4 — Wrapper system retirement (M1504)

### M1504 — Delete `kiso/wrappers.py`, `kiso/registry.py`, `cli/wrapper.py`, and friends

**Why.** After M1502 the briefer no longer reads `wrappers` or
`exclude_recipes`. After M1503 new users get capabilities via
the default preset, not via wrapper install. The wrapper
subsystem has no live consumers in `kiso/core`. Time to delete
the dead code and shrink the surface.

**Approach.** Hard deletion. No deprecation period — the
`kiso wrapper install` flow is replaced by "edit `mcp.json`"
which is functionally simpler. Users running with installed

documented in `docs/v0.10-migration.md` (single page,
hand-written): "if you used `kiso wrapper install <name>`, the
v0.10 equivalent is to add an entry in `~/.kiso/mcp.json`

Four standalone MCP server repos under the `kiso-run` GitHub
org, each published on PyPI. They are the upstream for the
four Tier-2 entries in the default preset. Work in this track
happens *outside* this repo — these milestones track scope but

**Track B can start as soon as M1503 (default preset) is
underway**, because the preset references the four PyPI package
names. Track B does not block any Track A milestone — if a
Track B repo is delayed, the preset can ship with that entry
commented out and a docs note.

Before any Track B work starts, register the four package names

- `kiso-aider-mcp`
- `kiso-transcriber-mcp`

## M1506 — `kiso-run/aider-mcp`

**Why.** The kiso `wrapper-aider` repo
(`github.com/kiso-run/wrapper-aider`) provides codegen via
aider with OpenRouter routing and `MODEL_DEFAULTS`-driven model
selection. Public MCP ecosystem has no aider-equivalent server
maintained by a trusted upstream (verified in research dispatch:
JetBrains has `mcp-jetbrains` but requires a running IDE; every
"aider-mcp" project in the public registries is single-developer).
We fill the gap ourselves.

**Approach.** New repo `kiso-run/aider-mcp`. Python project
using the official `mcp` Python SDK. Single tool exposed:
`aider_codegen(prompt, editable_files, readonly_files,
model?)` returning `{success: bool, diff: str}`. Internally
shells out to `aider` the same way `wrapper-aider` does today.
Honors `OPENROUTER_API_KEY` from env; model defaults to a sane
codegen choice (e.g., `claude-sonnet-4-6` or whatever
`MODEL_DEFAULTS["codegen"]` is at the time of the port). Ships
on PyPI as `kiso-aider-mcp`. Entry point runs as `uvx
kiso-aider-mcp`.

**Tasks (in `kiso-run/aider-mcp` repo)**:
- [ ] New repo bootstrap (LICENSE, README, pyproject.toml, CI).
- [ ] Port `wrapper-aider` execution logic into `aider_codegen`
      MCP tool.
- [ ] OpenRouter env wiring.
- [ ] Tests: unit (mock subprocess), integration (real aider
      against a tiny fixture repo).
- [ ] CI: lint, test, publish-on-tag to PyPI.
- [ ] First release `0.1.0` published to PyPI.
- [ ] Sanity-check from `kiso/core`: a bare `mcp.json` with
      this entry connects and exposes the tool.
- [ ] Commit & push (in this repo's devplan: mark M1506 ✅
      with a link to the published PyPI version).

**Done when**:
- `uvx kiso-aider-mcp` starts a working MCP server.
- `aider_codegen` invoked from kiso produces a diff against a
  small test repo using OpenRouter as the LLM provider.
- The package is on PyPI under a `kiso-run` maintainer account.

`pypdf`, `python-docx`, `openpyxl`, stdlib. Trusted-upstream
MCP alternatives exist (`Unstructured-IO/UNS-MCP`,
`LlamaCloud MCP`) but **all are cloud-API-based and require
paid keys for non-trivial usage**. The kiso wrapper is the only
fully-local solution in this space — porting it to MCP
preserves that property, and it is the *easiest* of the four
ports because it has no LLM, no cloud, no API key, just pure
Python file parsing.

**Approach.** New repo `kiso-run/docreader-mcp`. Python +
official `mcp` SDK. Three tools (matching the current wrapper's
three actions):
- `read_document(file_path, page_range?, max_chars?)` →
  `{text: str, format: str, truncated: bool}`
- `document_info(file_path)` → `{format, pages?, sheets?, ...}`
- `list_uploads(dir_path)` → `{files: list[{name, size, format}]}`

Internally uses `pypdf`, `python-docx`, `openpyxl`, stdlib `csv`
— the same dependencies as the wrapper. Output capped at 100k
characters (preserves the wrapper's existing safeguard). No env
vars, no API keys. Ships on PyPI as `kiso-docreader-mcp`.

**Tasks (in `kiso-run/docreader-mcp` repo)**:
- [ ] Repo bootstrap.
- [ ] Port wrapper logic to three MCP tools.
- [ ] Tests: unit per format (PDF, DOCX, XLSX, CSV,
      plaintext), one fixture file each.
- [ ] CI + PyPI publish on tag.
- [ ] First release `0.1.0`.
- [ ] Sanity-check from `kiso/core`.
- [ ] Commit & push (mark M1509 ✅).

**Done when**:
- `uvx kiso-docreader-mcp` starts a working MCP server.
- All three tools return correct output on a fixture set
  covering each supported format.
- Server runs with zero env vars and zero network access.
