# kiso-run/core — v0.10 Development Plan

## Headline

**Convergence to Agent Skills + MCP, with Kiso-defined runtime policy,
full lifecycle parity with Claude Code / openclaw, and single-key
onboarding.**

Today kiso has four overlapping extension primitives — recipes,
wrappers, prompt modules, and MCP — with fuzzy routing rules
between them. The briefer (`kiso/brain/common.py:1382`) has to
disambiguate per task, the planner prompt has to explain the
difference, and contributors have to learn four concepts where
two would suffice. On top of that, the planner emits six task
types (`exec`, `wrapper`, `msg`, `search`, `mcp`, `replan`) —
two of which (`wrapper`, `search`) are internal capability
layers that duplicate what MCP already provides.

v0.10 collapses the model to **two orthogonal standards** (plus
exec as the fallback):

- **Agent Skills** (`agentskills.io`) — standard skill packages
  centered on `SKILL.md`, optionally bundling
  scripts/references/assets. They describe *how to think* about a
  class of problem. Replaces recipes, but not as a planner-only
  primitive.
- **MCP** (`modelcontextprotocol.io`) — the standardized protocol
  for *what to call*. Replaces wrappers **and the built-in search
  task type** as the capability surface.
- **Exec** — Pure shell commands. Unchanged.

The planner task taxonomy collapses from six to four:

```
before:  exec | wrapper | msg | search | mcp | replan
after:   exec | mcp     | msg | replan
```

- **exec** — unstructured shell command (universal fallback)
- **mcp** — structured call to any capability server
- **msg** — respond to the user
- **replan** — control flow: re-plan after investigation

Prompt modules (`<!-- MODULE: xxx -->` blocks in `planner.md`,
hardcoded list in `kiso/brain/common.py:1049-1053`) are NOT in
this convergence — they are prompt fragments, not capabilities,
and stay where they are (though their contents are rewritten in
M1505 to reflect the new taxonomy).

**Kiso/core does not keep a privileged in-process tool layer.**
Capability gaps in the public MCP ecosystem (codegen, web search,
transcription, OCR, local document parsing) are filled by
**standalone MCP servers published under the `kiso-run` GitHub
org and distributed via PyPI**. They are maintained on the same
release cadence as `kiso/core` itself but live in their own
repos, identical in protocol status to any third-party MCP server
from the user's point of view.

What *is* still Kiso/core's job, and it is substantial:

- load standard skill packages and project only the role-relevant
  parts into prompts
- manage MCP install/bootstrap UX (`add/install/test/env/logs`)
- maintain trust policy, provenance, health checks, and migration
  affordances for users
- own the runtime policy gap that neither standard fills:
  per-session context, secret scoping, role visibility, schema
  validation, install-from-URL, trust tiers

In other words: **the standards stay standard; Kiso defines the
runtime policy around them, and the runtime policy is the
product.**

## Single-key philosophy

The default preset is designed so that **`OPENROUTER_API_KEY` is
the only mandatory secret** for full capability. All
kiso-maintained MCP servers that need an LLM or search backend
route through OpenRouter:

- **codegen** (aider) → LLM via OpenRouter
- **web search** → Perplexity Sonar via OpenRouter
- **transcription** → Gemini multimodal via OpenRouter
- **OCR** → Gemini vision via OpenRouter

Servers that need no API key at all (filesystem, fetch, browser,
docreader) work out of the box. `GITHUB_TOKEN` is the only
*optional* extra for the `github` server.

This collapses the onboarding from three required keys
(`OPENROUTER_API_KEY` + `GEMINI_API_KEY` + a search API key)
to **one**. The single-key constraint is a design invariant for
the default preset — any future server added to the preset must
either need no key or route through OpenRouter. This invariant
is enforced by CI (M1503).

## Claude Code / openclaw parity baseline

v0.10 must match the lifecycle UX of comparable generalist
agents. Concretely, every one of the following flows must work
end-to-end on a fresh install with only `OPENROUTER_API_KEY`:

1. **Install a skill from a link.** `kiso skill install
   --from-url <github|zip|raw SKILL.md>` lands the skill,
   validates frontmatter, warns on `scripts/` or `allowed-tools`,
   and makes it selectable by the briefer on the next message.
2. **Install an MCP from a link.** `kiso mcp install --from-url
   <pulsemcp|github|npm:|pypi:|server.json>` (already works
   today) and the new server is exposed to the planner.
3. **Planner proposes install when missing.** Goal mentions a
   domain with no skill or MCP covering it → planner produces
   msg-only plan proposing the install command → user approves
   → exec install → replan picks up the new capability.
4. **Briefer selects the right skill.** Skill metadata alone is
   enough; no vector retrieval needed because
   `activation_hints` pre-filter scales the catalog.
5. **Planner sees the right instructions.** Skill `## Planner`
   section is injected; `## Worker` stays out of planner
   context.
6. **Worker executes with skill context.** `## Worker` section
   reaches the exec translator prompt.
7. **Reviewer checks against `expect` plus skill `## Reviewer`**
   heuristics, without demoting `expect` as the primary
   criterion.
8. **MCP sessions stay isolated.** Two concurrent sessions
   hitting the same MCP server get isolated workspace contexts;
   `${session:workspace}` resolves per-call, not at daemon
   boot.
9. **Secrets stay scoped.** `kiso mcp env set <name> KEY VAL`
   writes a credential that is actually delivered to the
   subprocess at spawn, never bleeds across servers, and never
   appears in telemetry.
10. **Broken plugin is recoverable.** Unhealthy MCP server or
    missing skill surfaces to the planner with a concrete
    remediation step (reinstall/re-auth/report), not a silent
    failure.

Each line above maps to one or more of the milestones below.
Nothing in this list is deferred to v0.11. The intent is that at
v0.10 close, a new user can recreate the Claude Code experience
on kiso by pointing at a SKILL.md URL and an MCP server URL with
no manual file editing.

## Standards boundary

This plan follows two external standards and draws a hard line
between "standard" and "Kiso policy":

- **Agent Skills standard** (`agentskills.io`) defines the skill
  package shape: `SKILL.md`, YAML frontmatter, optional
  `scripts/`, `references/`, and other bundled assets. Standard
  frontmatter fields include `name`, `description`, `license`,
  `compatibility`, `metadata`, `allowed-tools`. The standard also
  defines naming conventions (lowercase, hyphens, 1-64 chars).
- **MCP standard** (`modelcontextprotocol.io`) defines the
  client/server protocol, lifecycle handshake, transports (stdio,
  streamable HTTP), and three primitives: **Tools** (executable
  functions), **Resources** (data sources), and **Prompts**
  (templates). Kiso already implements Streamable HTTP
  (2025-06-18) at `kiso/mcp/http.py:58`.

These standards do **not** decide:

- which internal Kiso role sees which skill content
- how Kiso selects skills for a request
- how Kiso installs or bootstraps a server package on the local
  machine
- how Kiso stores trust/provenance/health information
- how per-session context (workspace, secrets, user role) reaches
  long-lived MCP servers
- how capability discovery degrades on cold start

v0.10 therefore introduces a **Kiso runtime profile** on top of
both standards:

- **Kiso Skill Profile** — a backward-compatible convention for
  role-scoped consumption of a standard `SKILL.md`, with
  deterministic `activation_hints` pre-filtering.
- **Kiso MCP Runtime Policy** — a consistent install/bootstrap/
  health/trust/session layer around standard MCP servers.

## Non-goals (explicit)

- **No vector retrieval / embeddings for skills.** Skill
  discovery remains metadata-driven through the briefer plus a
  deterministic `activation_hints` pre-filter (M1502). At >100
  installed skills the pre-filter keeps the briefer budget sane;
  embeddings remain off-scope.
- **No registry, no catalog runtime.** Kiso sees only what is
  installed locally in `mcp.json` and `~/.kiso/skills/`. If a
  capability is not installed, it does not exist for kiso. The
  planner prompt already contains this rule for MCP
  (`kiso/roles/planner.md:72-75`) and extends it to skills in
  M1505. `docs/recommended-mcps.md` and
  `docs/recommended-skills.md` compensate by listing community
  options that users can install manually.
- **No marketplace, no service.** The "ecosystem" is curated
  Markdown pages in the docs.
- **No tools in `kiso/core`.** Not as wrappers, not as builtin
  modules, not as in-tree MCP servers. Anything that does work
  for the user lives in a separate repo.
- **No pretending that MCP standardizes installation.** It
  doesn't. Kiso must still own the user-facing install/bootstrap
  workflow around standard servers.
- **No pretending that Agent Skills standardizes role routing.**
  It doesn't. Kiso must still decide how skills are projected into
  planner/worker/reviewer/messenger prompts.
v0.10 is the last cycle before public release. **Nothing that is
legitimately part of the MCP + Skills lifecycle is deferred to
v0.11.** That includes MCP Resources, MCP Prompts, MCP Sampling,
sandboxing parity, connector simplification, and behavior
injection into the messenger role. All land in v0.10 via
milestones in Phase 10.

## Design principles

1. **Standards first, Kiso profile second.** Never fork the skill
   or MCP standards when a local convention will do.
2. **One capability surface.** User-facing capabilities come from
   MCP, not from an internal wrapper subsystem or built-in task
   types.
3. **Role-scoped instruction flow.** Skills are not planner-only;
   each role sees only the slice relevant to it.
4. **CLI-managed lifecycle.** Users should not have to hand-edit
   config files to install, test, bootstrap, or inspect MCP
   servers or skills.
5. **Soft migration before hard deletion.** Remove wrappers from
   runtime routing first; delete compatibility shims only after
   the MCP path is feature-complete.
6. **Single key.** The default preset runs entirely on
   `OPENROUTER_API_KEY`. CI enforces that any kiso-maintained
   MCP in the preset needing a backend routes through OpenRouter.
7. **Per-session correctness over daemon-wide shortcuts.** Where
   the standards are session-agnostic and kiso is multi-session,
   kiso's runtime policy resolves per-session. No session bleed.
8. **Honest regressions.** If a feature is lost in the migration,
   it is named explicitly and either ported or documented as
   recovered elsewhere. No silent drop.
9. **No grace periods, no backward compatibility.** Kiso/core is
   pre-release. Every deprecation is a clean cut: no
   "accepted-but-ignored" schema fields, no dual-name env var
   fallbacks, no migration shims that survive past the PR that
   introduced them. If a rename or removal is the right call, it
   lands in one commit without a compatibility path. Existing dev
   configs, dev databases, and dev tests that relied on removed
   behavior get updated in the same commit — or deleted.

## Trust rule for the default preset

The default preset shipped with `kiso init --preset default`
contains **only MCP servers maintained by trustworthy upstreams**:

- **Tier 1**: organizations with public release pipelines and
  verifiable track records (Anthropic `@modelcontextprotocol/*`,
  Microsoft `@playwright/mcp`, GitHub `@github/*`).
- **Tier 2**: `kiso-run/*-mcp` servers maintained by the same
  team that maintains `kiso/core`. Listed at the same trust
  level only because the kiso project itself is the upstream.

Explicitly excluded from the default preset:
- Single-developer experiments, however popular.
- Forks of enterprise servers.
- Servers requiring paid API keys *for basic operation* (free
  tier OK; required paid key not OK in default).
- Community servers from public registries (`mcp.so`,
  `glama.ai`, `smithery.ai`) without org backing.

These can still be **recommended in docs** under
`docs/recommended-mcps.md` with explicit disclaimers about
provenance and maintenance status — but they are not in the
default preset, ever.

Every future PR that modifies `kiso/presets/default.mcp.json` must
demonstrate that the new entry passes this rule. The rule lives
in the `Done when` of M1503 and in the docs page itself.

For **user-installed** MCP servers and skills (outside the
preset), kiso maintains a **trust store** at `~/.kiso/trust.json`
managed via `kiso mcp trust add/remove/list` and `kiso skill
trust add/remove/list` (M1515). Untrusted sources are not
blocked; they install with a loud warning that the user must
acknowledge. This mirrors `pip install` and `npx` UX.

## Kiso Skill Profile

Kiso consumes **standard Agent Skills** from `~/.kiso/skills/`.
The canonical package form is:

```text
~/.kiso/skills/<name>/
├── SKILL.md
├── scripts/...
├── references/...
└── assets/...
```

Single-file skills remain acceptable for lightweight cases, but
directory skills are the default because they match the standard
best and allow bundled assets.

`SKILL.md` remains standard Markdown with YAML frontmatter.

**Standard fields** (defined by `agentskills.io`):
- `name` (required, 1-64 chars lowercase+hyphens)
- `description` (required, 1-1024 chars)
- `license`, `compatibility`, `metadata`, `allowed-tools`
  (optional)

**Kiso extension fields** (optional, backward-compatible):
- `when_to_use` — natural-language activation hint for the briefer
- `audiences` — list of role names that should see this skill
  (subset of `planner`, `worker`, `reviewer`, `messenger`;
  defaults to all roles when absent)
- `activation_hints` — structured pre-filter: `{applies_to:
  ["...", "..."], excludes: ["...", "..."]}` evaluated
  deterministically before the briefer runs. Equivalent to the
  current `applies_to`/`excludes` in recipes.
- `version` — semantic version string

The loader accepts all frontmatter keys permissively. Standard
fields are preserved as-is; Kiso extension fields enable richer
runtime behavior when present.

**Naming convention** (per the standard): lowercase, hyphens
only, 1-64 characters. The loader validates this.

The body may optionally contain role-scoped headings:
`## Planner`, `## Worker`, `## Reviewer`, `## Messenger`.
If those headings are absent, the body defaults to
**planner-only guidance** for backward compatibility with
recipe-like skills.

This gives Kiso a portable default:

- a generic skill from the wider ecosystem still works
- a Kiso-authored skill can be richer without breaking the
  standard
- a migrated recipe (planner-only body) still works unchanged

## Role visibility model

### Briefer

The briefer sees **skill metadata only**, never full skill bodies:

- name
- description
- `when_to_use` when present
- `audiences` summary (for scoping hints)

The briefer returns a **positive selection**:

- `skills: ["python-debug", "research-verify"]`

It no longer returns `exclude_recipes`.

Before the briefer runs, an `activation_hints` pre-filter (M1502)
deterministically removes skills whose `applies_to` does not
match the current message OR whose `excludes` matches. This
keeps briefer prompt size bounded regardless of how many skills
the user has installed.

### Planner

The planner sees:

- briefer-selected skills
- only the `## Planner` section for each selected skill, or the
  full body when no role sections exist
- available MCP methods with their descriptions and input
  schemas (M1513)

This is the main replacement for today's recipe flow.

### Worker / exec translator

The worker sees:

- task detail
- system environment
- preceding task outputs
- only the `## Worker` sections for skills relevant to the
  current task

This replaces today's narrow `Recipe Contracts` section
(`kiso/brain/text_roles.py:449-466`, `_add_section(context_parts,
"Recipe Contracts", recipe_contracts_text)`) with an explicit
role-scoped mechanism. Worker skill sections are injected into
the exec translator prompt under a new `## Skills (worker
guidance)` header, adjacent to the existing `## System
Environment`, `## Workspace Files`, `## Preceding Task
Outputs`, and `## Task` headers.

### Reviewer

The reviewer still treats `expect` as the primary success
criterion, but may also receive:

- only the `## Reviewer` sections for skills relevant to the
  current task

Injected into the reviewer prompt
(`kiso/brain/reviewer.py:250-290`) under a new `## Skills
(reviewer heuristics)` header, after `## Expected Outcome` and
before `## Actual Output`. The section docstring inside the
prompt explicitly frames these as heuristics supporting
`expect`, never overriding it.

### Messenger

Messenger receives only `## Messenger` sections for skills
relevant to the current response. Wired in M1520. Injected into
the messenger prompt alongside `MemoryPack` context, under a
new `## Skills (output style)` header.

## Secret management for MCP

MCP servers receive secrets via two channels, both owned by
Kiso's runtime policy (since the MCP standard defines none):

1. **`env` block in `mcp.json`** — static per-server env vars.
   Supports `${env:VAR}` expansion at parse time (existing
   behavior from `kiso/mcp/config.py:204-221`).
2. **Per-server credential files** at `~/.kiso/mcp/<name>.env`
   with mode `0600`, managed via `kiso mcp env set/unset/show`.
   **Today these files are written by CLI but never read by the
   spawner** — a silent integration gap. M1512 wires the spawner
   to merge these files into the subprocess environment at
   client initialization, closing the gap.

Per-session ephemeral secrets (analog of wrapper
`session_secrets`, `kiso/wrappers.py:88-90`) are handled by
M1512's per-session client pool: when a session contributes a
secret that a specific MCP method declares as required, Kiso
spawns a session-scoped client instance with that secret in its
environment, and the client is evicted on session end.

`OPENROUTER_API_KEY` is the only secret kiso-maintained servers
need. It flows as follows:
- user env → `KISO_LLM_API_KEY` (already set for LLM calls)
- preset `[mcp.aider].env` contains `{OPENROUTER_API_KEY =
  "${env:KISO_LLM_API_KEY}"}` (canonical rename at preset
  install time)
- spawner merges `~/.kiso/mcp/aider.env` on top (user override
  possible)

## Health monitoring for MCP

Kiso applies two tiers of health monitoring:

- **kiso-maintained servers** (`kiso-run/*-mcp`): each exposes a
  `doctor` tool that checks dependencies and credentials. The
  CLI can invoke it via `kiso mcp test <name>` which will
  automatically call `doctor` when the server declares it.
- **Third-party servers**: no `doctor` contract exists. Kiso
  relies on the existing **circuit breaker** in
  `kiso/mcp/manager.py:248-261` (3 failures in 60 seconds →
  server marked unhealthy, retry after cooldown). The MCP client
  also retries once on `MCPTransportError` before failing the
  task.

The `kiso mcp test` command works on both tiers: for
kiso-maintained servers it calls `doctor`; for third-party
servers it sends `initialize` + `tools/list` + shutdown and
reports success/failure (existing behavior at
`cli/mcp.py:386-420`).

To eliminate the cold-start briefer gap (empty
`list_methods_cached_only` until first explicit call), M1518
adds **background catalog warm-up** on daemon startup: an async
task iterates configured servers and primes
`method_cache` non-blockingly. The briefer on a first message
that races the warm-up sees a partial catalog rather than an
empty one, and by the second message the cache is fully warm.

## `needs_install` flow after wrapper removal

Today the planner sets `needs_install=true` to propose a wrapper
installation (msg-only → user approves → exec install → replan).
After v0.10, the same UX applies to both MCP servers and
skills:

- The planner detects that a capability is missing from the
  available MCP methods OR that no installed skill covers the
  current domain.
- It sets `needs_install=["<name-or-url>"]` and emits a `msg`
  task proposing:
  - `kiso mcp install --from-url <url>` (if MCP), or
  - `kiso skill install --from-url <url>` (if skill)
- The user approves.
- The planner re-plans with the new capability available.

The underlying `needs_install` field and the install-approved
routing suppression (M1234) are unchanged. Only the *target* of
the install extends from wrappers to MCP servers AND skills
(M1516).

Since kiso maintains no registry, the planner never invents
URLs — it asks the user for the concrete source. The hard rule at
`kiso/roles/planner.md:74` ("Kiso does NOT maintain a registry of
MCP servers") is preserved verbatim for MCP and mirrored for
skills in M1505.

## Per-session MCP client pool

A critical architectural shift in v0.10. Today `MCPManager`
(`kiso/mcp/manager.py:67-85`) keys its client pool by
`server_name` only, and `${env:VAR}` substitution in
`kiso/mcp/config.py:204-221` happens at parse time. A default
preset entry like `"${WORKSPACE}"` in `args` is therefore baked
into the `MCPServer` dataclass at daemon boot and cannot change
per session — a correctness bug for any multi-session daemon
running a session-scoped server like `@modelcontextprotocol/server-filesystem`.

M1512 reworks the pool:

- Pool key becomes `(server_name, scope_key)` where `scope_key`
  is `"_global"` by default and `session_id` when the server's
  config references any `${session:*}` token or opts into
  per-session via `[mcp.<name>] scope = "session"`.
- `kiso/mcp/config.py` gains `${session:workspace}` and
  `${session:id}` placeholders. These are NOT expanded at parse
  time; they are preserved verbatim and resolved per-call inside
  `MCPManager._get_or_spawn(name, scope)` using the caller's
  session context.
- `MCPManager.call_method(name, method, args, *, session: str
  | None = None)` gains a session parameter. `kiso/worker/mcp.py`
  passes `ctx.session`.
- Idle eviction: a per-session client unused for
  `mcp_session_idle_timeout` seconds (default 1800s, configurable
  in `config.toml`) is gracefully shut down and evicted.
- Pool bound: at most `mcp_max_session_clients_per_server`
  clients per server (default 32, configurable). LRU eviction
  above the bound. Health state is per `(server, scope_key)`.

Servers that do not reference `${session:*}` continue to pool
globally with no behavioral change.

## MCP configuration and install UX

Kiso adopts **standard `mcp.json` semantics** as the user-facing
shape, but keeps a first-class CLI for manipulating it. The
canonical user flows remain:

- `kiso mcp add`
- `kiso mcp install --from-url <url>`
- `kiso mcp test`
- `kiso mcp env set/unset/show`
- `kiso mcp logs`
- `kiso mcp trust add/remove/list` (NEW, M1515)

Manual editing remains possible, but it is not the recommended
path.

Beside the MCP config itself, Kiso maintains local metadata
files under `~/.kiso/mcp/` for:

- provenance (`installed_via`, source URL, trust tier) —
  `~/.kiso/mcp/<name>.provenance.json` (M1515)
- credentials (`<name>.env`, mode 0600) — wired to spawner in
  M1512
- stderr logs (`<name>.err.log`)
- OAuth tokens (`credentials/<name>.json`, mode 0600) — existing
  behavior

These metadata files are **not** part of the MCP standard; they
are purely Kiso runtime policy.

## Skill CLI and install UX

Parallel command surface for skills:

- `kiso skill list` (M1506)
- `kiso skill info <name>` (M1506)
- `kiso skill add <path>` (M1506) — copy a local directory
- `kiso skill install --from-url <url>` (M1514) — download,
  validate, install
- `kiso skill remove <name>` (M1506)
- `kiso skill test <name>` (M1515) — frontmatter + referenced
  file integrity check
- `kiso skill trust add/remove/list` (M1515)

URL forms supported by `kiso skill install --from-url` (M1514):

- `https://github.com/<owner>/<repo>` — clones repo; expects
  either a top-level `SKILL.md` or a `skills/<name>/SKILL.md`
  subdirectory; copies the matching skill dir into
  `~/.kiso/skills/`.
- `https://github.com/<owner>/<repo>/tree/<ref>/path/to/skill`
  — installs the pointed subdirectory.
- `https://raw.githubusercontent.com/.../SKILL.md` — fetches a
  single-file skill; creates `~/.kiso/skills/<derived-name>/SKILL.md`.
- `https://agentskills.io/skills/<slug>` — fetched through the
  agentskills.io public manifest endpoint (if/when available).
  Until then, translates to the backing github.com URL.
- Zip URL (`*.zip`) — downloads, unpacks, expects a directory
  with `SKILL.md` at root or one level deep.
- Bare local path — `kiso skill add` equivalent; kept for
  composability.

Pre-install checks (M1515):
- trust tier lookup against `~/.kiso/trust.json` + hardcoded
  Tier 1 list
- presence of `scripts/` → warning (requires `--yes` or
  interactive confirmation if untrusted)
- `allowed-tools` frontmatter referencing `Bash(*)`, `Write(*)`,
  or network tools → warning
- frontmatter naming validation per the standard
- SKILL.md size sanity check (> 50KB is flagged)

## Phase 2 — Briefer schema convergence (M1502)

### M1502 — Briefer schema + `activation_hints` pre-filter + MCP role filtering

**Why.** Today the briefer JSON output
(`kiso/brain/common.py:1023-1032`) has 8 fields covering 4
categories. After convergence the briefer reasons about two
orthogonal primitives: skills and MCP methods. Its output
schema simplifies accordingly. The fewer categories the briefer
has to disambiguate, the fewer routing errors.

This milestone also introduces **activation_hints pre-filter**,
the deterministic gate that keeps the skill catalog from
exploding the briefer prompt when a user has 50+ skills
installed. It is the scalability mechanism that makes "no
embeddings" viable for real-world ecosystems.

And it folds in the *only* wrapper-side feature worth
preserving: per-user role filtering
(`kiso/wrappers.py:225-250`,
`build_planner_wrapper_list(user_role, user_wrappers, ...)`).
The same filter is applied to the MCP method catalog at
briefer-input assembly time. The other four wrapper-side
features (session/workspace injection, `consumes` routing, arg
alias normalization, healthy reporting) are addressed in Phase 7
(M1512, M1513, M1519) rather than dropped silently.

**Approach.**

- New briefer schema: `skills`, `mcp_methods`, `modules`,
  `context`, `output_indices`, `relevant_tags`,
  `relevant_entities`. Drop `wrappers` and `exclude_recipes`.
- Update `kiso/brain/common.py:1023` (response schema), 1263-1280
  (context pool sections: replace `wrappers` and `recipes` with
  `skills`), 1283-1317 (`format_mcp_catalog` — keep but extend
  per M1513), 1480-1496 (post-filter to drop hallucinated skill
  names), and the consumers in `kiso/brain/planner.py:1259-1289`
  and `1291-1338` (fallback path).
- **New pre-filter**: in `build_planner_messages()`, before the
  context pool is built, run
  `skill_runtime.filter_by_activation_hints(skills, message,
  is_replan)` to eliminate skills whose `activation_hints`
  rule out the current message. Word-boundary tokenization
  matching the recipe rules
  (`kiso/recipe_loader.py:135-183`), so the migrated recipes
  behave identically. This runs regardless of whether the
  briefer is enabled (deterministic, ~1ms).
- **User role filtering on MCP catalog**: add

### M1512 — Per-session MCP client pool + session context + env file integration

**Why.** Three distinct bugs / gaps collapse into one
infrastructure change:

1. `${env:VAR}` substitution is parse-time
   (`kiso/mcp/config.py:9-15`). A preset arg of `"${WORKSPACE}"`
   is baked at daemon boot, so every session shares the same
   workspace — wrong for a multi-session daemon. Claude Code
   runs one process per session, kiso does not.
2. `MCPManager` keys by `server_name` only
   (`kiso/mcp/manager.py:81`). No per-session isolation.
   `call_method(name, method, args)` has no session parameter
   (`kiso/mcp/manager.py:135-167`).
3. `~/.kiso/mcp/<name>.env` credential files are written by the
   CLI (`cli/mcp.py:446-520`) but **never read by the
   spawner** (`kiso/mcp/stdio.py:311-323`). A user setting a key
   via `kiso mcp env set` today gets no subprocess-side effect.
   This is a silent regression hazard.

**Approach.**

- Add `${session:workspace}` and `${session:id}` tokens to
  `kiso/mcp/config.py`. These are NOT expanded at parse time —
  the regex `_ENV_REF_RE` at line 34 is extended to match, but
  the expander preserves them verbatim in the `MCPServer`
  dataclass. A separate function
  `resolve_session_tokens(server, session)` applied at
  `call_method` time does the substitution in a copy of the
  server's args/env.
- Extend `MCPManager` pool to key on `(server_name, scope_key)`:
  - `scope_key = "_global"` when the server config has no
    `${session:*}` tokens AND no declared
    `scope = "session"`.
  - `scope_key = session_id` otherwise.
- `MCPManager.call_method(name, method, args, *, session:
  str | None = None)` — `session` passed by
  `kiso/worker/mcp.py:111-113` (already has `ctx.session`).
- Add session-scoped `list_methods`: same key, separate cache
  entry per scope. `list_methods_cached_only` accepts a scope
  hint.
- Add idle eviction loop: background coroutine in `MCPManager`
  iterates session-scoped clients; any client unused for
  `mcp_session_idle_timeout` seconds (default 1800) is
  gracefully shut down. Runs every 60s.
- Add LRU eviction at `mcp_max_session_clients_per_server`
  bound (default 32).
- **Env file integration**: in the client spawn path
  (`kiso/mcp/stdio.py` `_build_subprocess_env` around line 311),
  read `~/.kiso/mcp/<name>.env` if present, parse KEY=VAL
  lines (mode 0600 check), and merge into the subprocess env
  AFTER the config `env` block (file wins, so the user's `kiso
  mcp env set` takes effect).
- Session secret scoping (parity with wrapper
  `session_secrets`): if a session contributes secrets (rare but
  real; passed from user message via a structured input
  mechanism), they are merged into a session-scoped client's env
  at spawn, NEVER into a global-scope client. The scoping is
  enforced by the pool key: global clients cannot receive
  session secrets by construction.

Config settings added to `kiso/config.py`:
- `mcp_session_idle_timeout` (default 1800, clamp 60-7200)
- `mcp_max_session_clients_per_server` (default 32, clamp 1-256)

**Tasks**:

      `${session:workspace}` and `${session:id}`. Keep them
      un-substituted at parse time; emit them verbatim.
- [ ] Add `resolve_session_tokens(server, session)` in
      `kiso/mcp/config.py` returning a shallow-copied

- [ ] Detect session-scoped servers (presence of

### M1515 — Trust model, safety gates, `kiso skill test`, recommended-skills docs

**Why.** `kiso mcp install` and `kiso skill install` both pull
code from the network. Silent trust-on-first-use is risky for
skills with `scripts/` or MCP servers that can run arbitrary
commands under `uvx`/`npx`. `pip install` and `npx` UX is the
reference: warn loudly, don't block, let the user opt in.
Additionally, skills need a local integrity check (`kiso skill
test`) and the ecosystem page promised in M1503 needs to be
populated.

**Approach.**

{

    "prefixes": [
      "npm:@modelcontextprotocol/*",
      "npm:@playwright/*",
      "npm:@github/*",
      "pypi:kiso-*-mcp"
    ],
    "custom": []
  },
  "skill": {
    "prefixes": [
      "github.com/anthropic/skills/*",
      "github.com/kiso-run/skills/*"
    ],
    "custom": []
  }
}
```

Tier 1 prefixes are hardcoded in `kiso/mcp/trust.py` and
`kiso/skill_trust.py` and cannot be edited via CLI (only via
code change). `custom` is the user-editable list via:

- `kiso mcp trust add <prefix>` / `remove` / `list`
- `kiso skill trust add <prefix>` / `remove` / `list`

Install-time behavior:
- source matches hardcoded Tier 1 prefix OR user-custom prefix
  → silent install
- source does NOT match → warn with:
  - source URL
  - resolved package name
  - detected risk factors: `scripts/` present,
    `allowed-tools` non-trivial, unsigned release, etc.
  - prompt `Install anyway? [y/N]` (unless `--yes`)

`kiso skill test <name>`:
- validate frontmatter strictly (name, description required;
  naming convention)
- check all referenced paths in frontmatter and body exist
  (relative links to `scripts/` / `references/` / `assets/`)
- check role sections parse cleanly
- if `allowed-tools` lists external commands, check they exist
  on PATH (warning only)
- print a structured report; exit non-zero on hard failures

`docs/recommended-skills.md` populated with a starting set:
- 5-10 community skills from well-known repos (anthropics,

- each with: source URL, brief description, risk level, why

`kiso/skill_trust.py::detect_risk_factors(skill_path)`:
- `scripts/` directory present
- frontmatter `allowed-tools` references `Bash(*)`, `Write(*)`,
  `Edit(*)`, or network-adjacent tools
- SKILL.md exceeds 50KB
- referenced assets exceed 5MB total

### M1518 — MCP catalog warm-up + cold-start handling

### M1519 — `x-kiso-consumes`, `mcp_recovery` module, and richer descriptions

**Why.** The wrapper `consumes` field (`kiso/wrappers.py:23`,

**`x-kiso-consumes` extension**: MCP Tool descriptions in the
protocol are free-form strings. Kiso-run/*-mcp servers can

Extract text from an audio file.

<x-kiso: {"consumes": ["audio"]}>
```

mirroring `build_planner_wrapper_list()` lines 289-311.

planner falls back to semantic matching.

`wrapper_recovery`. When an MCP server in the catalog is marked
unhealthy (circuit breaker tripped), the planner prompt gains:

```
- <server> is currently UNHEALTHY. Do not route work to it.

  reported, then `kiso mcp reset-health <server>` (or restart
  the daemon).
```

Requires adding `kiso mcp reset-health <server>` CLI command
that calls `manager.reset_health()` without full daemon
restart. `manager.reset_health()` exists already at
`kiso/mcp/manager.py:248-261` semantically; expose via CLI.

**Richer descriptions in kiso-run/* servers**: update the Track
B server package contracts to populate tool descriptions with
the same level of detail as wrapper `usage_guide`s. This is
documentation, not code — lives in each Track B repo's M1507-M1511
tasks.

### M1526 — MCP stdio sandbox_uid symmetry with exec

**Why.** Today `sandbox_uid` drops privileges for exec
(`kiso/worker/exec.py` via `_run_subprocess(..., uid=...)`)
and wrapper subprocesses
(`kiso/worker/wrapper.py`) when the caller has `role="user"`
(per `kiso/worker/loop.py:1696-1697`). MCP stdio subprocesses
bypass this: `kiso/mcp/stdio.py:311-323` spawns as the kiso
process user, no `user=uid` in `asyncio.create_subprocess_exec`.

With wrappers retired, MCP becomes the only subprocess capability
surface besides exec. Leaving MCP unsandboxed for user-role
sessions is an asymmetry that breaks the security model of
`role=user`: a restricted user could route their request through
an untrusted MCP server and execute arbitrary code with the
daemon's privileges.

Closing this symmetry is part of the pre-release hardening.
Pairs naturally with the per-session client pool (M1512) because
sandbox UID is session-bound anyway.

**Approach.**

- Extend `MCPStdioClient.__init__` to accept
  `sandbox_uid: int | None`.
- Pass through to
  `asyncio.create_subprocess_exec(..., user=sandbox_uid)` in the
  spawn call.
- Client pool key becomes `(server_name, scope_key,
  sandbox_uid)`. Distinct UIDs get distinct clients. Admin calls
  (`sandbox_uid=None`) pool separately from user calls.
- Pre-spawn: `_session_workspace(session, sandbox_uid=uid)`
  sets ownership so the subprocess can read/write the workspace
  (existing behavior; confirm MCP spawn uses the same path).
- Config opt-out per-server: `[mcp.<name>] sandbox = "never"`
  for servers that genuinely need higher privilege (e.g., a
  `systemd-mcp` that controls system services). Default is
  "respect role" — honor `sandbox_uid` when the calling role is
  `user`.
- CLI calls (`kiso mcp test`) run without UID drop (admin
  context) — no change.

**Tasks**:
- [ ] Add `sandbox_uid` parameter to `MCPStdioClient`.
- [ ] Wire through in `kiso/mcp/manager.py::_get_or_spawn`
      from the session context.
- [ ] Extend pool key to include `sandbox_uid`.
- [ ] Add `sandbox` field to `MCPServer` dataclass
      (default `"role_based"`, also accepts `"never"`).
- [ ] Ensure `_session_workspace()` ownership is correctly
      set before MCP spawn.
- [ ] Unit test: a user-role session spawns a stdio MCP
      subprocess with its dedicated UID; file writes in the
      workspace are owned by that UID; attempts to touch files
      outside the workspace fail.
- [ ] Document the sandbox model in `docs/security.md`
      (covered by M1523 docs sweep).
- [ ] Commit & push.

**Done when**:
- A user-role request exercising an MCP server runs the
  subprocess under the session's sandbox UID.
- Admin-role requests and `kiso mcp test` continue to run
  without UID drop.
- Pool correctly isolates per-UID, per-session, per-server.

### M1527 — MCP Resources support (list, read, briefer visibility, planner routing)

**Why.** The MCP protocol defines three primitives: Tools,
Resources, Prompts. Today kiso consumes Tools only
(`kiso/mcp/stdio.py:151-182`, `http.py:155-184` issue
`tools/list`). Resources are static or dynamic file-like
objects a server exposes — log files, database rows, doc
pages — that kiso should be able to read and hand off to the
planner/worker as context. Without Resources, servers that
expose data (rather than actions) are invisible to kiso.

This is the kind of feature that distinguishes a shallow MCP
client from a production one. Pre-launch = must-have.

**Approach.**

- Extend `MCPClient` interface with `list_resources()` and
  `read_resource(uri)`. Parallel to `list_methods()` /
  `call_method()` but semantically distinct: resources are
  data, not functions.
- `MCPManager` caches the resource catalog (60s TTL parity with
  methods). Background warm-up (M1518) extended to also prime
  the resource cache.
- `kiso/brain/common.py` gains `format_mcp_resources(manager)`
  rendering a `## Available Resources` section, bulleted:
  `- <server>:<uri> — <description>`. Included in the planner
  context pool parallel to `mcp_methods`.
- Planner task taxonomy does NOT grow a new type. Resource reads
  happen via a distinguished MCP call shape:
  `{type: "mcp", server: "...", method: "__resource_read",
  args: {uri: "..."}}`. The `__resource_read` method name is
  special-cased in `kiso/worker/mcp.py::_handle_mcp_task` to
  call `read_resource` instead of `call_method`. This keeps the
  task schema stable (four types: exec/mcp/msg/replan) and
  avoids inventing a fifth.
- Result rendering: `render_mcp_resource_result(uri, blob)`
  parallel to `render_mcp_result()` — resources with MIME types
  are written to `pub/` (images, PDFs) or inlined in stdout
  (text).
- Briefer post-filter: resource URIs go through
  `_filter_briefer_names()` the same way methods do.
- Planner prompt (`skills_and_mcp` module) gains a section
  explaining how to route reads through the
  `__resource_read` pseudo-method.

**Tasks**:
- [ ] Extend `MCPClient` base class with
      `list_resources()` + `read_resource(uri)` abstract
      methods.
- [ ] Implement in `kiso/mcp/stdio.py` (uses `resources/list`,
      `resources/read` protocol methods).
- [ ] Implement in `kiso/mcp/http.py` (same protocol methods,
      Streamable HTTP transport).
- [ ] `MCPManager` adds `list_resources_cached_only`,
      `list_resources`, `read_resource`. Extend warm-up
      (M1518) to also warm resources.
- [ ] `MCPResource` dataclass in `kiso/mcp/schemas.py`: `uri`,
      `name`, `description`, `mime_type`.
- [ ] `format_mcp_resources()` in `kiso/brain/common.py`.
- [ ] Extend briefer schema: new `mcp_resources: list[str]`
      field. Positive selection, URI form.
- [ ] Thread through
      `build_planner_messages()` → context pool.
- [ ] Special-case `__resource_read` in
      `kiso/worker/mcp.py::_handle_mcp_task`.
- [ ] `render_mcp_resource_result()` in `kiso/mcp/result.py`.
- [ ] `validate_plan()`: allow `method = "__resource_read"`
      only when `args = {uri: str}`.
- [ ] Update `skills_and_mcp` planner module to document

- [ ] Unit tests: stdio + http clients list resources;

- [ ] Commit & push.

### M1529 — MCP `sampling/createMessage` support

**Why.** The MCP standard allows a server to request an LLM
completion FROM the client via `sampling/createMessage`. Servers
that cannot (or should not) carry their own LLM credentials use
this to delegate LLM calls to the client, which already has
credentials + model routing. Example: a `brainstorm-mcp` server
exposes a `brainstorm(topic)` tool that needs an LLM, doesn't
want to carry an `OPENROUTER_API_KEY`, and asks kiso to do the
completion on its behalf.

Pre-launch, this is the last MCP surface we don't implement. Not
supporting it is a compatibility gap with any server that uses
it. Leaving it out would force server authors to bundle
credentials, weakening the single-key philosophy across the
ecosystem.

**Approach.**

- In the stdio and http client message loops, handle incoming
  `sampling/createMessage` requests (server → client). Today
  kiso's client is strictly request-initiator; this makes it
  bidirectional for a specific request type.
- On receipt, construct an LLM call via the existing
  `kiso.brain.common.call_llm` infrastructure using a
  dedicated role, e.g., `mcp_sampling`, added to
  `_MODEL_METADATA` in `kiso/config.py` with a sensible default
  (e.g., `google/gemini-2.5-flash`).
- Response-shape: honor the server's requested `systemPrompt`,
  `messages`, `maxTokens`, `temperature` per spec, but clamp
  `maxTokens` to a policy ceiling (e.g., 4096) for safety.
- Policy gate: config setting
  `mcp_sampling_enabled` (default `true`). When `false`, the
  client replies with the spec's `method not supported` error.
  Admins can disable if they don't want their OpenRouter
  budget spent by MCP servers.
- Budget accounting: treat sampling calls as billable LLM
  calls; they count toward `max_llm_calls_per_message`
  (existing setting).

**Tasks**:
- [ ] Add `mcp_sampling` role to `_MODEL_METADATA` with a
      sensible default.
- [ ] Add `mcp_sampling_enabled` setting
      (default `true`, clamp to bool).
- [ ] In `kiso/mcp/stdio.py` message read loop, dispatch
      incoming requests by method. For
      `sampling/createMessage`, call a new
      `kiso.mcp.sampling.handle_sampling_request(config, req)`
      and write the JSON-RPC response.
- [ ] Same dispatch in `kiso/mcp/http.py` (Streamable HTTP
      server-to-client is via SSE events coming IN; handle
      these inside the existing event pump).
- [ ] `handle_sampling_request`: build an LLM call using the
      `mcp_sampling` role, apply policy clamps, return the
      spec-shaped response.
- [ ] Count against `max_llm_calls_per_message`.
- [ ] Unit tests with a fake server that emits a
      `sampling/createMessage` request; verify kiso replies
      with a correctly-shaped response.
- [ ] Integration test: a fixture MCP server that uses
      sampling internally to implement a tool; kiso
      successfully completes the call.
- [ ] When `mcp_sampling_enabled = false`, the client responds
      with `method not supported`.
- [ ] Commit & push.

**Done when**:
- A fixture MCP server requesting `sampling/createMessage`
  receives a valid LLM response from kiso.
- Disabling the setting cleanly refuses the request.
- Sampling calls count against the per-message LLM call
  budget.
