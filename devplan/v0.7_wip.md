### M661 ÔÇö Fix worker/reviewer `|| true` contradiction

**Problem:** Worker.md tells the exec translator to append `|| true` to verification
commands ("ensure exit 0"). But Reviewer.md has a specific rule: "exit 1 + empty =
'nothing found' = valid ok". These contradict each other.

The `|| true` wrapper masks the exit code Ôćĺ the reviewer never sees exit 1 Ôćĺ can't
apply its "nothing found = ok" rule. Verification commands that find nothing appear
to succeed (exit 0 + empty output), and the reviewer can't distinguish "found nothing"
from "ran successfully with no output".

This caused the live test `test_replan_after_failed_exec` to be flaky (M614) ÔÇö the
`ls /nonexistent` command with `|| true` returned exit 0, and the reviewer sometimes
said "ok" instead of "replan".

**Files:** `kiso/roles/worker.md`

**Change:**

- [ ] Remove "ensure exit 0 (append `|| true`)" from verification task rules
- [ ] Replace with: "Verification: may exit 0 or 1. Exit 1 + empty output = 'not found'
      (handled by reviewer). Do NOT append `|| true` to verification commands."
- [ ] Keep "Action tasks: ensure exit 0" for non-verification commands
- [ ] Existing unit tests pass (prompt text tests may need updating)

### M666 ÔÇö `_call_role()` stallÔćĺfallback for all simple roles

**Problem:** `run_messenger` calls `call_llm` directly via `_call_role()` (M656).
When the SSE stream stalls, `LLMStallError` propagates up uncaught ÔÇö the messenger
has no retry, no fallback. M652 only protects roles that use
`_retry_llm_with_validation` (planner, reviewer, curator, briefer).

This affects ALL simple roles: messenger, summarizer, paraphraser, searcher,
exec_translator. None of them have stall protection.

**Files:** `kiso/brain.py` (`_call_role`)

**Change:**

- [ ] Add `fallback_model` parameter to `_call_role()`:
      ```python
      async def _call_role(config, role, messages, error_class, session="",
                           fallback_model=None):
          try:
              return await call_llm(config, role, messages, session=session)
          except LLMStallError:
              if fallback_model:
                  log.warning("SSE stall on %s, trying fallback %s", role, fallback_model)
                  return await call_llm(config, role, messages, session=session,
                                        model_override=fallback_model)
              raise error_class(f"LLM stall with no fallback")
          except LLMError as e:
              raise error_class(f"LLM call failed: {e}")
      ```
- [ ] Update `run_messenger` to pass `fallback_model` (e.g., gemini-2.5-flash-lite)
- [ ] Update `run_exec_translator` to pass `fallback_model` (same)
- [ ] Other simple roles (summarizer, paraphraser, searcher): optional, lower priority
- [ ] Test: mock LLMStallError on first call Ôćĺ verify second call uses fallback model
- [ ] Test: no fallback Ôćĺ raises error_class immediately

---

### M667 ÔÇö Briefer max_tokens: 500 Ôćĺ 2048

**Problem:** The briefer has `max_tokens: 500` in `MAX_TOKENS_DEFAULTS`. The briefer
must return a JSON with 6 fields, including `context` (a free-form string of compressed
facts). For complex queries (F7: research + write file + publish), the context pool
is large and the briefer tries to compress a lot of information into `context` ÔÇö the
response exceeds 500 tokens and gets **truncated mid-JSON string**.

Result: `"Unterminated string starting at: line 6 column 5"` ÔÇö the JSON is broken
because the model ran out of tokens before closing the string. Retrying produces the
same truncation (same input + same limit = same output length).

F7 logs: briefer failed 6 times (3 retries ├Ś 2 calls), wasting ~180s, before falling
back to full context.

**Root cause:** 500 tokens Ôëł 375 words. A briefer JSON with `context` containing 3
facts + modules + tools + tags + entities easily exceeds this for complex queries.

**Files:** `kiso/config.py` (`MAX_TOKENS_DEFAULTS`)

**Change:**

- [ ] Change `MAX_TOKENS_DEFAULTS["briefer"]` from `500` to `2048`
- [ ] The briefer model (gemini-2.5-flash-lite) is fast and cheap ÔÇö 2048 tokens
      costs fractions of a cent and adds negligible latency
- [ ] Existing briefer tests pass unchanged (they mock call_llm, don't hit token limits)

### M670 ÔÇö Broaden install_proposal detection for msg-only plans on fresh instances

**Problem:** F1 browser install test is flaky. When the planner produces a msg-only plan
proposing a tool install, `install_proposal` depends on two signals:

1. `plan.get("needs_install")` ÔÇö requires the LLM to populate the field (unreliable)
2. `saw_uninstalled_tool` ÔÇö requires the planner to have first attempted a tool task
   that failed validation (doesn't happen when planner goes straight to msg-only)

When both signals miss (LLM sets `needs_install: null` and skips the tool task),
`install_proposal = False` Ôćĺ next turn has `install_approved = False` Ôćĺ install exec
is blocked by validation Ôćĺ browser never gets installed.

This is a real UX bug: user approves install, system says "you can't install yet".

**Root cause:** The detection in `run_planner` (line ~1204) doesn't cover the case where
the planner directly produces a valid msg-only plan without going through the
uninstalled-tool validation error path.

**Files:** `kiso/brain.py` (`run_planner`, around line 1204)

**Change:**

- [x] Broaden the `install_proposal` detection with a third condition
- [x] No impact when tools are installed ÔÇö `installed_names` is non-empty, condition never fires
- [x] Zero latency cost ÔÇö no validation retry, just a broader post-hoc check
- [x] 4 unit tests: msg-only+no-toolsÔćĺTrue, msg-only+toolsÔćĺFalse, explicit needs_installÔćĺTrue, exec+msgÔćĺFalse

## Phase 56: Task Execution Improvements ÔÇö Parallel Groups + Dynamic Replan Limit

Two orthogonal improvements to plan execution. Parallel groups enable scatter-gather
workflows (critical for marketing/growth presets: multi-source research, competitor
analysis, batch processing). Dynamic replan limit forces focus at deep replan depths.

**Why now:** Phase 55 presets create personas (performance marketer, SEO specialist)
whose core workflows are multi-source fan-out: search 5 competitors, fetch 3 data

need to be ÔÇö the difference between "usable" and "painful" for batch workflows.

**Reference:** 14 of 34 documented OpenClaw use cases are "highly parallelizable"
with the same scatter-gather pattern. OpenClaw itself is sequential per session ÔÇö
this would be a concrete architectural advantage.
