### M661 ‚Äî Fix worker/reviewer `|| true` contradiction ‚úÖ

**Problem:** Worker.md tells the exec translator to append `|| true` to verification
commands ("ensure exit 0"). But Reviewer.md has a specific rule: "exit 1 + empty =
'nothing found' = valid ok". These contradict each other.

The `|| true` wrapper masks the exit code ‚Üí the reviewer never sees exit 1 ‚Üí can't
apply its "nothing found = ok" rule. Verification commands that find nothing appear
to succeed (exit 0 + empty output), and the reviewer can't distinguish "found nothing"
from "ran successfully with no output".

This caused the live test `test_replan_after_failed_exec` to be flaky (M614) ‚Äî the
`ls /nonexistent` command with `|| true` returned exit 0, and the reviewer sometimes
said "ok" instead of "replan".

**Files:** `kiso/roles/worker.md`

**Change:**

- [ ] Remove "ensure exit 0 (append `|| true`)" from verification task rules
- [ ] Replace with: "Verification: may exit 0 or 1. Exit 1 + empty output = 'not found'
      (handled by reviewer). Do NOT append `|| true` to verification commands."
- [ ] Keep "Action tasks: ensure exit 0" for non-verification commands
- [ ] Existing unit tests pass (prompt text tests may need updating)

### M666 ‚Äî `_call_role()` stall‚Üífallback for all simple roles ‚úÖ

**Problem:** `run_messenger` calls `call_llm` directly via `_call_role()` (M656).
When the SSE stream stalls, `LLMStallError` propagates up uncaught ‚Äî the messenger
has no retry, no fallback. M652 only protects roles that use
`_retry_llm_with_validation` (planner, reviewer, curator, briefer).

This affects ALL simple roles: messenger, summarizer, paraphraser, searcher,
exec_translator. None of them have stall protection.

**Files:** `kiso/brain.py` (`_call_role`)

**Change:**

- [ ] Add `fallback_model` parameter to `_call_role()`:
      ```python
      async def _call_role(config, role, messages, error_class, session="",
                           fallback_model=None):
          try:
              return await call_llm(config, role, messages, session=session)
          except LLMStallError:
              if fallback_model:
                  log.warning("SSE stall on %s, trying fallback %s", role, fallback_model)
                  return await call_llm(config, role, messages, session=session,
                                        model_override=fallback_model)
              raise error_class(f"LLM stall with no fallback")
          except LLMError as e:
              raise error_class(f"LLM call failed: {e}")
      ```
- [ ] Update `run_messenger` to pass `fallback_model` (e.g., gemini-2.5-flash-lite)
- [ ] Update `run_exec_translator` to pass `fallback_model` (same)
- [ ] Other simple roles (summarizer, paraphraser, searcher): optional, lower priority
- [ ] Test: mock LLMStallError on first call ‚Üí verify second call uses fallback model
- [ ] Test: no fallback ‚Üí raises error_class immediately

---

### M667 ‚Äî Briefer max_tokens: 500 ‚Üí 2048 ‚úÖ

**Problem:** The briefer has `max_tokens: 500` in `MAX_TOKENS_DEFAULTS`. The briefer
must return a JSON with 6 fields, including `context` (a free-form string of compressed
facts). For complex queries (F7: research + write file + publish), the context pool
is large and the briefer tries to compress a lot of information into `context` ‚Äî the
response exceeds 500 tokens and gets **truncated mid-JSON string**.

Result: `"Unterminated string starting at: line 6 column 5"` ‚Äî the JSON is broken
because the model ran out of tokens before closing the string. Retrying produces the
same truncation (same input + same limit = same output length).

F7 logs: briefer failed 6 times (3 retries √ó 2 calls), wasting ~180s, before falling
back to full context.

**Root cause:** 500 tokens ‚âà 375 words. A briefer JSON with `context` containing 3
facts + modules + tools + tags + entities easily exceeds this for complex queries.

**Files:** `kiso/config.py` (`MAX_TOKENS_DEFAULTS`)

**Change:**

- [ ] Change `MAX_TOKENS_DEFAULTS["briefer"]` from `500` to `2048`
- [ ] The briefer model (gemini-2.5-flash-lite) is fast and cheap ‚Äî 2048 tokens
      costs fractions of a cent and adds negligible latency
- [ ] Existing briefer tests pass unchanged (they mock call_llm, don't hit token limits)

### M670 ‚Äî Broaden install_proposal detection for msg-only plans on fresh instances ‚úÖ

**Problem:** F1 browser install test is flaky. When the planner produces a msg-only plan
proposing a tool install, `install_proposal` depends on two signals:

1. `plan.get("needs_install")` ‚Äî requires the LLM to populate the field (unreliable)
2. `saw_uninstalled_tool` ‚Äî requires the planner to have first attempted a tool task
   that failed validation (doesn't happen when planner goes straight to msg-only)

When both signals miss (LLM sets `needs_install: null` and skips the tool task),
`install_proposal = False` ‚Üí next turn has `install_approved = False` ‚Üí install exec
is blocked by validation ‚Üí browser never gets installed.

This is a real UX bug: user approves install, system says "you can't install yet".

**Root cause:** The detection in `run_planner` (line ~1204) doesn't cover the case where
the planner directly produces a valid msg-only plan without going through the
uninstalled-tool validation error path.

**Files:** `kiso/brain.py` (`run_planner`, around line 1204)

**Change:**

- [x] Broaden the `install_proposal` detection with a third condition
- [x] No impact when tools are installed ‚Äî `installed_names` is non-empty, condition never fires
- [x] Zero latency cost ‚Äî no validation retry, just a broader post-hoc check
- [x] 4 unit tests: msg-only+no-tools‚ÜíTrue, msg-only+tools‚ÜíFalse, explicit needs_install‚ÜíTrue, exec+msg‚ÜíFalse

## Phase 56: Task Execution Improvements ‚Äî Parallel Groups + Dynamic Replan Limit

Two orthogonal improvements to plan execution. Parallel groups enable scatter-gather
workflows (critical for marketing/growth presets: multi-source research, competitor
analysis, batch processing). Dynamic replan limit forces focus at deep replan depths.

**Why now:** Phase 55 presets create personas (performance marketer, SEO specialist)
whose core workflows are multi-source fan-out: search 5 competitors, fetch 3 data
sources, analyze N pages. Without parallelism these plans are 5√ó slower than they
need to be ‚Äî the difference between "usable" and "painful" for batch workflows.

**Reference:** 14 of 34 documented OpenClaw use cases are "highly parallelizable"
with the same scatter-gather pattern. OpenClaw itself is sequential per session ‚Äî
this would be a concrete architectural advantage.

      ```python
          Batch with 1 item = sequential. Batch with 2+ items = parallel.
      ```

proposal exists, and we're processing a new message. It doesn't fabricate user

## Phase 52 ‚Äî Taxonomy Rename: MD skills ‚Üí recipes, tools ‚Üí skills

**Motivation:** Aligning with the AgentSkills ecosystem (Claude Code, OpenClaw, Nanobot, Codex, Copilot, Cursor, Gemini CLI). The spec defines "skill" as an installable capability package (SKILL.md + scripts/ + assets/), which maps to our current "tools" (browser, aider, websearch). Our "MD skills" are a distinct concept ‚Äî pure planner instructions without executable code ‚Äî better described as "recipes" (a set of instructions to produce a specific result).

### M828 ‚Äî Propagate `KISO_LLM_API_KEY` to tool subprocesses ‚úÖ

**Problem:** `build_tool_env()` in `kiso/tools.py` constructs a minimal env for tool

aider-specific; any tool that calls an LLM needs a key, and forcing every user

providers is bad UX.

### M829 ‚Äî Artifact-mismatch validation: skip on replan ‚úÖ

**Problem:** `validate_plan()` (brain.py:869-881) rejects plans where the goal
contains artifact verbs+nouns (e.g., "create report") but the plan has no
exec/tool task. This rule is correct for first plans ‚Äî it prevents the planner
from embedding file content in msg tasks.

explain *why* the previous attempt failed (e.g., "the directory doesn't exist").

the best course is to inform the user, that's valid. The artifact-mismatch
rule should only apply to first plans where the planner never tried. This also
applies to any future scenario where the planner responds to a failure by
proposing a communication-only plan.

## Phase 57 ‚Äî Planner context reliability

Results from full test run after Phase 53-56: 6 previously failing tests now
pass (M828, M829, M830, M833, M841 confirmed). 5 remaining failures analyzed.

### Root cause analysis

Three recurring planner failures share the SAME root cause:
1. Planner uses `pip install` instead of `uv pip install`
2. Planner investigates (exec+replan) instead of setting `needs_install`
3. Planner treats kiso tools as system packages

**Root cause:** The briefer is too aggressive in filtering planner modules.
When `briefer_enabled=True` (default, and used in live/functional tests), the
briefer prompt says "Most requests need ZERO modules" and "AGGRESSIVE filtering.
Default to EXCLUDING." For install requests like "installa flask", the briefer
returns `modules: []`.

The planner then receives NO `kiso_native` module ‚Äî which contains ALL the
rules about `uv pip install`, `registry_hints`, `needs_install`, and the
decision tree for system packages vs Python libs vs kiso tools.

**Why unit tests pass:** They use `briefer_enabled=False`, which triggers the
fallback path (brain.py:1148-1167) that includes `kiso_native` by default.

**Why the fix is NOT "make the briefer smarter":** Relying on an LLM to
correctly select a module that contains safety constraints is fragile by
design. The constraint "never use bare pip install" is not contextual ‚Äî it
ALWAYS applies. Rules that always apply belong in the core prompt, not in
optional modules.

**The code gap:** brain.py:1138-1147 has a safety net that forces
`plugin_install` into the briefer path when no tools are installed, but does
NOT force `kiso_native`. The kiso_native module was treated as optional
context, but it contains mandatory system constraints.

### M878 ‚Äî Invalidate tools cache before replan planner call ‚úÖ

**Problem (kiso behavior):** When a plan installs a tool and then replans,
the replan's `run_planner()` call hits the 30-second TTL cache in
`discover_tools()` and sees the OLD tool list. The planner doesn't know the
tool was just installed ‚Üí generates exec tasks instead of tool tasks.

This is the core reason F17 fails: after installing OCR, the replan planner
still sees the old list without OCR ‚Üí generates `exec: Run Tesseract OCR`
instead of `type="tool", tool="ocr"`. Same for aider.

**Evidence:** In `_execute_plan()` (loop.py:1641/1692), the cache IS
refreshed after each task execution. But in the replan loop (loop.py:2343),
`run_planner()` is called WITHOUT invalidating the cache first.

**Principle:** Every planning decision must see the current state. The TTL
cache is fine for performance within a single plan execution (tasks happen
fast). But between plans (replan, new message), the state may have changed
(tools installed, config reloaded). Cache must be invalidated at plan
boundaries.

**Fix:** Call `invalidate_tools_cache()` before `run_planner()` in:
1. The replan loop (loop.py ~line 2340)
2. The initial planner call (loop.py ~line 2560) ‚Äî already fresh, but
   explicit invalidation prevents future bugs

This is one line of code. The impact is: every planner call sees fresh tool
state. The TTL cache still helps within `_execute_plan` (multiple
discover_tools calls in the same plan execution stay cached).

**Files:** `kiso/worker/loop.py`, `tests/test_worker.py`

**Tasks:**
- [ ] Add `invalidate_tools_cache()` before `run_planner()` in replan loop
- [ ] Add `invalidate_tools_cache()` before initial `run_planner()` call
- [ ] Unit test: verify that replan path invalidates cache

### M880 ‚Äî Compress install turns in replan context ‚úÖ

**Problem:** After the install-approve-install cycle (3-5 messages), the
planner's recent message context is full of install noise. When the planner
finally needs to USE the tool, it's confused by the install history and
generates exec tasks instead of tool tasks.

This is why F17 fails: after 3 install cycles (browser, ocr, aider), the
context has 15+ install-related messages. The planner can't distinguish
between "install this tool" and "use this tool" because both are in
recent history.

F30 passes because tools are pre-installed ‚Äî zero install noise in context.

**Fix: structured compression of install turns.**

In `build_recent_context()` or in the planner message building, detect
consecutive install-related messages (propose ‚Üí approve ‚Üí install result)
and compress them into a single structured line:

Before (5 messages):
```
[user] Use aider to write word_count.py
[kiso] Vuoi installare aider?
[user] s√¨, installa il tool aider
[kiso] Tool aider installato. Replan...
[user] Use aider to write word_count.py  ‚Üê repeated by helper
```

After (2 lines):
```
[install completed] aider is now available.
[user] Use aider to write word_count.py
```

**Implementation approach:**

In `build_recent_context()` (worker/utils.py), add a compression pass:
1. Scan messages for install patterns (kiso msg with "install" keywords +
   user reply with approval keywords + kiso msg with install result)
2. Replace the group with a single "[install completed] {tool} available."
   line + keep the original user request
3. Non-install messages pass through unchanged

This is deterministic (pattern matching on message content), generalizable
(works for any tool install, not just F17), and doesn't depend on LLM.

**Alternative considered:** Have the briefer filter install messages. Rejected
because the briefer is an LLM call (adds latency + flakiness) and we need
deterministic behavior here.

**Files:** `kiso/worker/utils.py`, `tests/test_worker.py`

3. Replan notifications use English only (system messages, user understands)
4. Delete `l10n.py`, `detect_user_lang`, `LANG_NAMES`, `LANG_MARKERS`,
   `REPLAN_TEMPLATES`

### M886 ‚Äî Test runner: compact failure summary for LLM paste ‚úÖ

**Problem:** When tests fail, `run_tests.sh` says "Some suites failed ‚Äî
check output above." and you have to scroll through hundreds of lines of
passing tests, Docker build output, and pytest boilerplate to find the
failures. Pasting the full output into an LLM wastes context on noise.

**Goal:** At the end of a run with failures, print a compact debug block
that contains ONLY:
1. One-line-per-failure summary (test name + short error)
2. For each failure: assertion error + captured log (classifier output,
   LLM calls, briefer decisions ‚Äî the INFO lines that matter)
3. No passing tests, no Docker build output, no traceback frames from
   stdlib/dependencies (only the final assertion + app frames)

The block should be copy-pasteable as-is into an LLM context.

**Approach:** Capture pytest stdout/stderr to a temp file via `tee` while
still showing output live. On failure, parse the captured output:

1. Extract the `=== FAILURES ===` section (pytest groups each failure with
   its captured log/stderr)
2. For each failure block:
   - Extract test name from the `___ TestName ___` header
   - Extract the assertion line (last `E   ...` lines or `AssertionError`)
   - Extract `Captured log call` section (contains classifier, briefer,
     planner INFO lines ‚Äî the actual debug data)
   - Extract `Captured stderr call` section (contains same INFO lines
     when running in Docker)
   - Skip stdlib traceback frames (only keep frames from `kiso/` or
     `tests/`)
3. Compress: truncate captured log to last 30 lines (most recent LLM
   calls are the relevant ones)
4. Print the assembled summary with a clear delimiter:
   ```
   ‚îÑ‚îÑ‚îÑ FAILURE SUMMARY (paste into LLM) ‚îÑ‚îÑ‚îÑ
   ## test_russian_response_quality FAILED
   Error: TimeoutError (120s)
   Log:
   ```

## Phase 70 ‚Äî Production diagnostic logging

### M934 ‚Äî Eliminate msg-first plans (prevent announce hallucination) ‚úÖ

**Problem:** The planning_rules module (planner.md L44) says:
"Multi-step plans: insert intermediate msg tasks every 4‚Äì5 tasks."

The planner interprets this as: put a msg as the FIRST task to
announce upcoming steps. The detail becomes:
"Briefly tell the user what the system is about to do. Steps:
tool: Navigate to..., tool: Take a screenshot..."

The messenger receives this and faces an irreconcilable conflict:
- messenger.md L8: "Never describe what you WILL do next"
- messenger.md L13: "Never fabricate information not in task outputs"
- messenger.md L15: "Never claim actions unsupported by task outputs"

But the task detail asks it to describe future actions. The model
resolves the conflict by hallucinating that steps already completed
‚Äî inventing file paths (sandbox:/mnt/data/screenshot.png), fake
downloads, and non-existent URLs. This happens deterministically:
ANY model would fail because the prompt is self-contradictory.

**Approach:** Rewrite planning_rules L44-45 to make the rule
unambiguous:

```
- The first task must be exec, tool, search, or replan ‚Äî never msg.
  Intermediate msg tasks report results from already-completed
  tasks only. Never describe upcoming steps or announce the plan.
  For plans with 8+ tasks: one intermediate msg after every 5
  completed exec/tool/search tasks. Shorter plans need only the
  final msg.
```

This preserves intermediate msgs for long plans (webhook/API users
need progress updates) while eliminating the announce pattern.
