### M661 ÔÇö Fix worker/reviewer `|| true` contradiction Ôťů

**Problem:** Worker.md tells the exec translator to append `|| true` to verification
commands ("ensure exit 0"). But Reviewer.md has a specific rule: "exit 1 + empty =
'nothing found' = valid ok". These contradict each other.

The `|| true` wrapper masks the exit code Ôćĺ the reviewer never sees exit 1 Ôćĺ can't
apply its "nothing found = ok" rule. Verification commands that find nothing appear
to succeed (exit 0 + empty output), and the reviewer can't distinguish "found nothing"
from "ran successfully with no output".

This caused the live test `test_replan_after_failed_exec` to be flaky (M614) ÔÇö the
`ls /nonexistent` command with `|| true` returned exit 0, and the reviewer sometimes
said "ok" instead of "replan".

**Files:** `kiso/roles/worker.md`

**Change:**

- [ ] Remove "ensure exit 0 (append `|| true`)" from verification task rules
- [ ] Replace with: "Verification: may exit 0 or 1. Exit 1 + empty output = 'not found'
      (handled by reviewer). Do NOT append `|| true` to verification commands."
- [ ] Keep "Action tasks: ensure exit 0" for non-verification commands
- [ ] Existing unit tests pass (prompt text tests may need updating)

### M666 ÔÇö `_call_role()` stallÔćĺfallback for all simple roles Ôťů

**Problem:** `run_messenger` calls `call_llm` directly via `_call_role()` (M656).
When the SSE stream stalls, `LLMStallError` propagates up uncaught ÔÇö the messenger
has no retry, no fallback. M652 only protects roles that use
`_retry_llm_with_validation` (planner, reviewer, curator, briefer).

This affects ALL simple roles: messenger, summarizer, paraphraser, searcher,
exec_translator. None of them have stall protection.

**Files:** `kiso/brain.py` (`_call_role`)

**Change:**

- [ ] Add `fallback_model` parameter to `_call_role()`:
      ```python
      async def _call_role(config, role, messages, error_class, session="",
                           fallback_model=None):
          try:
              return await call_llm(config, role, messages, session=session)
          except LLMStallError:
              if fallback_model:
                  log.warning("SSE stall on %s, trying fallback %s", role, fallback_model)
                  return await call_llm(config, role, messages, session=session,
                                        model_override=fallback_model)
              raise error_class(f"LLM stall with no fallback")
          except LLMError as e:
              raise error_class(f"LLM call failed: {e}")
      ```
- [ ] Update `run_messenger` to pass `fallback_model` (e.g., gemini-2.5-flash-lite)
- [ ] Update `run_exec_translator` to pass `fallback_model` (same)
- [ ] Other simple roles (summarizer, paraphraser, searcher): optional, lower priority
- [ ] Test: mock LLMStallError on first call Ôćĺ verify second call uses fallback model
- [ ] Test: no fallback Ôćĺ raises error_class immediately

### M667 ÔÇö Briefer max_tokens: 500 Ôćĺ 2048 Ôťů

**Problem:** The briefer has `max_tokens: 500` in `MAX_TOKENS_DEFAULTS`. The briefer
must return a JSON with 6 fields, including `context` (a free-form string of compressed
facts). For complex queries (F7: research + write file + publish), the context pool
is large and the briefer tries to compress a lot of information into `context` ÔÇö the
response exceeds 500 tokens and gets **truncated mid-JSON string**.

Result: `"Unterminated string starting at: line 6 column 5"` ÔÇö the JSON is broken
because the model ran out of tokens before closing the string. Retrying produces the
same truncation (same input + same limit = same output length).

F7 logs: briefer failed 6 times (3 retries ├Ś 2 calls), wasting ~180s, before falling
back to full context.

**Root cause:** 500 tokens Ôëł 375 words. A briefer JSON with `context` containing 3
facts + modules + tools + tags + entities easily exceeds this for complex queries.

**Files:** `kiso/config.py` (`MAX_TOKENS_DEFAULTS`)

**Change:**

- [ ] Change `MAX_TOKENS_DEFAULTS["briefer"]` from `500` to `2048`
- [ ] The briefer model (gemini-2.5-flash-lite) is fast and cheap ÔÇö 2048 tokens
      costs fractions of a cent and adds negligible latency
- [ ] Existing briefer tests pass unchanged (they mock call_llm, don't hit token limits)

### M670 ÔÇö Broaden install_proposal detection for msg-only plans on fresh instances Ôťů

**Problem:** F1 browser install test is flaky. When the planner produces a msg-only plan
proposing a tool install, `install_proposal` depends on two signals:

1. `plan.get("needs_install")` ÔÇö requires the LLM to populate the field (unreliable)
2. `saw_uninstalled_tool` ÔÇö requires the planner to have first attempted a tool task
   that failed validation (doesn't happen when planner goes straight to msg-only)

When both signals miss (LLM sets `needs_install: null` and skips the tool task),
`install_proposal = False` Ôćĺ next turn has `install_approved = False` Ôćĺ install exec
is blocked by validation Ôćĺ browser never gets installed.

This is a real UX bug: user approves install, system says "you can't install yet".

**Root cause:** The detection in `run_planner` (line ~1204) doesn't cover the case where
the planner directly produces a valid msg-only plan without going through the
uninstalled-tool validation error path.

**Files:** `kiso/brain.py` (`run_planner`, around line 1204)

**Change:**

- [x] Broaden the `install_proposal` detection with a third condition
- [x] No impact when tools are installed ÔÇö `installed_names` is non-empty, condition never fires
- [x] Zero latency cost ÔÇö no validation retry, just a broader post-hoc check
- [x] 4 unit tests: msg-only+no-toolsÔćĺTrue, msg-only+toolsÔćĺFalse, explicit needs_installÔćĺTrue, exec+msgÔćĺFalse

## Phase 56: Task Execution Improvements ÔÇö Parallel Groups + Dynamic Replan Limit

Two orthogonal improvements to plan execution. Parallel groups enable scatter-gather
workflows (critical for marketing/growth presets: multi-source research, competitor
analysis, batch processing). Dynamic replan limit forces focus at deep replan depths.

**Why now:** Phase 55 presets create personas (performance marketer, SEO specialist)
whose core workflows are multi-source fan-out: search 5 competitors, fetch 3 data
sources, analyze N pages. Without parallelism these plans are 5├Ś slower than they
need to be ÔÇö the difference between "usable" and "painful" for batch workflows.

**Reference:** 14 of 34 documented OpenClaw use cases are "highly parallelizable"
with the same scatter-gather pattern. OpenClaw itself is sequential per session ÔÇö
this would be a concrete architectural advantage.

      ```python
      def _build_execution_batches(tasks):
          """Group consecutive tasks by parallel_group into batches.

          Returns list of batches. Each batch is a list of (index, task_row) tuples.
          Batch with 1 item = sequential. Batch with 2+ items = parallel.

          batches = []
          current_batch = []
          current_group = None
          for i, t in enumerate(tasks):
              g = t.get("parallel_group")
              if g is not None and g == current_group:
                  current_batch.append((i, t))
              else:
                  if current_batch:

          if current_batch:

          return batches
      ```

## Phase 52 ÔÇö Taxonomy Rename: MD skills Ôćĺ recipes, tools Ôćĺ skills

**Motivation:** Aligning with the AgentSkills ecosystem (Claude Code, OpenClaw, Nanobot, Codex, Copilot, Cursor, Gemini CLI). The spec defines "skill" as an installable capability package (SKILL.md + scripts/ + assets/), which maps to our current "tools" (browser, aider, websearch). Our "MD skills" are a distinct concept ÔÇö pure planner instructions without executable code ÔÇö better described as "recipes" (a set of instructions to produce a specific result).

## Phase 57 ÔÇö Planner context reliability

Results from full test run after Phase 53-56: 6 previously failing tests now
pass (M828, M829, M830, M833, M841 confirmed). 5 remaining failures analyzed.

### Root cause analysis

Three recurring planner failures share the SAME root cause:
1. Planner uses `pip install` instead of `uv pip install`
2. Planner investigates (exec+replan) instead of setting `needs_install`
3. Planner treats kiso tools as system packages

**Root cause:** The briefer is too aggressive in filtering planner modules.
When `briefer_enabled=True` (default, and used in live/functional tests), the
briefer prompt says "Most requests need ZERO modules" and "AGGRESSIVE filtering.
Default to EXCLUDING." For install requests like "installa flask", the briefer
returns `modules: []`.

The planner then receives NO `kiso_native` module ÔÇö which contains ALL the
rules about `uv pip install`, `registry_hints`, `needs_install`, and the
decision tree for system packages vs Python libs vs kiso tools.

**Why unit tests pass:** They use `briefer_enabled=False`, which triggers the
fallback path (brain.py:1148-1167) that includes `kiso_native` by default.

**Why the fix is NOT "make the briefer smarter":** Relying on an LLM to
correctly select a module that contains safety constraints is fragile by
design. The constraint "never use bare pip install" is not contextual ÔÇö it
ALWAYS applies. Rules that always apply belong in the core prompt, not in
optional modules.

**The code gap:** brain.py:1138-1147 has a safety net that forces
`plugin_install` into the briefer path when no tools are installed, but does
NOT force `kiso_native`. The kiso_native module was treated as optional
context, but it contains mandatory system constraints.

### ~~M855~~ ÔÇö DROPPED: tool cache breaks test realism

**Problem:** Every functional test run installs browser/ocr/aider from scratch:
git clone + uv sync + deps.sh (playwright + chromium = ~100MB download). F1a
alone takes several minutes just for the install. This cost is paid every single
run because `_func_kiso_dir` creates a fresh temp directory.

**Why copy/symlink won't work:** Python venvs contain absolute paths in
shebang lines, pyvenv.cfg, and pip metadata. Copying a venv from one path to
another breaks it. A cache that copies tools in/out is fundamentally broken.

**Approach: fixed KISO_DIR path backed by Docker volume.**

Instead of `tmp_path_factory.mktemp("kiso_home")`, use a fixed path like
`/opt/kiso-test/` mounted as a Docker volume. Tools installed there persist
across runs because:
- The path is always `/opt/kiso-test/tools/browser/.venv/` ÔÇö venv absolute

### M878 ÔÇö Invalidate tools cache before replan planner call Ôťů

**Problem (kiso behavior):** When a plan installs a tool and then replans,

boundaries.

**Fix:** Call `invalidate_tools_cache()` before `run_planner()` in:
1. The replan loop (loop.py ~line 2340)
2. The initial planner call (loop.py ~line 2560) ÔÇö already fresh, but
   explicit invalidation prevents future bugs

This is one line of code. The impact is: every planner call sees fresh tool
state. The TTL cache still helps within `_execute_plan` (multiple
discover_tools calls in the same plan execution stay cached).

### M880 ÔÇö Compress install turns in replan context Ôťů

**Problem:** After the install-approve-install cycle (3-5 messages), the
planner's recent message context is full of install noise. When the planner
finally needs to USE the tool, it's confused by the install history and
generates exec tasks instead of tool tasks.

This is why F17 fails: after 3 install cycles (browser, ocr, aider), the
context has 15+ install-related messages. The planner can't distinguish
between "install this tool" and "use this tool" because both are in
recent history.

F30 passes because tools are pre-installed ÔÇö zero install noise in context.

**Fix: structured compression of install turns.**

In `build_recent_context()` or in the planner message building, detect
consecutive install-related messages (propose Ôćĺ approve Ôćĺ install result)
and compress them into a single structured line:

Before (5 messages):
```
[user] Use aider to write word_count.py
[kiso] Vuoi installare aider?
[user] s├Č, installa il tool aider
[kiso] Tool aider installato. Replan...
[user] Use aider to write word_count.py  ÔćÉ repeated by helper
```

After (2 lines):
```
[install completed] aider is now available.
[user] Use aider to write word_count.py
```

**Implementation approach:**

In `build_recent_context()` (worker/utils.py), add a compression pass:
1. Scan messages for install patterns (kiso msg with "install" keywords +
   user reply with approval keywords + kiso msg with install result)
2. Replace the group with a single "[install completed] {tool} available."
   line + keep the original user request
3. Non-install messages pass through unchanged

This is deterministic (pattern matching on message content), generalizable
(works for any tool install, not just F17), and doesn't depend on LLM.

**Alternative considered:** Have the briefer filter install messages. Rejected
because the briefer is an LLM call (adds latency + flakiness) and we need
deterministic behavior here.

**Files:** `kiso/worker/utils.py`, `tests/test_worker.py`
